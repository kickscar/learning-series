# 1. Instance 생성이 의미하는 것

## 1. "서버 한 대"가 아니라 연결 구조를 만드는 작업이다

EC2 Instance 생성은 "서버 한 대를 만든다"가 아니라, VPC 안에 네트워크 인터페이스(ENI)를 붙이고, 보안 경계(Security Group)를 걸고, OS 이미지(AMI)로 부팅 가능한 Compute 리소스를 하나 배치하는 작업이다. 이후에 SSH로 접속하려면 네트워크 경로와 보안 규칙이 동시에 맞아야 한다.

이 Section에서는 Default VPC를 사용해 최소 구성으로 Instance를 만든다. 네트워크 설계(Custom VPC, Subnet, Routing)는 Ch04에서 별도로 다룬다.

### ① 연결 구조(SSH 기준)

```mermaid
flowchart LR
  U[사용자 PC] -->|SSH 22| IP[Public IP]
  IP --> ENI[ENI]
  ENI --> SG[Security Group]
  SG --> EC2[EC2 Instance]
  EC2 --> VPC[Default VPC]
  VPC --> IGW[Internet Gateway]
```

이 다이어그램은 SSH 접속이 단순히 Instance 상태(Running)만으로 결정되지 않는다는 점을 보여준다. Public IP, SG Inbound(22), IGW 경로가 동시에 필요하다.

# 2. Launch Instance 마법사에서 결정하는 것들

## 1. AMI / Instance Type / Key Pair / Network / Security Group

Launch Instance는 "한 번에 많은 설정을 묻는 화면"이 아니라, 아래 선택들을 통해 Instance의 성격을 결정하는 과정이다.

### ① AMI(Amazon Machine Image)

AMI는 OS + 기본 소프트웨어가 포함된 부팅 이미지다. 같은 Instance Type이라도 AMI가 다르면 서버 내부 환경이 달라진다.

### ② Instance Type

Instance Type은 CPU/Memory/Network 성능을 정의한다. 이 시리즈는 성능 튜닝이 아니라 연결 구조 이해가 목적이므로 최소 사양으로 진행한다.

### ③ Key Pair

Key Pair는 SSH 접속을 위한 공개키/개인키 쌍이다. Console에서 Key Pair를 선택하는 순간, Instance 내부에 해당 공개키가 심어진 상태로 부팅된다.

### ④ Network과 Security Group

이 Section에서는 Default VPC를 사용한다. Default VPC는 IGW와 Public Subnet 등이 이미 구성되어 있어 "인터넷으로 나갈 수 있는 네트워크"가 기본으로 제공된다.

Security Group은 Instance 수준 방화벽이다. SSH 접속을 위해서는 Inbound 규칙으로 22 포트를 허용해야 한다. 상세한 보안 경계 설계는 Ch04에서 다룬다.

# 3. SSH 접속과 Instance Lifecycle

## 1. SSH 접속 방식

SSH 접속은 "키를 가진 사용자가 네트워크로 Instance에 도달할 수 있는가"를 검증하는 가장 단순한 방법이다.

### ① Linux/Mac: ssh -i

로컬에서 `.pem` 파일 권한을 안전하게 설정하고 `ssh -i`로 접속한다. 이 과정에서 AWS CLI는 사용하지 않는다.

### ② Windows: PuTTY 또는 OpenSSH

중요한 점은 "개인키를 안전하게 보관하고, 올바른 사용자 계정으로 접속한다"는 것이다.

## 2. Instance Lifecycle

Instance는 Running, Stopped, Terminated 상태를 가진다.

- Stop은 "전원 종료"에 가깝다. 다시 Start하면 부팅되지만 Public IP는 바뀔 수 있다.
- Terminate는 "리소스 삭제"다. 삭제 후에는 복구할 수 없다.

이 Lifecycle은 비용과도 연결된다. 불필요한 Instance를 Terminate하지 않으면 비용이 계속 발생할 수 있다.

---

# 핵심 정리

- EC2 Instance 생성은 AMI/Instance Type/Key Pair/Network/Security Group 조합으로 "배치 가능한 서버"를 만드는 작업이다.
- SSH 접속 성공 여부는 Public IP, Security Group(22 허용), 네트워크 경로(IGW 등)까지 포함한 연결 구조로 결정된다.
- Stop/Start/Terminate는 상태 전이이면서 비용/운영 관점과 직결된다.
- 이 Section은 Default VPC를 사용하고, Custom VPC 설계는 Ch04에서 다룬다.

# [실습] lab06: EC2 Instance 생성과 SSH 접속

Default VPC 환경에서 Amazon Linux 기반 EC2 Instance를 생성하고, Security Group으로 SSH(22) 접근을 허용한 뒤 로컬에서 SSH로 접속한다. 이후 Stop/Start/Terminate를 수행하며 Instance Lifecycle을 확인한다.

### 실습 목표

- Default VPC에서 EC2 Instance를 생성한다.
- Security Group으로 SSH(22) 접근을 허용한다.
- 로컬에서 SSH로 접속해 동작을 확인한다.
- Stop/Start/Terminate로 Instance Lifecycle을 확인한다.

⚠️ 비용 주의: 본 실습에서는 EC2 리소스를 생성하므로 과금이 발생할 수 있다. 실습 종료 시 자원 정리를 수행한다.

# 1. 전체 아키텍처

```mermaid
flowchart LR
  U[사용자 PC] -->|SSH 22| IGW[Internet Gateway]
  IGW --> VPC[Default VPC]
  VPC --> ENI[ENI]
  ENI --> SG[Security Group]
  SG --> EC2[EC2 Instance]
```

이 아키텍처는 "외부에서 SSH로 접속"하는 최소 흐름을 보여준다. Default VPC는 이미 IGW와 Public Subnet을 포함하므로, 실습의 핵심은 Security Group(22 허용)과 Key Pair를 올바르게 설정하는 것이다.

# 4. 실행 및 결과 검증

## 1. 로컬 SSH 접속(Linux/Mac 기준)

```bash
chmod 400 **{path-to-key.pem}**
ssh -i **{path-to-key.pem}** **{user}**@**{public-ip}**
```

예상 결과:

- 접속 성공 시, 쉘 프롬프트가 보인다.


## 2. 결과 검증

- SSH 접속이 성공한다.
- Stop/Start 후 상태가 정상 전환된다(접속 가능 여부 포함).
- Terminate 후 Instance가 삭제된다.
