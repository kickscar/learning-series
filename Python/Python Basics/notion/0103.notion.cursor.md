# 1.3 가상환경(venv)과 의존성 격리

## 개요

파이썬 프로젝트를 개발하다 보면, 프로젝트마다 다른 버전의 패키지가 필요할 수 있다. 예를 들어 프로젝트 A는 `requests==2.25.1`을 쓰고, 프로젝트 B는 `requests==2.31.0`을 쓴다면, 전역적으로 하나의 버전만 설치할 수 있는 환경에서는 충돌이 생긴다. 가상환경(venv)은 이런 문제를 해결하기 위해 **프로젝트별로 격리된 Python 실행 환경**을 만든다.

가상환경을 이해하지 못하면 "전역에 패키지를 설치해서 쓰면 되지 않나?"라는 오해가 생기고, 프로젝트 간 의존성 충돌이나 "어디서 설치한 패키지인지 모르겠다"는 혼란이 생긴다. venv는 단순히 "패키지를 따로 설치하는 도구"가 아니라, **Python 실행 파일 자체와 site-packages를 프로젝트별로 격리**하는 메커니즘이다. 이 차이를 이해하면 이후 패키지 관리, 배포, CI/CD에서 venv의 역할이 명확해진다.

---

## 핵심 개념

### 가상환경이란

가상환경(venv)은 **독립된 Python 실행 환경**이다. 각 가상환경은:

1. **자체 Python 실행 파일**을 가진다(또는 시스템 Python을 심볼릭 링크로 가리킨다).
2. **자체 `site-packages` 디렉터리**를 가진다. 여기에 설치된 패키지는 이 가상환경에서만 보인다.
3. **자체 스크립트 디렉터리**(`bin`/`Scripts`)를 가진다. `pip`, `python` 등이 여기에 있다.

전역 Python 환경과 완전히 분리되어 있어, 한 가상환경에 설치한 패키지는 다른 가상환경이나 전역 환경에 영향을 주지 않는다.

### venv 생성

가상환경은 `python -m venv {경로}`로 만든다. 보통 프로젝트 루트에 `.venv`라는 이름으로 만든다.

```bash
python -m venv .venv
```

이 명령은 `.venv` 디렉터리를 만들고, 그 안에 다음 구조를 만든다:

```
.venv/
├── bin/          # Unix/macOS: 실행 파일들 (python, pip 등)
│   └── activate  # 활성화 스크립트
├── Scripts/      # Windows: 실행 파일들 (python.exe, pip.exe 등)
│   └── activate  # 활성화 스크립트
├── lib/
│   └── python3.x/
│       └── site-packages/  # 패키지가 설치되는 곳
└── pyvenv.cfg    # 가상환경 설정 파일
```

`python -m venv`는 **현재 실행 중인 Python**을 기준으로 가상환경을 만든다. 예를 들어 `python3.11`로 venv를 만들면, 그 venv의 Python도 3.11이다.

### 활성화(Activation)

가상환경을 **활성화**하면, 현재 셸 세션에서 그 가상환경의 Python과 패키지를 사용하게 된다.

**Unix/macOS:**

```bash
source .venv/bin/activate
```

**Windows (CMD):**

```cmd
.venv\Scripts\activate.bat
```

**Windows (PowerShell):**

```powershell
.venv\Scripts\Activate.ps1
```

활성화되면 프롬프트 앞에 `(.venv)` 같은 표시가 나타나고, 다음이 일어난다:

1. **`PATH` 환경 변수 앞에 venv의 `bin`/`Scripts`가 추가**된다. 이렇게 하면 `python`, `pip` 같은 명령이 venv의 것을 먼저 찾게 된다.
2. **`VIRTUAL_ENV` 환경 변수**가 venv 경로로 설정된다.
3. **`sys.prefix`가 venv 경로로 바뀐다**(Python 내부적으로).

비활성화는 `deactivate` 명령으로 한다.

### site-packages와 패키지 설치

가상환경이 활성화된 상태에서 `pip install {패키지}`를 실행하면, 패키지는 **venv의 `site-packages`**에 설치된다. 전역 `site-packages`가 아니라.

예를 들어:

```bash
$ source .venv/bin/activate
(.venv) $ pip install requests
```

이렇게 하면 `requests`는 `.venv/lib/python3.x/site-packages/requests/`에 설치된다. 전역 Python 환경의 `site-packages`에는 설치되지 않는다.

`pip list`를 실행하면, 현재 활성화된 가상환경(또는 전역 환경)에 설치된 패키지만 보인다. 다른 가상환경의 패키지는 보이지 않는다.

### sys.prefix와 sys.executable

가상환경이 활성화되면, Python 내부적으로 다음이 바뀐다:

- **`sys.prefix`**: venv의 루트 경로(예: `/path/to/proj/.venv`)로 설정된다. 전역 Python에서는 시스템 Python 경로(예: `/usr`)가 된다.
- **`sys.executable`**: venv의 Python 실행 파일 경로로 설정된다.

이 값들을 확인하면 현재 어떤 환경에서 실행 중인지 알 수 있다:

```python
import sys
print(f"sys.prefix: {sys.prefix}")
print(f"sys.executable: {sys.executable}")
```

### PYTHONPATH와의 관계

`PYTHONPATH` 환경 변수는 모듈 검색 경로에 추가되는 경로 목록이다. 가상환경과의 우선순위는 다음과 같다:

1. **`sys.path[0]`**: 실행 모드에 따라 설정(Ch1.2 참고).
2. **가상환경의 `site-packages`**: `sys.prefix/lib/python3.x/site-packages`.
3. **`PYTHONPATH`**: 환경 변수로 설정된 경로들.
4. **표준 라이브러리 경로**.
5. **전역 `site-packages`**: 가상환경이 활성화되어 있으면 보통 검색되지 않는다.

가상환경이 활성화되어 있으면, venv의 `site-packages`가 `PYTHONPATH`보다 우선순위가 높다. 다만 `PYTHONPATH`에 명시적으로 경로를 추가하면 그 경로도 검색된다.

### requirements.txt와 의존성 관리

가상환경에서 설치한 패키지 목록을 `requirements.txt`에 저장해, 다른 환경에서도 같은 패키지를 설치할 수 있다.

**패키지 목록 저장:**

```bash
(.venv) $ pip freeze > requirements.txt
```

**다른 환경에서 설치:**

```bash
$ python -m venv .venv
$ source .venv/bin/activate
(.venv) $ pip install -r requirements.txt
```

`requirements.txt`는 프로젝트의 **의존성 명세** 역할을 한다. 버전 관리 시스템(Git 등)에 포함시켜, 팀원들이 같은 환경을 재현할 수 있게 한다.

## 코드 예제

### 1) venv 생성과 활성화

```bash
# 가상환경 생성
$ python -m venv .venv

# 활성화 (Unix/macOS)
$ source .venv/bin/activate
(.venv) $ 

# Python 경로 확인
(.venv) $ which python
/path/to/proj/.venv/bin/python

# sys.prefix 확인
(.venv) $ python -c "import sys; print(sys.prefix)"
/path/to/proj/.venv
```

**아하 모먼트**: 활성화 후 `which python`이 venv의 Python을 가리킨다. `sys.prefix`도 venv 경로로 바뀐다.

### 2) 패키지 설치와 site-packages

```bash
# 가상환경 활성화
$ source .venv/bin/activate
(.venv) $ pip install requests

# 설치된 패키지 확인
(.venv) $ pip list
Package    Version
---------- -------
pip        23.0.1
requests   2.31.0
...

# site-packages 위치 확인
(.venv) $ python -c "import site; print(site.getsitepackages())"
['/path/to/proj/.venv/lib/python3.11/site-packages']
```

**비활성화 후:**

```bash
(.venv) $ deactivate
$ pip list
# 전역 환경의 패키지 목록이 보임 (requests는 없을 수 있음)
```

**아하 모먼트**: 같은 `pip list`라도 활성화된 환경에 따라 다른 패키지 목록이 보인다. venv의 `site-packages`와 전역 `site-packages`가 분리되어 있기 때문이다.

### 3) 여러 가상환경 비교

```bash
# 프로젝트 A
$ cd proj_a
$ python -m venv .venv
$ source .venv/bin/activate
(.venv) $ pip install requests==2.25.1

# 프로젝트 B (다른 터미널)
$ cd proj_b
$ python -m venv .venv
$ source .venv/bin/activate
(.venv) $ pip install requests==2.31.0
```

두 프로젝트는 서로 다른 버전의 `requests`를 사용할 수 있다. 각각의 venv가 독립적인 `site-packages`를 가지기 때문이다.

### 4) requirements.txt 사용

```bash
# 개발 환경에서 의존성 저장
(.venv) $ pip freeze > requirements.txt

# requirements.txt 내용 예시
# requests==2.31.0
# numpy==1.24.3
```

**다른 환경에서 재현:**

```bash
$ python -m venv new_venv
$ source new_venv/bin/activate
(new_venv) $ pip install -r requirements.txt
```

**아하 모먼트**: `requirements.txt`로 프로젝트의 의존성을 코드처럼 관리할 수 있다. 버전 관리에 포함시키면 팀 전체가 같은 환경을 재현할 수 있다.

### 5) sys.prefix로 환경 확인

```python
# script.py
import sys
import os

print(f"sys.prefix: {sys.prefix}")
print(f"sys.executable: {sys.executable}")
print(f"VIRTUAL_ENV: {os.environ.get('VIRTUAL_ENV', 'Not set')}")

if '.venv' in sys.prefix or 'venv' in sys.prefix:
    print("가상환경에서 실행 중")
else:
    print("전역 Python 환경에서 실행 중")
```

```bash
# 전역 환경
$ python script.py
sys.prefix: /usr
가상환경에서 실행 중: False

# venv 활성화 후
$ source .venv/bin/activate
(.venv) $ python script.py
sys.prefix: /path/to/proj/.venv
가상환경에서 실행 중: True
```

## 주의사항 (Pitfalls)

### 1) "전역에 설치하면 되지 않나?"라는 오해

파이썬 입문자나 다른 언어(C/Java 등)에 익숙한 사람은 "전역에 패키지를 설치해서 모든 프로젝트에서 쓰면 되지 않나?"라고 생각하기 쉽다. 하지만:

- **버전 충돌**: 프로젝트 A는 `requests==2.25.1`, 프로젝트 B는 `requests==2.31.0`을 써야 할 때, 전역에는 하나만 설치할 수 있다.
- **의존성 혼란**: 어떤 패키지가 어떤 프로젝트에서 필요한지 알기 어렵다.
- **시스템 Python 오염**: 시스템 도구가 파이썬에 의존하는 경우, 전역 설치가 시스템을 망가뜨릴 수 있다.

가상환경은 이런 문제를 해결한다.

### 2) 활성화하지 않고 venv 사용

venv를 만들기만 하고 활성화하지 않으면, 여전히 전역 Python을 사용하게 된다. `python` 명령이 여전히 전역 Python을 가리키고, `pip install`도 전역 `site-packages`에 설치된다.

**잘못된 사용:**

```bash
$ python -m venv .venv
$ python script.py  # 전역 Python 사용!
$ pip install requests  # 전역에 설치됨!
```

**올바른 사용:**

```bash
$ python -m venv .venv
$ source .venv/bin/activate  # 활성화 필수!
(.venv) $ python script.py
(.venv) $ pip install requests
```

### 3) venv 디렉터리를 버전 관리에 포함

`.venv` 디렉터리는 **버전 관리에 포함하지 않는다**. `.gitignore`에 추가해야 한다:

```
.venv/
venv/
env/
```

이유:
- venv는 **재현 가능**하다: `requirements.txt`만 있으면 `pip install -r requirements.txt`로 재생성할 수 있다.
- venv는 **플랫폼/환경에 종속적**이다: 다른 OS나 Python 버전에서는 동작하지 않을 수 있다.
- venv는 **용량이 크다**: 수백 MB~수 GB가 될 수 있어 Git 저장소를 불필요하게 크게 만든다.

### 4) 다른 Python 버전으로 venv 만들기

`python -m venv`는 **현재 실행 중인 Python**을 기준으로 venv를 만든다. Python 3.11로 venv를 만들려면 `python3.11 -m venv`를 실행해야 한다.

```bash
# Python 3.11로 venv 생성
$ python3.11 -m venv .venv

# Python 3.12로 venv 생성
$ python3.12 -m venv .venv
```

시스템에 여러 Python 버전이 설치되어 있을 때, 어떤 버전으로 venv를 만들었는지 확인하려면:

```bash
(.venv) $ python --version
Python 3.11.5
```

### 5) PYTHONPATH와 venv의 충돌

`PYTHONPATH`에 경로를 설정하면, 그 경로가 모듈 검색에 포함된다. 하지만 venv의 `site-packages`가 우선순위가 높으므로, 보통 문제가 되지 않는다. 다만 `PYTHONPATH`에 venv와 충돌하는 경로를 넣으면 예상과 다른 모듈을 import할 수 있다.

가상환경을 사용할 때는 `PYTHONPATH`를 설정하지 않는 것이 일반적이다. venv 자체가 격리를 제공하기 때문이다.

### 6) Windows에서 PowerShell 실행 정책

Windows PowerShell에서 `Activate.ps1`을 실행할 때, 실행 정책 때문에 오류가 날 수 있다:

```
cannot be loaded because running scripts is disabled on this system
```

해결 방법:

```powershell
# 현재 세션에만 허용
Set-ExecutionPolicy -ExecutionPolicy RemoteSigned -Scope Process

# 또는 CMD 사용
.venv\Scripts\activate.bat
```

## Deep Insight

### venv의 내부 구조

venv는 단순히 디렉터리를 만드는 것이 아니라, Python의 모듈 검색 메커니즘과 통합되어 있다. `pyvenv.cfg` 파일에는 다음과 같은 정보가 들어 있다:

```
home = /usr/bin
include-system-site-packages = false
version = 3.11.5
```

`include-system-site-packages = false`가 기본값이며, 이는 전역 `site-packages`를 검색하지 않는다는 의미다. `true`로 설정하면(venv 생성 시 `--system-site-packages` 옵션) 전역 패키지도 함께 검색할 수 있지만, 격리가 약해진다.

### sys.prefix의 역할

`sys.prefix`는 Python이 자신이 "어디에 설치되어 있는가"를 나타내는 경로다. 가상환경에서는 이 값이 venv 루트로 설정되어, Python이 venv의 `site-packages`를 찾을 수 있게 한다. `site` 모듈이 `sys.prefix`를 기반으로 `site-packages` 경로를 구성한다.

### PATH 조작 vs sys.prefix

venv 활성화는 두 가지 메커니즘을 사용한다:

1. **셸 레벨**: `PATH` 환경 변수에 venv의 `bin`/`Scripts`를 추가해, `python`, `pip` 명령이 venv의 것을 먼저 찾게 한다.
2. **Python 레벨**: `sys.prefix`를 venv 경로로 설정해, Python 내부적으로 venv의 `site-packages`를 찾게 한다.

셸에서 `python`을 실행하면 `PATH` 덕분에 venv의 Python이 실행되고, 그 Python은 `sys.prefix` 덕분에 venv의 `site-packages`를 사용한다. 이 두 메커니즘이 함께 작동해 완전한 격리를 만든다.
