### 1. 수동 apply의 위험

팀 환경에서 `terraform apply`를 로컬에서 실행할 때 발생하는 문제:

**State Locking 충돌:**
- 여러 엔지니어가 동시에 apply하면 State 잠금 충돌 발생
- S3 native locking(`use_lockfile = true`, TF 1.10+)이 동시 쓰기를 방지하지만, 충돌 시 수동 해제 필요
- 누가 언제 잠금을 잡았는지 추적 어려움

**무단 변경:**
- 코드 리뷰 없이 인프라 변경 가능
- main 브랜치에 반영되지 않은 로컬 변경사항으로 apply 가능
- 팀원 간 State drift 유발

**감사 추적 부재:**
- 누가, 언제, 어떤 변경을 apply했는지 기록 없음
- 장애 발생 시 원인 추적 불가
- 규정 준수(compliance) 요건 충족 불가

**해결 방향:**
```
로컬 개발 → PR 생성 → 자동 plan → 코드 리뷰 → merge → 자동 apply
```

### 2. GitHub Actions for Terraform

#### hashicorp/setup-terraform action

**현재 버전:** v4.0.0 (2026-02-24)

**Inputs:**

| Input | 설명 | 기본값 |
|-------|------|--------|
| `terraform_version` | 설치할 버전. semver range 지원 (`~1.14.0`, `1.14.6`) | `latest` |
| `terraform_wrapper` | stdout/stderr/exitcode 캡처 wrapper 활성화 | `true` |
| `cli_config_credentials_token` | HCP Terraform / TFE API 토큰 | — |
| `cli_config_credentials_hostname` | TFE/HCP 호스트명 | `app.terraform.io` |

**Wrapper 동작:**
- `terraform_wrapper: true`일 때 terraform 명령 실행 결과를 step output으로 노출
- `steps.{step_id}.outputs.stdout` — 표준 출력
- `steps.{step_id}.outputs.stderr` — 에러 출력
- `steps.{step_id}.outputs.exitcode` — 종료 코드
- PR comment에 plan 결과를 게시할 때 `stdout`를 직접 참조 가능

**기본 사용:**
```yaml
- uses: hashicorp/setup-terraform@v4
  with:
    terraform_version: "1.14.6"
```

**플랫폼:** ubuntu-latest, macos-latest, windows-latest 지원

#### Job 구조

```yaml
jobs:
  terraform:
    runs-on: ubuntu-latest
    permissions:
      contents: read        # 코드 체크아웃
      id-token: write       # OIDC 토큰 발급 (필수)
      pull-requests: write  # PR comment 작성
    defaults:
      run:
        working-directory: ./infra  # monorepo 대응
    steps:
      - uses: actions/checkout@v4
      - uses: hashicorp/setup-terraform@v4
        with:
          terraform_version: "1.14.6"
      - uses: aws-actions/configure-aws-credentials@v6
        with:
          role-to-assume: ${{ secrets.AWS_ROLE_ARN }}
          aws-region: ap-northeast-2
      - run: terraform init
      - run: terraform plan -input=false -no-color
```

**핵심 permissions:**
- `id-token: write` — GitHub OIDC 토큰 발급에 필수. 이 권한 없이는 AWS 인증 불가
- `pull-requests: write` — PR comment 작성에 필요
- `contents: read` — 코드 체크아웃 기본 권한

### 3. AWS OIDC 인증 (GitHub → AWS)

#### 동작 원리

```
GitHub Actions Runner → GitHub OIDC Provider → JWT 토큰 발급
    → AWS STS AssumeRoleWithWebIdentity → 임시 자격 증명 획득
```

1. GitHub Actions가 워크플로우 실행 시 OIDC 토큰(JWT)을 발급
2. `aws-actions/configure-aws-credentials` action이 이 토큰으로 AWS STS에 AssumeRole 요청
3. AWS IAM이 Trust Policy 조건(repo, branch 등)을 검증
4. 조건 충족 시 임시 자격 증명(Access Key, Secret Key, Session Token) 발급
5. 이후 step에서 이 자격 증명으로 AWS API 호출

**장점:** 장기 자격 증명(Access Key/Secret Key)을 GitHub Secrets에 저장할 필요 없음.

#### aws-actions/configure-aws-credentials

**현재 버전:** v6.1.0

**OIDC 관련 Inputs:**

| Input | 설명 | 기본값 |
|-------|------|--------|
| `role-to-assume` | IAM Role ARN (OIDC에서 필수) | — |
| `aws-region` | AWS 리전 | — |
| `audience` | JWT audience claim | `sts.amazonaws.com` |
| `role-duration-seconds` | 세션 유효 시간 (900~43200초) | `3600` |
| `role-session-name` | 세션 이름 (CloudTrail 추적용) | `GitHubActions` |

```yaml
- uses: aws-actions/configure-aws-credentials@v6
  with:
    role-to-assume: arn:aws:iam::123456789012:role/GitHubActions-TerraformRole
    aws-region: ap-northeast-2
```

#### AWS 측 설정

**1) IAM OIDC Identity Provider 생성:**

| 항목 | 값 |
|------|---|
| Provider Type | OpenID Connect |
| Provider URL | `https://token.actions.githubusercontent.com` |
| Audience | `sts.amazonaws.com` |

**2) IAM Role Trust Policy:**

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "Federated": "arn:aws:iam::123456789012:oidc-provider/token.actions.githubusercontent.com"
      },
      "Action": "sts:AssumeRoleWithWebIdentity",
      "Condition": {
        "StringEquals": {
          "token.actions.githubusercontent.com:aud": "sts.amazonaws.com"
        },
        "StringLike": {
          "token.actions.githubusercontent.com:sub": "repo:OWNER/REPO:*"
        }
      }
    }
  ]
}
```

**Condition Key 패턴:**

| Condition Key (sub claim) | 의미 |
|---------------------------|------|
| `repo:OWNER/REPO:ref:refs/heads/main` | 특정 브랜치만 허용 |
| `repo:OWNER/REPO:environment:production` | 특정 GitHub Environment만 허용 |
| `repo:OWNER/REPO:*` | 해당 repo의 모든 워크플로우 허용 |
| `repo:OWNER/REPO:pull_request` | PR 워크플로우만 허용 |

**보안 권장사항:**
- `StringLike`로 repo를 반드시 한정 — 와일드카드(`*`)를 Organization 수준에서 사용하지 않음
- 프로덕션 apply는 `ref:refs/heads/main`으로 브랜치 제한
- Plan 전용 Role과 Apply 전용 Role을 분리하면 더 안전

**3) IAM Role 권한:**
- Plan 전용: `ReadOnlyAccess` 또는 필요한 Describe/List 권한
- Apply 전용: 실제 리소스 생성/수정/삭제 권한 + S3 State 접근

### 4. PR-based Plan 워크플로우

#### 트리거

```yaml
on:
  pull_request:
    branches: [main]
    types: [opened, synchronize]
```

- `opened`: PR 최초 생성 시
- `synchronize`: PR 브랜치에 새 커밋 push 시 (업데이트)

#### Plan 실행 및 PR Comment

**방법 1: wrapper stdout + actions/github-script**

```yaml
- name: Terraform Plan
  id: plan
  run: terraform plan -input=false -no-color
  continue-on-error: true

- name: Post Plan Comment
  uses: actions/github-script@v7
  with:
    github-token: ${{ secrets.GITHUB_TOKEN }}
    script: |
      const output = `#### Terraform Plan 📋
      \`\`\`
      ${{ steps.plan.outputs.stdout }}
      \`\`\`
      *Triggered by @${{ github.actor }}, Commit: \`${{ github.event.pull_request.head.sha }}\`*`;

      // 기존 comment 찾기 (업데이트용)
      const { data: comments } = await github.rest.issues.listComments({
        owner: context.repo.owner,
        repo: context.repo.repo,
        issue_number: context.issue.number,
      });
      const botComment = comments.find(c =>
        c.user.type === 'Bot' && c.body.includes('Terraform Plan')
      );

      if (botComment) {
        await github.rest.issues.updateComment({
          owner: context.repo.owner,
          repo: context.repo.repo,
          comment_id: botComment.id,
          body: output
        });
      } else {
        await github.rest.issues.createComment({
          owner: context.repo.owner,
          repo: context.repo.repo,
          issue_number: context.issue.number,
          body: output
        });
      }
```

**핵심 포인트:**
- `-no-color` 플래그: ANSI 색상 코드 제거 (PR comment에서 깨짐 방지)
- `continue-on-error: true`: plan 실패해도 comment step 실행
- 기존 comment 업데이트: PR에 push할 때마다 새 comment가 쌓이지 않도록 기존 comment를 찾아서 업데이트
- `${{ secrets.GITHUB_TOKEN }}`: 별도 PAT 불필요, 워크플로우 자동 제공

**Plan 출력 포맷팅:**
- GitHub PR comment는 65535자 제한 — 대규모 plan은 truncation 필요
- `<details>` 태그로 접기 가능:
  ```
  <details><summary>Show Plan</summary>

  ```text
  {plan output}
  ```

  </details>
  ```

### 5. Merge-to-Apply 워크플로우

#### 트리거

```yaml
on:
  push:
    branches: [main]
    paths:
      - '**.tf'
      - '**.tfvars'
```

main에 merge될 때만 실행.

#### Apply 전략

**전략 A: Re-plan + auto-approve (단순, 일반적)**

```yaml
- run: terraform plan -input=false -out=tfplan
- run: terraform apply -input=false tfplan
```

또는:

```yaml
- run: terraform apply -input=false -auto-approve
```

**전략 B: Saved plan artifact (엄격)**

PR 워크플로우에서 plan 파일을 artifact로 저장하고, apply 워크플로우에서 다운로드하여 실행.

```yaml
# PR workflow
- run: terraform plan -input=false -out=tfplan
- uses: actions/upload-artifact@v4
  with:
    name: tfplan
    path: tfplan

# Apply workflow
- uses: actions/download-artifact@v4
  with:
    name: tfplan
- run: terraform apply -input=false tfplan
```

**주의:** plan 파일에는 민감 정보(변수 값, provider 자격 증명)가 평문 포함됨. artifact 보존 기간과 접근 권한 관리 필요.

**실무 권장:** 대부분의 팀에서 전략 A(re-plan + apply)를 사용. PR에서 plan을 리뷰했으므로 main merge 후 re-plan하더라도 동일한 변경이 적용됨. State locking이 동시 apply를 방지.

#### Concurrency 제어

```yaml
concurrency:
  group: terraform-apply
  cancel-in-progress: false
```

- `group`: 같은 그룹의 워크플로우는 동시 실행 불가. 큐에 대기
- `cancel-in-progress: false`: 실행 중인 apply를 취소하지 않음 (terraform apply 중 취소는 위험)
- 환경별 그룹: `group: terraform-${{ github.event.inputs.environment }}`

### 6. GitHub Actions Workflow YAML 문법 정리

#### 전체 구조

```yaml
name: Terraform

on:
  pull_request:
    branches: [main]
    types: [opened, synchronize]
    paths: ['**.tf', '**.tfvars']
  push:
    branches: [main]
    paths: ['**.tf', '**.tfvars']
  workflow_dispatch:
    inputs:
      environment:
        description: 'Target environment'
        required: true
        type: choice
        options: [dev, prod]

permissions:
  contents: read
  id-token: write
  pull-requests: write

env:
  TF_VERSION: "1.14.6"
  AWS_REGION: "ap-northeast-2"

concurrency:
  group: terraform-${{ github.ref }}
  cancel-in-progress: false

jobs:
  plan:
    if: github.event_name == 'pull_request'
    runs-on: ubuntu-latest
    defaults:
      run:
        working-directory: ./infra   # monorepo
    steps:
      - uses: actions/checkout@v4
      - uses: hashicorp/setup-terraform@v4
        with:
          terraform_version: ${{ env.TF_VERSION }}
      - uses: aws-actions/configure-aws-credentials@v6
        with:
          role-to-assume: ${{ secrets.AWS_ROLE_ARN }}
          aws-region: ${{ env.AWS_REGION }}
      - run: terraform init -input=false
      - run: terraform plan -input=false -no-color

  apply:
    if: github.event_name == 'push'
    runs-on: ubuntu-latest
    defaults:
      run:
        working-directory: ./infra
    steps:
      - uses: actions/checkout@v4
      - uses: hashicorp/setup-terraform@v4
        with:
          terraform_version: ${{ env.TF_VERSION }}
      - uses: aws-actions/configure-aws-credentials@v6
        with:
          role-to-assume: ${{ secrets.AWS_ROLE_ARN }}
          aws-region: ${{ env.AWS_REGION }}
      - run: terraform init -input=false
      - run: terraform apply -input=false -auto-approve
```

**주요 문법 요소:**

| 요소 | 설명 |
|------|------|
| `on:` | 워크플로우 트리거 정의 |
| `permissions:` | GITHUB_TOKEN 권한 범위 (최소 권한 원칙) |
| `env:` | 워크플로우 전역 환경 변수 |
| `concurrency:` | 동시 실행 제어 |
| `jobs:` | 병렬/순차 실행 단위 |
| `runs-on:` | Runner 환경 |
| `defaults.run.working-directory:` | monorepo에서 TF 디렉토리 지정 |
| `steps:` | job 내 순차 실행 단위 |
| `if:` | 조건부 실행 |
| `secrets.*` | GitHub Secrets 참조 |
| `env.*` | 환경 변수 참조 |

### 1. 품질 게이트 파이프라인

#### 실행 순서

```
fmt -check → validate → tflint → tfsec/trivy → plan
```

각 단계의 역할:

| 단계 | 도구 | 검사 대상 | 실패 의미 |
|------|------|----------|----------|
| fmt | `terraform fmt -check` | 코드 포맷 | 포맷 불일치 (exit code 3) |
| validate | `terraform validate` | HCL 문법, 참조 일관성 | 구문 오류 |
| lint | `tflint` | Provider 규칙 기반 정적 분석 | 잘못된 instance_type 등 |
| security | `tfsec` / `trivy` | 보안 미스컨피그 | SG 0.0.0.0/0, 암호화 미설정 등 |
| plan | `terraform plan` | 실제 변경 사항 | 리소스 충돌, 권한 부족 등 |

#### Fail-fast vs continue-on-error

```yaml
steps:
  - name: Format Check
    run: terraform fmt -check -recursive
    # fail-fast: fmt 실패 시 이후 step 실행하지 않음 (기본 동작)

  - name: Validate
    run: terraform validate

  - name: TFLint
    run: tflint --recursive
```

기본 동작은 **fail-fast** — 앞 step 실패 시 이후 step 중단. `continue-on-error: true`로 개별 step의 실패를 무시하고 계속 진행 가능하지만, 품질 게이트에서는 fail-fast가 적절.

#### Exit Code 정리

| 도구 | 성공 | 실패 |
|------|------|------|
| `terraform fmt -check` | 0 (포맷 일치) | 3 (불일치 파일 존재) |
| `terraform validate` | 0 | 1 (문법/참조 오류) |
| `tflint` | 0 | 2 (warning), 3 (error) |
| `terraform plan -detailed-exitcode` | 0 (변경 없음) | 1 (오류), 2 (변경 있음) |

#### 통합 워크플로우 예시

```yaml
jobs:
  quality-gate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: hashicorp/setup-terraform@v4
        with:
          terraform_version: "1.14.6"

      - name: Format Check
        run: terraform fmt -check -recursive

      - name: Init
        run: terraform init -backend=false

      - name: Validate
        run: terraform validate

      - name: TFLint
        uses: terraform-linters/setup-tflint@v4
      - run: tflint --init && tflint --recursive

      - name: Security Scan
        uses: aquasecurity/tfsec-action@v1.0.3
        with:
          soft_fail: false
```

`terraform init -backend=false`: backend 설정 없이 init 실행. validate와 lint에는 실제 backend 연결이 불필요.

### 2. tfsec / trivy 보안 스캐닝

#### tfsec-action

**현재 버전:** v1.0.3

```yaml
- uses: aquasecurity/tfsec-action@v1.0.3
  with:
    working_directory: ./infra
    format: sarif
    soft_fail: false        # true면 발견해도 빌드 실패 안 함
```

**Inputs:**

| Input | 설명 | 기본값 |
|-------|------|--------|
| `working_directory` | 스캔 대상 디렉토리 | `.` |
| `version` | tfsec 버전 | `latest` |
| `format` | 출력 형식 (json, csv, sarif, junit) | — |
| `additional_args` | 추가 인수 | — |
| `soft_fail` | 발견해도 빌드 통과 | `false` |

**주요 탐지 항목:**

| 카테고리 | 예시 |

#### trivy-action (tfsec 통합/후속)

**현재 버전:** v0.36.0

tfsec은 Aqua Security의 Trivy로 통합되는 추세. Trivy는 컨테이너 이미지, 파일시스템, IaC 등 다양한 대상을 스캔하는 통합 도구.

```yaml
- name: Trivy IaC Scan
  uses: aquasecurity/trivy-action@v0.36.0
  with:
    scan-type: 'config'         # IaC 스캔 모드
    scan-ref: '.'
    severity: 'CRITICAL,HIGH'
    format: 'sarif'
    output: 'trivy-results.sarif'
    exit-code: '1'              # 발견 시 빌드 실패

- name: Upload SARIF
  uses: github/codeql-action/upload-sarif@v4
  with:
    sarif_file: trivy-results.sarif
  if: always()
```

**GitHub Security 탭 연동:**
- SARIF 형식으로 출력 → `github/codeql-action/upload-sarif@v4`로 업로드
- GitHub Security 탭 > Code scanning alerts에서 결과 확인
- `security-events: write` 권한 필요

### 4. Branch Protection + Required Checks

#### GitHub Branch Protection Rules

main 브랜치에 설정하는 보호 규칙:

| 규칙 | 설명 |
|------|------|
| Require a pull request before merging | direct push 차단 |
| Required approving reviews | 최소 1명 리뷰 승인 필수 |
| Require status checks to pass before merging | CI 통과 필수 |
| Require branches to be up to date before merging | main 최신 상태 동기화 필수 |
| Do not allow bypassing the above settings | 관리자도 규칙 우회 불가 |

**설정 경로:**
GitHub > Settings > Branches > Branch protection rules > Add rule
- Branch name pattern: `main`
- ✅ Require status checks to pass before merging
- Search and select: `quality-gate`, `terraform-plan`

### 1. 멀티 환경 배포

#### Directory-based 환경 분리 (Ch07 연결)

```text
infra/
├── envs/
│   ├── dev/
│   │   ├── main.tf
│   │   ├── locals.tf
│   │   ├── terraform.tfvars
│   │   └── providers.tf
│   └── prod/
│       ├── main.tf
│       ├── locals.tf
│       ├── terraform.tfvars
│       └── providers.tf
└── modules/
    ├── network/
    ├── platform/
    └── workload/
```

각 환경 디렉토리가 독립된 State를 가짐. 같은 modules/를 참조하되 terraform.tfvars로 환경별 값을 주입.

#### Matrix Strategy

```yaml
jobs:
  deploy:
    strategy:
      matrix:
        environment: [dev, prod]
      max-parallel: 1         # 순차 실행
      fail-fast: true          # 앞 환경 실패 시 중단
    runs-on: ubuntu-latest
    environment: ${{ matrix.environment }}
    defaults:
      run:
        working-directory: ./infra/envs/${{ matrix.environment }}
    steps:
      - uses: actions/checkout@v4
      - uses: hashicorp/setup-terraform@v4
      - uses: aws-actions/configure-aws-credentials@v6
        with:
          role-to-assume: ${{ secrets.AWS_ROLE_ARN }}
          aws-region: ap-northeast-2
      - run: terraform init -input=false
      - run: terraform apply -input=false -auto-approve
```

**핵심:**
- `max-parallel: 1` — dev 완료 후 prod 실행 (순차 배포)
- `fail-fast: true` — dev 실패 시 prod는 실행하지 않음
- `environment: ${{ matrix.environment }}` — GitHub Environment 연동 (approval gate)

#### 순차 배포: dev → staging → prod

matrix의 `max-parallel: 1`로 순차 실행하거나, 별도 job으로 분리하여 `needs:` 의존성 설정:

```yaml
jobs:
  deploy-dev:
    environment: dev
    # ...

  deploy-prod:
    needs: deploy-dev
    environment: prod
    # ...
```

`needs:`를 사용하면 dev job이 성공한 후에만 prod job이 시작됨.

### 2. Manual Approval Gate

#### GitHub Environments

GitHub > Settings > Environments에서 환경별 보호 규칙 설정.

**Environment 설정 항목:**

| 항목 | 설명 |
|------|------|
| Required reviewers | 배포 승인자 지정 (최대 6명) |
| Wait timer | 승인 후 대기 시간 (분) |
| Deployment branches | 배포 허용 브랜치 제한 |
| Environment secrets | 환경별 시크릿 |
| Environment variables | 환경별 변수 |

**Required Reviewers:**
- prod 환경에 1명 이상의 승인자를 지정
- 워크플로우가 prod environment에 도달하면 실행이 일시 중지
- 지정된 승인자가 GitHub UI에서 "Approve and deploy" 클릭 후 계속 진행
- 72시간 내 승인하지 않으면 자동 거부

#### Workflow YAML에서 Environment 사용

```yaml
jobs:
  deploy-prod:
    runs-on: ubuntu-latest
    environment:
      name: prod
      url: https://prod.example.com   # 선택: 배포 URL
    steps:
      - run: terraform apply -input=false -auto-approve
```

`environment:` 키로 GitHub Environment을 지정. 해당 Environment에 Required reviewers가 설정되어 있으면 자동으로 승인 대기 상태가 됨.

**Approval UI:**
- GitHub Actions 실행 화면에서 "Waiting for review" 배지 표시
- 승인자에게 알림 발송
- "Review deployments" 버튼 → 환경 선택 → "Approve and deploy" 또는 "Reject"
- 승인/거부 사유 코멘트 가능

#### Environment Secrets

환경별로 다른 AWS Role ARN 등을 저장:

```yaml
# dev: secrets.AWS_ROLE_ARN → arn:aws:iam::111:role/dev-terraform
# prod: secrets.AWS_ROLE_ARN → arn:aws:iam::222:role/prod-terraform

- uses: aws-actions/configure-aws-credentials@v6
  with:
    role-to-assume: ${{ secrets.AWS_ROLE_ARN }}  # 환경별 자동 바인딩

### 3. Concurrency와 Locking

#### S3 Native Locking + CI

**Terraform 레벨 (State locking):**

```hcl
terraform {
  backend "s3" {
    bucket       = "tf-core-tfstate"
    key          = "gallery/dev/terraform.tfstate"
    region       = "ap-northeast-2"
    use_lockfile = true
  }
}
```

- `use_lockfile = true` (TF 1.10+): S3에 `{key}.tflock` 파일을 생성하여 잠금
- DynamoDB 테이블 불필요 (DynamoDB 기반 잠금은 deprecated, 향후 제거 예정)
- 동일 State에 대한 동시 apply 방지

**GitHub Actions 레벨 (Concurrency group):**

```yaml
concurrency:
  group: terraform-${{ matrix.environment }}
  cancel-in-progress: false
```

- 환경별 concurrency group으로 같은 환경에 대한 동시 워크플로우 실행 방지
- `cancel-in-progress: false` — 실행 중인 apply를 취소하지 않음 (State 손상 방지)
- 새 워크플로우는 큐에서 대기

**이중 보호:**
- CI concurrency: 같은 환경에 대한 워크플로우 동시 실행 방지 (1차)
- S3 lockfile: terraform 명령 수준에서 State 동시 접근 방지 (2차)
- 둘 다 설정하는 것이 안전

**Lock 충돌 처리:**
```bash
# 잠금 강제 해제 (비상 시에만)
terraform force-unlock LOCK_ID
```

CI에서 잠금 충돌 발생 시: 워크플로우가 에러로 실패. 이전 실행이 비정상 종료되었다면 `force-unlock` 수동 실행 필요.

### 4. Drift Detection

#### Scheduled Workflow

```yaml
name: Drift Detection

on:
  schedule:
    - cron: '0 9 * * 1-5'   # 평일 09:00 UTC
  workflow_dispatch: {}       # 수동 실행도 가능

permissions:
  contents: read
  id-token: write
  issues: write               # Issue 생성용

jobs:
  detect-drift:
    strategy:
      matrix:
        environment: [dev, prod]
    runs-on: ubuntu-latest
    defaults:
      run:
        working-directory: ./infra/envs/${{ matrix.environment }}
    steps:
      - uses: actions/checkout@v4
      - uses: hashicorp/setup-terraform@v4
        with:
          terraform_version: "1.14.6"
      - uses: aws-actions/configure-aws-credentials@v6
        with:
          role-to-assume: ${{ secrets.AWS_ROLE_ARN }}
          aws-region: ap-northeast-2

      - run: terraform init -input=false

      - name: Detect Drift
        id: plan
        run: terraform plan -input=false -detailed-exitcode -no-color
        continue-on-error: true

      - name: Create Issue on Drift
        if: steps.plan.outputs.exitcode == '2'
        uses: actions/github-script@v7
        with:
          script: |
            await github.rest.issues.create({
              owner: context.repo.owner,
              repo: context.repo.repo,
              title: `🔍 Drift detected in ${{ matrix.environment }}`,
              body: `## Drift Detection Report\n\n` +
                    `**Environment:** ${{ matrix.environment }}\n` +
                    `**Detected at:** ${new Date().toISOString()}\n\n` +
                    `### Plan Output\n\`\`\`\n${`${{ steps.plan.outputs.stdout }}`}\n\`\`\`\n\n` +
                    `> Review the drift and determine if correction is needed.`,
              labels: ['drift', '${{ matrix.environment }}']
            });
```

#### terraform plan -detailed-exitcode

| Exit Code | 의미 | Drift 판단 |
|-----------|------|-----------|
| 0 | 변경 없음 (State와 실제 인프라 일치) | ❌ drift 없음 |
| 1 | 에러 발생 | ⚠️ 검사 실패 |
| 2 | 변경 존재 (State와 실제 인프라 불일치) | ✅ **drift 감지** |

`continue-on-error: true` 필수 — exit code 2는 GitHub Actions에서 실패로 처리되므로, 이후 step(Issue 생성)이 실행되려면 에러를 무시해야 함.

#### 알림 방법

**방법 1: GitHub Issue 생성** (위 예시)
- 별도 도구 불필요
- Issue label로 환경별 분류
- 담당자 자동 할당 가능

**방법 2: Slack Webhook**

```yaml
- name: Slack Notification
  if: steps.plan.outputs.exitcode == '2'
  run: |
    curl -X POST -H 'Content-type: application/json' \
      --data '{"text":"🔍 Drift detected in ${{ matrix.environment }}"}' \
      ${{ secrets.SLACK_WEBHOOK_URL }}
```

### 5. Terraform Cloud / HCP Terraform 대안 (간략 비교)

| 항목 | Self-hosted (GitHub Actions) | Terraform Cloud / HCP |
|------|------------------------------|----------------------|

| 승인 워크플로우 | GitHub Environments | Sentinel Policy + Run approval |

**이 시리즈의 선택:**
Self-hosted GitHub Actions 방식을 다룬다. Terraform의 CI/CD 메커니즘을 직접 구축하면서 각 구성 요소의 역할을 이해하는 것이 목적이다. Terraform Cloud는 이 과정을 추상화하므로 학습 관점에서는 GitHub Actions 직접 구성이 적합하다.

## Draft 작성 시 유의사항

1. **OIDC 설정은 Terraform 코드(HCL)로 보여준다** — IAM OIDC Provider, IAM Role, Trust Policy를 `aws_iam_openid_connect_provider`, `aws_iam_role` 리소스로 작성. 이 시리즈는 Terraform 도구 학습이므로, AWS 콘솔 클릭 안내 대신 HCL로 인프라를 코드화하는 접근이 일관적이다.

2. **GitHub Actions YAML은 별도 언어** — HCL이 아닌 YAML 파일이므로, Terraform 코드와 명확히 구분하여 제시한다.

3. **Lab 구성 제안:**
   - 09.05 lab01: OIDC Provider + IAM Role을 TF로 생성 → workflow YAML 작성 → PR plan comment
   - 09.05 lab02: merge-to-apply workflow + concurrency
   - 09.06 lab01: 품질 게이트 파이프라인 (fmt + validate + tflint + tfsec) + pre-commit
   - 09.06 lab02: branch protection 설정
   - 09.07 lab01: matrix 기반 멀티 환경 배포 + GitHub Environments approval
