## 1. variable validation (Ch02 recap + 고급 패턴)

### 기본 구조

```hcl
variable "environment" {
  type    = string
  default = "dev"

  validation {
    condition     = contains(["dev", "stg", "prod"], var.environment)
    error_message = "Environment must be dev, stg, or prod."
  }
}
```

### 핵심 규칙

| 항목 | 내용 |
|------|------|
| 위치 | `variable` 블록 내부 |
| 실행 시점 | input 처리 단계 (plan 이전) |
| 참조 범위 | **TF < 1.9**: 자기 자신의 변수만 (`var.{self}`) |
| | **TF 1.9+**: 같은 모듈 내 다른 input variable, data source 등 참조 가능 |
| 복수 블록 | 하나의 variable에 여러 `validation` 블록 선언 가능 |
| 실패 동작 | **error** — 즉시 중단 |
| 최소 버전 | Terraform v0.13.0 |

### TF 1.9 변경: cross-reference 허용

v1.9.0 changelog 원문:
> "Input variable validation rules can refer to other objects: Previously input variable validation rules could refer only to the variable being validated. Now they are general expressions, similar to those elsewhere in a module, which can refer to other input variables and to other objects such as data resources."

- 기존: `condition`에서 `var.{자기 자신}`만 참조 가능
- v1.9+: 같은 모듈의 다른 `var.*`, `data.*` 등 참조 가능
- 단, **Ch09 시점(TF 1.14)에서는 이미 사용 가능**

### error_message 표현식

공식 문서:
> "The error_message argument supports any expression that evaluates to a string. This includes literal strings, heredocs, and template expressions."

- `format()` 함수로 null, list, map 등을 포맷팅 가능
- TF 1.9+에서 cross-reference가 허용되면서 error_message에서도 다른 변수 참조 가능
- **주의**: condition이 false인데 error_message가 아직 unknown인 경우(다른 변수가 plan 단계에서만 확정) "Failed to evaluate condition error message" 발생 가능

### can() 함수

공식 문서:
> "can evaluates the given expression and returns a boolean value indicating whether the expression produced a result without any errors."

- **주 용도**: variable validation의 condition에서 사용
- 동작: 표현식 평가 성공 → `true`, 에러 발생 → `false`
- `try()` 와 달리 validation 전용으로 권장

```hcl
# can() + regex() 패턴
variable "image_id" {
  type = string

  validation {
    condition     = can(regex("^ami-", var.image_id))
    error_message = "The image_id must start with \"ami-\"."
  }
}
```

```hcl
# can() + formatdate() 패턴
variable "timestamp" {
  type = string

  validation {
    condition     = can(formatdate("", var.timestamp))
    error_message = "The timestamp must be a valid RFC 3339 date string."
  }
}
```

- **제한**: 선언되지 않은 값(undeclared locals 등)에 대한 참조 에러는 catch하지 못함. 런타임 에러만 처리

### 고급 패턴 예시

```hcl
# 복수 validation 블록
variable "env" {
  type = string

  validation {
    condition     = contains(["dev", "stg", "prod"], var.env)
    error_message = "env must be dev, stg, or prod."
  }

  validation {
    condition     = length(var.env) <= 4
    error_message = "env must be 4 characters or fewer."
  }
}
```

```hcl
# CIDR 검증
variable "vpc_cidr" {
  type = string

  validation {
    condition     = can(cidrhost(var.vpc_cidr, 0))
    error_message = "vpc_cidr must be a valid CIDR notation."
  }
}
```

```hcl
# 복합 object 검증
variable "instance_config" {
  type = object({
    type   = string
    count  = number
  })

  validation {
    condition     = var.instance_config.count > 0 && var.instance_config.count <= 10
    error_message = "count must be between 1 and 10."
  }
}
```

## 2. precondition

### 기본 구조

```hcl
resource "aws_instance" "web" {
  ami           = data.aws_ami.amazon_linux.id
  instance_type = var.instance_type

  lifecycle {
    precondition {
      condition     = data.aws_ami.amazon_linux.architecture == "x86_64"
      error_message = "The selected AMI must be for the x86_64 architecture."
    }
  }
}
```

### 배치 위치

| 블록 유형 | 위치 |
|-----------|------|
| resource | `lifecycle {}` 블록 내부 |
| data source | `lifecycle {}` 블록 내부 |
| output | **직접** output 블록 내부 (lifecycle 없음) |

**output 블록에서의 precondition:**

```hcl
output "instance_public_ip" {
  value = aws_instance.web.public_ip

  precondition {
    condition     = aws_instance.web.associate_public_ip_address
    error_message = "The instance must have a public IP address."
  }
}
```

### 핵심 규칙

| 항목 | 내용 |
|------|------|
| 실행 시점 | plan 생성 후, resource 생성/변경 **전** |
| 참조 범위 | 다른 resource, variable, local, data source 모두 참조 가능 |
| `self` 참조 | 불가 (아직 생성 전이므로) |
| 복수 블록 | 하나의 lifecycle에 여러 precondition 선언 가능 |
| 실패 동작 | **error** — 즉시 중단, apply 차단 |
| 최소 버전 | Terraform v1.2.0 |

### variable validation과의 차이

- variable validation: 단일 변수 값만 검증 (TF 1.9+ 이전)
- precondition: 여러 값을 조합한 교차 검증 가능
- 예: "이 AMI는 x86_64여야 하고, 선택한 instance_type과 호환되어야 한다"

### 참고 URL

- https://developer.hashicorp.com/terraform/language/expressions/custom-conditions#preconditions-and-postconditions
- https://developer.hashicorp.com/terraform/language/meta-arguments/lifecycle

## 3. postcondition

### 기본 구조

```hcl
resource "aws_instance" "web" {
  ami                         = data.aws_ami.amazon_linux.id
  instance_type               = var.instance_type
  associate_public_ip_address = true

  lifecycle {
    postcondition {
      condition     = self.public_ip != ""
      error_message = "The instance must receive a public IP address."
    }
  }
}
```

### 배치 위치

| 블록 유형 | 위치 |
|-----------|------|
| resource | `lifecycle {}` 블록 내부 |
| data source | `lifecycle {}` 블록 내부 |
| output | **직접** output 블록 내부 (lifecycle 없음) |

### 핵심 규칙

| 항목 | 내용 |
|------|------|
| 실행 시점 | resource 생성/변경 또는 data source 읽기 **후** |
| 참조 범위 | `self`로 해당 리소스의 결과 속성 참조 가능 |
| `self` 참조 | **가능** — postcondition의 핵심 기능 |
| 복수 블록 | 하나의 lifecycle에 여러 postcondition 선언 가능 |
| 실패 동작 | **error** — 즉시 중단, 후속 리소스 apply 차단 |
| 최소 버전 | Terraform v1.2.0 |

### self 참조 패턴

```hcl
# data source에서의 postcondition
data "aws_ami" "amazon_linux" {
  most_recent = true
  owners      = ["amazon"]

  filter {
    name   = "name"
    values = ["al2023-ami-*-x86_64"]
  }

  lifecycle {
    postcondition {
      condition     = self.tags["Component"] == "nomad-server"
      error_message = "tags[\"Component\"] must be \"nomad-server\"."
    }
  }
}
```

### Use Case

- 리소스 생성 후 기대 결과 검증
  - public IP가 할당되었는지
  - 태그가 올바르게 설정되었는지
  - 특정 상태가 되었는지
- 잘못된 리소스가 생성된 경우 조기에 발견하여 후속 리소스로의 전파 차단

### 참고 URL

- https://developer.hashicorp.com/terraform/language/expressions/custom-conditions#preconditions-and-postconditions
- https://developer.hashicorp.com/terraform/language/meta-arguments/lifecycle

## 4. check 블록 (TF 1.5+)

### 기본 구조

```hcl
check "health_check" {
  data "http" "app" {
    url = "http://${aws_instance.web.public_ip}:8080/health"
  }

  assert {
    condition     = data.http.app.status_code == 200
    error_message = "${data.http.app.url} returned an unhealthy status code."
  }
}
```

### 핵심 규칙

| 항목 | 내용 |
|------|------|
| 위치 | **top-level 블록** (resource/data 밖, 독립) |
| 실행 시점 | plan/apply의 **마지막 단계** (모든 리소스 처리 후) |
| 실패 동작 | **warning** — apply를 차단하지 않음 |
| scoped data source | check 블록 내부에 선언한 data source는 해당 블록에서만 사용 가능 |
| 복수 assert | 하나의 check 블록에 여러 `assert` 블록 선언 가능 |
| 외부 참조 | 블록 외부의 resource, variable, output, data source 참조 가능 |
| 최소 버전 | Terraform v1.5.0 |

### Scoped Data Source

- check 블록 내부에 선언된 `data` 블록은 해당 check 블록 scope에서만 접근 가능
- 블록 외부에서는 참조 불가
- 이 data source의 실패도 warning으로 처리 (check 블록의 비차단 원칙 유지)

### Data Source 없는 check 블록

data source 없이 기존 리소스를 직접 assert할 수도 있다:

```hcl
check "instance_has_public_ip" {
  assert {
    condition     = aws_instance.web.public_ip != ""
    error_message = "The web instance does not have a public IP."
  }
}
```

### HCP Terraform 연속 검증

HCP Terraform에서 health check를 활성화하면 check 블록이 정기적으로 실행되어 인프라 상태를 지속 모니터링한다.

### 참고 URL

- https://developer.hashicorp.com/terraform/language/checks
- https://developer.hashicorp.com/terraform/language/expressions/custom-conditions

## 5. 실행 순서 / 타이밍

공식 문서 원문:
1. "Terraform executes input variable validations immediately, before generating a plan"
2. "Terraform executes preconditions after generating a plan but before creating the resource"
3. "Terraform executes postconditions after planning and applying changes"
4. "Terraform executes checks at the end of plan and apply operations"

```text
terraform plan/apply 실행 흐름:

  ┌─────────────────────────────────────────────────────────┐
  │ 1. Input Processing                                     │
  │    └── variable validation  ← 즉시 실행, error 시 중단  │
  ├─────────────────────────────────────────────────────────┤
  │ 2. Plan Generation                                      │
  │    └── precondition  ← 리소스별 plan 전, error 시 중단   │
  ├─────────────────────────────────────────────────────────┤
  │ 3. Apply (resource 생성/변경)                            │
  │    └── postcondition  ← 리소스별 apply 후, error 시 중단 │
  ├─────────────────────────────────────────────────────────┤
  │ 4. Final Step                                           │
  │    └── check  ← 모든 리소스 처리 후, warning (비차단)     │
  └─────────────────────────────────────────────────────────┘
```

## 6. 비교표

| | validation | precondition | postcondition | check |
|---|---|---|---|---|
| **위치** | `variable {}` 내부 | resource/data: `lifecycle {}` 내부, output: 직접 | resource/data: `lifecycle {}` 내부, output: 직접 | top-level 블록 |
| **실행 시점** | input 처리 (plan 이전) | plan 생성 후, apply 전 | apply 후 | plan/apply 마지막 단계 |
| **참조 범위** | 자기 변수 (TF 1.9+: 다른 객체도 가능) | 다른 resource, var, local, data | `self` (해당 리소스 결과 속성) | 외부 resource, var, data, output |
| **`self` 사용** | 불가 | 불가 | **가능** | 불가 |
| **실패 동작** | error (중단) | error (중단) | error (중단) | **warning (비차단)** |
| **scoped data** | 불가 | 불가 | 불가 | **가능** |
| **복수 블록** | 가능 | 가능 | 가능 | assert 복수 가능 |
| **최소 버전** | v0.13.0 | v1.2.0 | v1.2.0 | v1.5.0 |
| **Use Case** | 입력값 형식/범위 검증 | apply 전 교차 조건 검증 | 생성 결과 검증 | 상시 헬스 체크, drift 감지 |

## 7. TF 1.14.x 관련 사항

### validation / precondition / postcondition / check 변경

v1.14.0~v1.14.5 changelog 확인 결과: **변경 없음**. 이 네 가지 메커니즘은 1.14.x에서 그대로 유지된다.

### action_trigger (TF 1.14 신규)

v1.14.0에서 도입된 새 lifecycle 메커니즘. lifecycle guards와는 별개 기능이다.

```hcl
resource "aws_instance" "web" {
  # ...

  lifecycle {
    action_trigger {
      events    = ["after_create", "after_update"]
      condition = <expression>  # optional
      actions   = [<action references>]
    }
  }
}
```

- **목적**: CRUD 외 side-effect 작업을 리소스 lifecycle에 연결 (예: Lambda invoke, CloudFront invalidation)
- **이벤트**: `before_create`, `after_create`, `before_update`, `after_update`
- **v1.14.1 수정**: after_create / after_update가 리소스 apply 완료 후 실행되도록 수정
- **이 섹션과의 관계**: lifecycle guards (검증)와는 목적이 다름. action_trigger는 "액션 실행"이지 "조건 검증"이 아님. 09.02에서는 언급 수준으로만 다루고 별도 섹션에서 다룰 주제

- https://github.com/hashicorp/terraform/blob/v1.14.5/CHANGELOG.md
- https://developer.hashicorp.com/terraform/language/meta-arguments/lifecycle

## 8. output 블록에서의 precondition / postcondition 특이 사항

output 블록은 `lifecycle {}` 래퍼 없이 precondition/postcondition을 직접 선언한다:

```hcl
output "instance_ip" {
  value = aws_instance.web.public_ip

  precondition {
    condition     = aws_instance.web.associate_public_ip_address
    error_message = "Instance must have public IP enabled."
  }
}
```

이는 resource/data source와 다른 점이다. resource/data는 반드시 `lifecycle {}` 내부에 배치해야 한다.

## 9. 추가 참고: ephemeral 리소스

lifecycle 문서에서 precondition/postcondition이 `ephemeral` 블록에서도 사용 가능하다고 명시:
> "Available in data, ephemeral, and resource blocks."

ephemeral 리소스는 TF 1.10+에서 도입된 기능으로, 이 섹션에서는 다루지 않는다.
