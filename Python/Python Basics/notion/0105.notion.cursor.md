# 1.5 __main__과 실행 진입점

## 개요

파이썬에는 명시적인 `main()` 진입점 함수가 없다. **모듈 전체가 실행**되고, 그 모듈이 "직접 실행된 것인지" "import된 것인지"를 `__name__`으로 구분한다. `if __name__ == "__main__":` 블록은 "직접 실행될 때만" 동작하는 코드를 두는 표준 패턴이다.

이 패턴을 이해하지 못하면 라이브러리로 쓰일 모듈을 `import`할 때 의도치 않은 부작용(전체 코드 실행, 부수 효과 등)이 발생한다. 반대로, 이 패턴을 제대로 활용하면 **같은 파일에서 라이브러리(import용)와 CLI(실행용)** 를 깔끔하게 분리할 수 있다. `python -m pkg`로 패키지를 실행할 때 `pkg/__main__.py`가 진입점이 되는 메커니즘도 이 `__name__` 모델 위에 올라간다.

이 주제를 이해하면 Ch8(모듈과 패키지)에서 다루는 패키지 구조, 상대 import, `__init__.py`와 자연스럽게 연결된다.

---

## 핵심 개념

### __name__의 두 가지 값

모든 모듈에는 `__name__`이라는 내장 속성이 있다. 이 값은 **그 모듈이 어떻게 로드되었는가**에 따라 달라진다.

| 로드 방식 | `__name__` 값 |
|-----------|---------------|
| **직접 실행** | `'__main__'` |
| **import** | 모듈의 정규화된 이름 (예: `'foo'`, `'pkg.mod'`) |

직접 실행에는 REPL, `python script.py`, `python -m pkg.mod` 세 가지가 모두 해당한다. 이때 실행되는 "최상위 모듈"의 `__name__`은 `'__main__'`이다.

### if __name__ == "__main__": 패턴

```python
# 모듈 레벨 코드: import 시에도 실행됨
def useful_func():
    return "라이브러리로 쓸 함수"

# 이 블록: 직접 실행 시에만 실행됨
if __name__ == "__main__":
    print(useful_func())
    # CLI 로직, 테스트 코드 등
```

- **import될 때**: `__name__`은 `'foo'`(모듈 이름)이므로 블록이 실행되지 않는다. `useful_func`만 정의되고, 호출자는 원하는 대로 사용한다.
- **직접 실행될 때**: `__name__`은 `'__main__'`이므로 블록이 실행된다. CLI 진입점, 간단한 테스트 등이 여기 들어간다.

이 패턴은 **라이브러리와 실행 스크립트를 한 파일에 공존**시키는 파이썬의 관용적 방식이다. 진입점이 "특정 함수"가 아니라 "특정 조건에서 실행되는 블록"이라는 점이 파이썬의 특징이다.

### __main__.py와 python -m pkg

패키지에 `__main__.py` 파일이 있으면, `python -m pkg` 실행 시 이 파일이 **`__main__` 모듈**로 실행된다.

**프로젝트 구조 예:**

```
my_pkg/
├── __init__.py
├── __main__.py      # python -m my_pkg 시 진입점
└── core.py
```

**`my_pkg/__main__.py`:**

```python
# python -m my_pkg 로 실행될 때 __name__ == "__main__"
from .core import main

if __name__ == "__main__":
    main()
```

`python -m my_pkg`를 실행하면:

1. `sys.path`에서 `my_pkg` 패키지를 찾는다.
2. `my_pkg/__main__.py`가 있으면, 이 파일을 `__main__`이라는 이름으로 실행한다.
3. 따라서 `__main__.py` 내부에서 `__name__`은 `'__main__'`이다.

`__main__.py`가 없으면 `python -m my_pkg`는 `ModuleNotFoundError` 또는 패키지가 로드되지 않는다(패키지 구조에 따라 다름). **패키지에 CLI를 붙이는 표준 방식**이 `__main__.py`다.

### runpy.run_module()의 동작

`python -m pkg.mod`는 내부적으로 `runpy.run_module()`을 사용한다고 볼 수 있다.

```python
import runpy
runpy.run_module('pkg.mod', run_name='__main__')
```

이 함수는:

1. `pkg.mod`를 **import**한다. 의존성 해결, `__init__.py` 실행 등이 일어난다.
2. 그 모듈의 코드를 **`__main__`이라는 이름으로 다시 실행**한다. 이때 `__name__`이 `'__main__'`으로 설정된다.

그래서 `-m` 실행은 "import된 것처럼" 로드되면서 동시에 "직접 실행된 것처럼" `if __name__ == "__main__":` 블록이 실행된다.

## 코드 예제

### 1) __name__의 차이 확인 (lab01)

**`lab01/dual_role.py`:**

```python
# lab01/dual_role.py
def greet(name: str) -> str:
    return f"Hello, {name}!"

if __name__ == "__main__":
    print(greet("World"))
```

**직접 실행:**

```bash
$ python lab01/dual_role.py
Hello, World!
```

**import 시:**

```python
# other_script.py
from lab01.dual_role import greet
print(greet("Python"))  # Hello, Python!
# "Hello, World!"는 출력되지 않음
```

**아하 모먼트**: 같은 파일이 "실행 스크립트"와 "라이브러리" 두 역할을 한다. `__name__`이 그 역할을 나눈다.

### 2) __main__.py로 패키지 CLI 만들기 (lab02)

**프로젝트 구조:**

```
lab02/
├── calc/
│   ├── __init__.py
│   ├── __main__.py
│   └── core.py
```

**`calc/core.py`:**

```python
def add(a: int, b: int) -> int:
    return a + b

def main() -> None:
    import sys
    if len(sys.argv) >= 3:
        x, y = int(sys.argv[1]), int(sys.argv[2])
        print(add(x, y))
    else:
        print("Usage: python -m calc 1 2")
```

**`calc/__main__.py`:**

```python
from .core import main

if __name__ == "__main__":
    main()
```

**실행:**

```bash
$ cd lab02
$ python -m calc 3 5
8
```

**아하 모먼트**: `python -m calc`는 `calc/__main__.py`를 진입점으로 실행한다. 패키지를 CLI 도구처럼 쓸 수 있다.

### 3) runpy.run_module()로 -m 동작 흉내 내기 (lab03)

```python
# lab03/run_as_main.py
import runpy
import sys
from pathlib import Path

# lab02를 path에 추가하여 calc 패키지를 찾을 수 있게 함
lab02 = Path(__file__).resolve().parent.parent / "lab02"
sys.path.insert(0, str(lab02))

if __name__ == "__main__":
    # python lab03/run_as_main.py calc 3 5 와 같이 실행
    if len(sys.argv) < 2:
        print("Usage: python run_as_main.py <module> [args...]")
        sys.exit(1)
    mod_name = sys.argv[1]
    # calc.main()이 sys.argv[1], [2]를 사용하므로 argv 조정
    old_argv = sys.argv.copy()
    sys.argv = [f"-m {mod_name}"] + sys.argv[2:]
    try:
        runpy.run_module(mod_name, run_name="__main__")
    finally:
        sys.argv = old_argv
```

```bash
$ python lab03/run_as_main.py calc 3 5
8
```

**아하 모먼트**: `runpy.run_module()`이 `-m` 옵션의 핵심이다. 프로그래밍 방식으로 "모듈을 __main__으로 실행"할 수 있다.

### 4) __main__.py 없이 -m 실행 시

**`lab02/calc/`에서 `__main__.py` 삭제 후:**

```bash
$ python -m calc 3 5
# Python 3.x: 패키지에 __main__ 서브모듈이 없으면 에러
# __main__.py가 있어야 python -m pkg 가 동작함
```

**아하 모먼트**: `__main__.py`는 "패키지를 `-m`으로 실행할 때의 진입점"을 정의한다. 없으면 `python -m pkg`가 실패하거나 예상과 다르게 동작한다.

## 주의사항 (Pitfalls)

### 1) if __name__ == "__main__": 블록 없이 import 시 부작용

`if __name__ == "__main__":` 블록 없이 모듈 레벨에 실행 코드를 두면, 그 모듈을 `import`할 때마다 그 코드가 실행된다.

```python
# 나쁜 예
def helper():
    return 42

print(helper())  # import 시에도 실행됨!
```

라이브러리로 쓰일 모듈에서는 이런 부수 효과를 피해야 한다. CLI/테스트 코드는 `if __name__ == "__main__":` 안에 두자.

### 2) 진입점 함수에 대한 오해

파이썬에서는 **파일 전체**가 실행된다. `if __name__ == "__main__":` 블록은 "조건부 실행"일 뿐, 별도의 진입점 함수가 아니다. CLI 인자는 `sys.argv`로 받고, `argparse`나 `click` 같은 라이브러리를 쓰면 `main()` 함수를 관례적으로 만들 수 있지만, 그건 언어 차원의 진입점이 아니다.

### 3) __main__.py는 패키지 전용

일반 모듈(`foo.py`)에는 `__main__.py`가 없다. `__main__.py`는 **패키지 디렉터리** 안에 둔다. `python -m foo`로 단일 모듈을 실행할 때는 `foo.py` 자체가 `__main__`으로 실행되며, `__main__.py`는 사용되지 않는다.

### 4) __main__.py에서의 상대 import

`__main__.py`는 패키지의 일부이므로, 같은 패키지 내 모듈을 **상대 import**로 가져올 수 있다.

```python
# pkg/__main__.py
from .core import main  # OK
# from core import main  # 패키지 구조에 따라 실패할 수 있음
```

Ch8에서 상대 import를 다루지만, `__main__.py`에서는 `.`으로 시작하는 상대 import가 일반적이다.

### 5) -m 실행 시 sys.argv

`python -m calc 3 5` 실행 시 `sys.argv`는 `['-m', '3', '5']`가 아니라 `['/path/to/calc/__main__.py', '3', '5']`와 같이, **실행되는 스크립트 경로**가 `argv[0]`에 들어간다. `-m`은 `argv`에 남지 않는다.

## Deep Insight

### __main__은 "가상 모듈 이름"이다

`__main__`은 `sys.modules`에 등록되는 특별한 모듈 이름이다. REPL, `python script.py`, `python -m pkg.mod` 모두 "어떤 코드를 `__main__`이라는 이름으로 실행하는가"의 문제다. `import __main__`을 하면 현재 `__main__`으로 실행 중인 모듈의 네임스페이스에 접근할 수 있다. 이는 "현재 최상위 실행 컨텍스트"를 가리킨다.

### runpy의 두 가지 함수

`runpy` 모듈에는 `run_path()`와 `run_module()`이 있다.

- **`run_path(path)`**: 파일 경로를 받아 그 파일을 `__main__`으로 실행. `python script.py`와 유사.
- **`run_module(name, run_name='__main__')**: 모듈 이름을 받아 import한 뒤 `__main__`으로 실행. `python -m pkg.mod`와 동일.

두 함수 모두 "코드를 `__main__`이라는 이름으로 실행"한다는 공통점이 있다. 파이썬의 실행 모델은 결국 "어떤 코드를 어떤 모듈 이름으로 실행할 것인가"로 귀결된다.

### 패키지 CLI와 pyproject.toml [project.scripts]

`pip install`로 설치된 패키지는 `pyproject.toml`의 `[project.scripts]`에 정의된 콘솔 스크립트를 제공할 수 있다. 예: `[project.scripts] mycmd = "mypkg.__main__:main"`. 이렇게 하면 `mycmd` 명령이 `mypkg.__main__.main()`을 호출한다. `__main__.py`에 `main()` 함수를 두고, 그걸 콘솔 진입점으로 노출하는 구조가 현대적인 패키지의 표준이다. Ch8, `pyproject.toml`에서 더 다룬다.

## 요약

| 항목 | 내용 |
|------|------|
| **__name__** | 직접 실행 시 `'__main__'`, import 시 모듈 이름. `if __name__ == "__main__":`로 라이브러리/CLI 분리. |
| **if __name__ == "__main__":** | "직접 실행될 때만" 실행되는 블록. 같은 파일에서 라이브러리와 실행 스크립트를 공존시키는 표준 패턴. |
| **__main__.py** | 패키지에 있으면 `python -m pkg` 시 진입점. 패키지에 CLI를 붙이는 표준 방식. |
| **runpy.run_module()** | `python -m pkg.mod`의 내부 동작. 모듈을 import한 뒤 `__main__`으로 재실행. |
| **주의** | `if __name__ == "__main__":` 없이 모듈 레벨에 실행 코드를 두면 import 시 부작용. `__main__.py`는 패키지 전용. |
