## 주의사항

### `terraform graph -type` 옵션 — 4가지 (기존 문서와 차이)

과거 Terraform 버전에서는 `-type=plan`과 `-type=apply` 두 가지가 주로 사용되었다. 현재(1.14.x) 공식 문서에서 명시하는 타입은 다음 4가지다:

| 타입 | 설명 |
|------|------|
| `plan` | plan 단계의 상세 그래프 |
| `plan-refresh-only` | refresh-only plan의 그래프 |
| `plan-destroy` | destroy plan의 그래프 |
| `apply` | apply 단계의 상세 그래프 |

**기본 동작**(타입 미지정)은 리소스와 데이터 블록의 의존 관계만 보여주는 단순화된 그래프를 생성한다. `-type`을 지정하면 Terraform 런타임 내부 구현 세부사항이 노출되는 상세 그래프가 출력된다.

draft에서는 학습 목적상 기본 그래프(타입 미지정)를 주로 사용하고, `-type=plan`을 보충 설명으로 다루는 것이 적절하다. `plan-refresh-only`와 `plan-destroy`는 언급만 하고 넘어가도 충분하다.

### `-target`의 의존성 처리 — upstream vs downstream

`-target`은 대상 리소스와 그 **upstream 의존성**(대상이 의존하는 리소스)을 함께 처리한다. 그러나 **downstream 의존자**(대상에 의존하는 리소스)는 포함하지 않는다.

예: A → B → C 의존 체인에서 `-target=B`를 지정하면:
- A (B의 upstream): 함께 처리됨
- C (B의 downstream): 처리되지 않음

이로 인해 인프라 불일치가 발생할 수 있다. draft에서 "의존 관계를 무시한다"는 표현보다 "대상과 upstream만 처리하고 downstream은 건너뛴다"로 정확하게 서술해야 한다. 03.01 research에서도 동일한 주의를 기록한 바 있다.

### `depends_on`의 성능 영향

공식 문서에서 `depends_on` 사용 시 보수적 실행 계획이 생성된다고 경고한다. 의존 대상의 값이 `"(known after apply)"`로 처리되어 불필요한 리소스 교체가 발생할 수 있다. 가능하면 참조 표현식(암묵적 의존성)을 우선 사용하고, `depends_on`은 참조로 표현할 수 없는 경우에만 사용해야 한다.

이 내용은 02.03에서 `depends_on` 기초를 다루었으므로, 03.02에서는 "왜 암묵적 의존성이 우선인가"를 성능 관점에서 보강하면 좋다.

### `terraform graph`와 `terraform init` 관계

공식 문서에 `terraform graph` 실행 전 `init` 필수 여부를 명시하지 않는다. 실무적으로:

- Provider를 사용하는 설정: `init` 필요 (provider 스키마 로딩 필요)
- Provider 없는 순수 HCL 구조: 기본 그래프(타입 미지정)는 `init` 없이도 동작할 수 있으나, 보장되지 않음

**draft에서는 `terraform init` 후 `terraform graph`를 실행하는 흐름으로 작성하는 것이 안전하다.** Lab에서 AWS provider를 사용하므로 init은 필수다.

### 그래프 빌드 9단계 (internals/graph)

공식 internals 문서에서 설명하는 그래프 빌드 과정:

1. 설정의 리소스 노드 추가 (diff/state 메타데이터 첨부)
2. 리소스를 provisioner에 매핑
3. `depends_on`에서 명시적 의존성 edge 생성
4. orphan 리소스 포함 (state에 있지만 설정에서 제거된 리소스)
5. 리소스를 provider에 매핑, provider 설정 노드 및 의존성 edge 생성
6. interpolation 파싱으로 암묵적 의존성 식별
7. 모든 리소스를 연결하는 root 노드 생성
8. diff 존재 시 destroy/create 노드 분리 (ordering 처리)
9. 비순환 구조 + 단일 root 검증

draft에서 이 9단계를 모두 나열할 필요는 없다. "명시적 의존성(depends_on) → 암묵적 의존성(참조 파싱) → 검증" 수준의 요약이 적절하다. 관심 있는 독자를 위해 internals 문서 링크를 참고 자료로 제공하면 좋다.

### 그래프 노드 유형

| 노드 유형 | 설명 |
|-----------|------|
| Resource Node | 개별 리소스. count 사용 시 인스턴스별 노드 생성 |
| Provider Configuration Node | provider 초기화 (자격 증명 등) |
| Resource Meta-Node | count > 1일 때 리소스 그룹핑 노드 |

기본 그래프(타입 미지정)에서는 Resource Node와 의존 edge만 표시된다. `-type` 지정 시 Provider Configuration Node 등 내부 노드가 함께 표시된다.

### Lab 리소스 구성 — SSM 패턴 검증

plan에서 Lab 리소스로 SG + IAM Role + Instance Profile + Policy Attachment + EC2를 사용한다. 이 구성의 의존성 체인:

```text
aws_iam_role
  └─ aws_iam_role_policy_attachment (role ARN 참조 → 암묵적)
  └─ aws_iam_instance_profile (role name 참조 → 암묵적)
       └─ aws_instance (iam_instance_profile 참조 → 암묵적)
            └─ aws_security_group (vpc_security_group_ids 참조 → 암묵적)

aws_instance → depends_on → aws_iam_role_policy_attachment (명시적)
```

EC2 인스턴스는 `aws_iam_instance_profile`을 참조하므로 instance profile에 대한 암묵적 의존성이 있다. 그러나 `aws_iam_role_policy_attachment`는 EC2에서 직접 참조하지 않으므로 `depends_on`이 필요하다. 이 패턴은 02.03 lab03에서 이미 검증되었다.

이 구성은 암묵적 의존성과 명시적 의존성을 모두 보여주므로 `terraform graph` 시각화 Lab에 적합하다.

### `-parallelism` 적용 범위

`-parallelism=n` 플래그는 plan, apply, destroy 세 명령 모두에서 사용 가능하다. 기본값은 10이다. Terraform은 DAG를 depth-first로 순회하면서 세마포어로 동시 실행 수를 제한한다. 모든 의존성이 완료된 노드는 즉시 실행을 시작한다.
