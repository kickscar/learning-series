# review/content — 이론 섹션 검토

## 목적

이론/개념 섹션 draft의 구조·규칙 준수 여부를 검토한다.

## 체크리스트

### 1. 구조 준수

- [ ] 파일명: `{ch_no}.{sec_no}.claude.notion.md`
- [ ] 문서 시작: 챕터/섹션 제목 없이 개요 문단으로 시작
- [ ] 헤딩 계층: H1 번호 없음 → H2 아라비아 번호 → H3 원숫자
- [ ] 헤딩에 볼드 없음
- [ ] 마무리: `# 핵심 정리` + `# 참고 자료` 존재

### 2. 내용 완성도

- [ ] 개요 문단 존재 (1~2문장)
- [ ] 3개 이상 개념/주제 다룸
- [ ] HCL 또는 CLI 예시 포함
- [ ] 다음 섹션/챕터로의 연결 언급

### 3. 시리즈 원칙

- [ ] Azure 플랫폼 중심 (Terraform은 수단)
- [ ] Azure 서비스 자체의 기초 설명 없음 (Fundamentals 범위)
- [ ] 설계 당위 없음 (Architecture & Design 범위)
- [ ] TF 심화 없음 (Terraform Core 범위)

### 4. 코드 블록 규칙

- [ ] 언어 지정: `hcl`, `bash`, `text`, `json`, `yaml`, `mermaid`
- [ ] 인라인 코드: 파일명, 속성, 명령어에 백틱

### 5. 한국어 작문

- [ ] 영문 기술 용어 유지
- [ ] 첫 등장 시 한국어 병기
- [ ] ~다 체 사용
- [ ] 금지 문체 없음

### 6. 스타일 가이드

- [ ] 설정값: **`값`** (볼드+백틱)
- [ ] 리소스 주소: `azurerm_xxx.name` (백틱)
- [ ] 참고 자료 형식 준수

### 7. 다이어그램

- [ ] Mermaid 사용 시 `<br>` 줄바꿈
- [ ] 다이어그램 후 2~4줄 설명
- [ ] 복잡한 구조는 `[이미지: ...]` 플레이스홀더

## 판정

- **OK**: 수정 불필요
- **수정 필요**: 위치 + 사유 명시
- **기술 오류**: 플래그 (사용자 확인 필요)
