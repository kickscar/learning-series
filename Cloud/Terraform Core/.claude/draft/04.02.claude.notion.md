이전 섹션에서 State의 역할과 필요성을 다뤘다. 이번 섹션에서는 `terraform.tfstate` 파일을 열어 내부 구조를 분석한다. 각 필드가 무엇을 의미하는지 이해하면 State 관련 오류를 진단할 때 도움이 된다.

# terraform.tfstate 전체 구조

`terraform.tfstate`는 JSON 형식이다. 최상위 구조는 다음과 같다.

```json
{
  "version": 4,
  "terraform_version": "1.14.7",
  "serial": 3,
  "lineage": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "outputs": { ... },
  "resources": [ ... ],
  "check_results": [ ... ]
}
```

| 필드 | 설명 |
|------|------|
| `version` | State 파일 포맷 버전. 현재 **4** |
| `terraform_version` | 이 State를 마지막으로 수정한 Terraform 버전 |
| `serial` | State 변경 횟수. apply마다 증가 |
| `lineage` | State 이력 체인을 식별하는 UUID |
| `outputs` | `output` 블록의 값 |
| `resources` | 관리 중인 리소스 목록 |
| `check_results` | `check` 블록의 실행 결과 (TF 1.5+) |

---

# serial과 lineage

## 1. serial

`serial`은 State가 변경될 때마다 1씩 증가하는 정수다. `terraform apply`, `terraform state mv`, `terraform import` 등 State를 수정하는 모든 작업에서 증가한다.

Remote Backend에서 State를 저장할 때 serial을 확인한다. 저장하려는 State의 serial이 원격에 저장된 serial보다 낮으면 — 다른 팀원이 먼저 apply한 것이므로 — 저장을 거부한다. Locking과 함께 동시 수정을 방지하는 메커니즘이다.

## 2. lineage

`lineage`는 State가 처음 생성될 때 부여되는 UUID다. 동일한 인프라의 State 이력을 식별한다.

- `terraform init`으로 새 프로젝트를 시작하면 새 lineage가 생성된다
- `terraform init -migrate-state`로 Backend를 이전하면 lineage는 **변하지 않는다** — 같은 인프라의 연속이기 때문이다
- `terraform state push`로 State를 밀어넣을 때 lineage가 다르면 거부된다 (다른 인프라의 State이므로)

# resources 구조

`resources` 배열의 각 항목은 하나의 리소스 블록에 대응한다.

```json
{
  "mode": "managed",
  "type": "aws_security_group",
  "name": "instance_web",
  "provider": "provider[\"registry.terraform.io/hashicorp/aws\"]",
  "instances": [
    {
      "schema_version": 1,
      "attributes": {
        "id": "sg-0abc1234567890def",
        "name": "tf-core-lab01-sg-instance-web",
        "description": "tf-core-lab01 security group",
        "vpc_id": "vpc-xxxxxxxxxxxxxxxxx",
        "ingress": [],
        "egress": [ ... ],
        "tags": {
          "Name": "tf-core-lab01-sg-instance-web"
        },
        "tags_all": {
          "ManagedBy": "Terraform",
          "Name": "tf-core-lab01-sg-instance-web",
          "Project": "tf-core-lab01"
        }
      },
      "sensitive_attributes": [],
      "dependencies": [
        "data.aws_iam_policy_document.ec2_assume_role"
      ]
    }
  ]
}
```

## 1. 리소스 메타데이터

| 필드 | 설명 | 예시 |
|------|------|------|
| `mode` | `"managed"` (resource) 또는 `"data"` (data source) | `"managed"` |
| `type` | 리소스 타입 | `"aws_security_group"` |
| `name` | 코드에서 지정한 이름 (TF 레이블) | `"instance_web"` |
| `provider` | 이 리소스를 관리하는 Provider | `"provider[\"registry.terraform.io/hashicorp/aws\"]"` |

`mode`가 `"managed"`이면 `resource` 블록, `"data"`이면 `data` 블록이다.

## 2. instances

`instances` 배열은 리소스의 실제 인스턴스 정보를 담는다. `count`나 `for_each`를 사용하지 않으면 인스턴스는 하나다.

| 필드 | 설명 |
|------|------|
| `schema_version` | Provider가 정의한 리소스 스키마 버전 |
| `attributes` | 리소스의 모든 속성 값 (인수 + 계산된 속성) |
| `sensitive_attributes` | 민감한 속성의 JSON 경로 목록 |
| `dependencies` | 이 리소스가 의존하는 다른 리소스 주소 |

### ① attributes

`attributes`에는 코드에서 설정한 인수(`name`, `description`)와 AWS가 부여한 속성(`id`, `vpc_id`) 모두 포함된다. plan에서 `(known after apply)`로 표시되던 값이 apply 후 여기에 기록된다.

### ② sensitive_attributes

`sensitive = true`로 선언한 변수나 속성의 JSON 경로를 기록한다. CLI 출력에서 마스킹하는 데 사용된다. 실제 값은 `attributes`에 **plaintext로 그대로 저장**된다 — 마스킹은 출력 레이어에서만 동작한다.

### ③ dependencies

코드에서 리소스를 삭제하면 참조 표현식이 사라진다. 하지만 State에 기록된 `dependencies`로 올바른 삭제 순서를 결정할 수 있다. Ch03 Sec02에서 다룬 의존 관계 그래프의 정보가 State에도 유지되는 것이다.

# outputs 구조

```json
{
  "outputs": {
    "instance_web": {
      "value": {
        "id": "i-0abc1234567890def",
        "public_ip": "13.125.xxx.xxx"
      },
      "type": [
        "object",
        {
          "id": "string",
          "public_ip": "string"
        }
      ],
      "sensitive": false
    }
  }
}
```

`terraform output` 명령은 State 파일의 `outputs` 섹션을 읽는다. `sensitive: true`로 선언한 output은 CLI에서 `<sensitive>`로 마스킹되지만 State 파일에는 실제 값이 저장된다.

# terraform.tfstate.backup

`terraform apply`, `terraform state mv`, `terraform import` 등 State를 수정하는 작업을 실행하면 이전 State가 `terraform.tfstate.backup`으로 자동 저장된다. `terraform plan`은 State를 수정하지 않으므로 backup을 생성하지 않는다.

backup은 **한 세대**만 유지된다. 마지막 변경 직전 상태만 복구할 수 있다. Remote Backend에서 S3 versioning을 사용하면 모든 이력을 보관할 수 있다.

---

# State 파일을 직접 수정하면 안 되는 이유

State 파일은 JSON이므로 텍스트 에디터로 열어 수정할 수 있다. 하지만 직접 수정하면:

- `serial`이 갱신되지 않아 Remote Backend에서 충돌이 발생한다
- 리소스 간 `dependencies`가 불일치하면 삭제 순서가 꼬인다
- `attributes` 값이 실제 인프라와 불일치하면 잘못된 plan이 생성된다

State를 수정해야 할 때는 반드시 `terraform state` 명령을 사용한다. `state mv`, `state rm`, `import` 등이 serial 증가와 의존성 정합성을 자동으로 처리한다. State 명령어는 Sec05에서 다룬다.

디버깅 목적으로 State 내용을 확인할 때는 `terraform state pull`을 사용한다.

```bash
$ terraform state pull | jq '.resources[] | {type, name}'
```

---

# 핵심 정리

- `terraform.tfstate`는 JSON 형식이며, 포맷 버전 4가 현재 기준이다.
- `serial`은 State 변경마다 증가한다. Remote Backend에서 동시 수정을 방지하는 메커니즘이다.
- `lineage`는 State 이력 체인을 식별하는 UUID다. Backend 마이그레이션 시에도 유지된다.
- `resources`의 각 항목에 리소스 메타데이터(`mode`, `type`, `name`), 속성(`attributes`), 의존 관계(`dependencies`)가 기록된다.
- `sensitive_attributes`는 CLI 마스킹용이다. 실제 값은 State에 plaintext로 저장된다.
- State 파일을 직접 수정하지 않는다. `terraform state` 명령으로 안전하게 변경한다.

다음 섹션에서는 Remote Backend(S3)를 구성하고 Local State를 이전한다.
