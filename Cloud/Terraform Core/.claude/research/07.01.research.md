# Research: 07.01 — Workspace 개념

> 조사일: 2026-04-29
> 기준: Terraform >= 1.14.0, hashicorp/aws provider ~> 6.0

## 검증 결과

| 항목 | 상태 | 비고 |
|------|------|------|
| `terraform workspace` 서브커맨드 5종 | ✓ 정확 | new, list, select, show, delete |
| `default` workspace 삭제 불가 | ✓ 정확 | 공식 문서 명시 |
| `${terraform.workspace}` 표현식 | ✓ 정확 | locals, resource, data 등에서 사용 가능 |
| backend config에서 `terraform.workspace` 사용 불가 | ✓ 정확 | backend 블록은 정적 — 변수·표현식 불허 |
| Local backend state 경로 | ✓ 정확 | `terraform.tfstate.d/{workspace}/terraform.tfstate` |
| S3 backend `workspace_key_prefix` 기본값 `env:` | ✓ 정확 | 공식 문서 명시 |
| S3 경로 패턴 `{workspace_key_prefix}/{workspace}/{key}` | ✓ 정확 | default workspace는 `{key}` 직접 사용 |

| `-force` 플래그로 리소스 추적 중인 workspace 삭제 가능 | ✓ 정확 | dangling 리소스 발생 위험 |
| TF 1.14.x workspace 관련 변경사항 | ⚠ 주의 | plan 파일 workspace 교차 적용 시 명시적 에러 추가. 아래 상세 |

## 상세 조사

### 1. Workspace 서브커맨드

`terraform workspace` 명령의 서브커맨드 5종이다.

| 커맨드 | 설명 |
|--------|------|
| `terraform workspace list` | 현재 디렉토리의 workspace 목록 출력. 현재 workspace에 `*` 표시 |
| `terraform workspace new <name>` | 새 workspace 생성 및 자동 전환. `-state=<path>` 옵션으로 기존 state 복사 가능 |
| `terraform workspace select <name>` | 지정 workspace로 전환 |
| `terraform workspace show` | 현재 workspace 이름 출력 |
| `terraform workspace delete <name>` | workspace 삭제. 현재 workspace는 삭제 불가, 리소스 추적 중이면 `-force` 필요 |

#### `default` workspace

- Terraform은 초기화 시 `default` workspace를 자동 생성한다
- `default`는 삭제할 수 없다
- 명시적으로 workspace를 생성하지 않으면 모든 작업이 `default`에서 수행된다
- `terraform workspace show` 출력: `default`

#### `terraform workspace new` 동작

```bash
$ terraform workspace new dev

# 출력 예
Created and switched to workspace "dev"!

You're now on a new, empty workspace. Workspaces isolate their state,
so if you run "terraform plan" Terraform will not see any existing state
for this configuration.
```

새 workspace는 **빈 state**로 시작한다. 기존 리소스가 있는 상태에서 새 workspace로 전환하면 해당 리소스는 "보이지 않는" 상태가 된다 (다른 workspace의 state에만 존재).

### 2. `terraform.workspace` 표현식

`terraform.workspace`는 현재 선택된 workspace 이름을 문자열로 반환하는 내장 참조다.

#### 사용 가능한 위치

- `locals` 블록
- `resource` / `data` 블록의 인수
- `output` 블록의 `value`
- `variable` 블록의 `default` (단, 실무에서 권장하지 않음)
- 조건식, `for` 표현식 등 HCL 표현식이 허용되는 모든 곳

#### 사용 불가능한 위치

- `backend` 블록 — backend config는 `terraform init` 시점에 정적으로 평가되므로 변수, locals, 표현식 일체 사용 불가
- `required_providers` 블록

#### 활용 패턴

**리소스 이름에 workspace 반영:**

```hcl
resource "aws_instance" "web" {
  ami           = data.aws_ami.amazon_linux.id
  instance_type = local.instance_type

  tags = {
    Name = "${local.namespace}-instance-web"  # namespace에 terraform.workspace 포함
  }
}
```

**locals 조건 분기:**

```hcl
locals {
  env_config = {
    dev  = { instance_type = "t3.micro",  desired_count = 1 }
    prod = { instance_type = "t3.small",  desired_count = 2 }
  }

  current_config = local.env_config[terraform.workspace]
  instance_type  = local.current_config.instance_type
}
```

**namespace 조립 (Ch07 기준 — naming-tagging 규칙 반영):**

```hcl
locals {
  org         = "tf-core"
  project     = "lab01"
  environment = terraform.workspace  # var.env 대신 workspace 사용
  namespace   = "${local.org}-${local.project}-${local.environment}"
}
```

> Ch02.05~Ch06에서는 `var.env`로 environment를 주입했다. Ch07 workspace 실습에서는 `terraform.workspace`가 이 역할을 대신한다.

### 3. State 저장 경로

#### Local Backend

default workspace의 state는 작업 디렉토리에 직접 저장된다.

```text
project/
├── main.tf
├── terraform.tfstate          ← default workspace
└── terraform.tfstate.d/
    ├── dev/
    │   └── terraform.tfstate  ← dev workspace
    └── prod/
        └── terraform.tfstate  ← prod workspace
```

- `default` workspace: `terraform.tfstate` (루트)
- 사용자 정의 workspace: `terraform.tfstate.d/{workspace}/terraform.tfstate`

#### S3 Backend

S3 backend에서 workspace별 state 경로는 `workspace_key_prefix`와 `key`로 결정된다.

**backend 설정 예:**

```hcl
terraform {
  backend "s3" {
    bucket               = "tf-core-tfstate"
    key                  = "gallery/terraform.tfstate"
    region               = "ap-northeast-2"
    workspace_key_prefix = "env:"    # 기본값
    dynamodb_table       = "tf-core-tfstate-lock"
  }
}
```

**경로 패턴:**

| Workspace | S3 Key |
|-----------|--------|
| `default` | `gallery/terraform.tfstate` |
| `dev` | `env:/dev/gallery/terraform.tfstate` |
| `prod` | `env:/prod/gallery/terraform.tfstate` |

- `default` workspace: `{key}` 그대로 사용
- 사용자 정의 workspace: `{workspace_key_prefix}/{workspace_name}/{key}`
- `workspace_key_prefix` 기본값: `"env:"` (콜론 포함)

**`workspace_key_prefix` 커스텀:**

```hcl
terraform {
  backend "s3" {
    bucket               = "tf-core-tfstate"
    key                  = "gallery/terraform.tfstate"
    region               = "ap-northeast-2"
    workspace_key_prefix = "workspaces"   # 커스텀
  }
}
```

이 경우 `dev` workspace state 경로: `workspaces/dev/gallery/terraform.tfstate`

> `workspace_key_prefix`를 변경하면 기존 workspace state를 찾지 못한다. 변경 시 `terraform init -reconfigure` 필요.

### 4. Workspace 제한사항

#### ① 코드 공유로 인한 구조적 한계

모든 workspace가 **동일한 코드**를 공유한다. 환경별로 리소스 구성이 달라야 하는 경우(dev에는 없는 리소스가 prod에만 필요한 경우 등) workspace만으로는 깔끔하게 처리하기 어렵다.

- `count`나 `for_each`로 workspace 조건 분기를 넣으면 코드 복잡도가 급증한다
- 환경별 차이가 커질수록 `terraform.workspace == "prod"` 같은 조건이 곳곳에 생긴다
- 결과적으로 코드 가독성과 유지보수성이 저하된다

#### ② State 수준 잠금만 존재

Workspace 수준의 잠금(locking) 메커니즘은 없다. DynamoDB 등 backend의 state locking은 개별 state 파일 단위로 동작한다. 즉, `dev`와 `prod` workspace에서 동시에 `apply`를 실행하는 것은 가능하다 (각각 별도 state이므로). 하지만 같은 workspace에서 두 명이 동시에 `apply`하면 state lock이 충돌한다.

#### ③ `workspace delete`는 리소스를 destroy하지 않음

`terraform workspace delete <name>`는 **state 파일만 삭제**한다. 해당 workspace로 생성된 실제 AWS 리소스는 그대로 남는다.

올바른 순서:

```bash
# 1. 해당 workspace로 전환
$ terraform workspace select dev

# 2. 리소스 destroy
$ terraform destroy

# 3. default로 전환
$ terraform workspace select default

# 4. workspace 삭제
$ terraform workspace delete dev
```

리소스가 남아있는 workspace를 삭제하려면 `-force` 플래그가 필요하다.

```bash
$ terraform workspace delete -force dev
```

이 경우 리소스가 "dangling" 상태가 된다 — AWS에 존재하지만 Terraform이 더 이상 추적하지 못한다. 수동으로 AWS Console이나 CLI에서 삭제해야 한다.

#### ④ `default` workspace 삭제 불가

`default` workspace는 삭제할 수 없다. Terraform이 항상 하나 이상의 workspace를 유지해야 하기 때문이다.

#### ⑤ Backend 자격 증명 공유

동일 backend를 사용하므로 모든 workspace가 **같은 자격 증명**으로 접근한다. 환경별로 다른 AWS 계정이나 IAM 권한이 필요한 경우 workspace로는 분리할 수 없다.

#### ⑥ plan 파일 workspace 교차 사용 차단 (TF 1.14)

Terraform 1.14에서 추가된 변경사항: 한 workspace에서 생성한 plan 파일을 다른 workspace에서 `apply`하려 하면 **명시적 에러**가 발생한다. 이전 버전에서는 이 검증이 불충분했다.

### 5. Workspace vs 디렉토리 기반 환경 분리

#### 비교표

| 기준 | Workspace | 디렉토리 |
|------|-----------|---------|
| 코드 | 공유 (단일 복사본) | 분리 (환경별 복사본 또는 모듈 호출) |
| State | 분리 (workspace별) | 분리 (디렉토리별) |
| Backend 자격 증명 | 공유 | 환경별 분리 가능 |
| 환경별 리소스 차이 | 조건식으로 처리 (복잡도 증가) | 코드 레벨에서 자유롭게 차이 |
| 환경 전환 | `terraform workspace select` | `cd envs/dev && terraform init` |
| 실수 위험 | workspace 전환 깜빡하면 다른 환경에 apply 가능 | 디렉토리 자체가 다르므로 위험 낮음 |
| 코드 동기화 | 자동 (같은 코드) | 수동 관리 필요 (drift 가능) |
| CI/CD 통합 | workspace 이름 기반 분기 | 디렉토리 경로 기반 분기 |

#### Workspace가 적합한 경우

- 환경 간 인프라가 **거의 동일**하고 파라미터(instance_type, count 등)만 다른 경우
- 빠른 프로토타이핑, 개인 테스트 환경
- 동일 팀이 단일 자격 증명으로 관리하는 경우

#### 디렉토리가 적합한 경우

- 환경별로 인프라 구성이 **다를 수 있는** 프로덕션 운용
- 환경별 다른 AWS 계정/IAM 권한 사용
- 팀 간 접근 제어 분리 필요
- CI/CD 파이프라인에서 환경별 독립 배포

#### HashiCorp 공식 입장

HashiCorp 공식 문서에서 명시적으로 언급한 내용:

> "Workspaces are not appropriate for system decomposition or deployments requiring separate credentials and access controls."

> "To separate environments with potential configuration differences, use a directory structure. Use workspaces for environments that do not greatly deviate from one another."

Terraform Cloud/Enterprise에서는 "workspace"의 개념이 CLI workspace와 다르다. Terraform Cloud workspace는 디렉토리 분리에 가까운 개념으로, 각 workspace가 독립된 코드·변수·자격 증명을 가진다.

### 6. TF 1.14.x 버전 관련 변경사항

| 변경사항 | 버전 | 영향 |
|---------|------|------|
| plan 파일 workspace 교차 적용 시 명시적 에러 | 1.14.0 | workspace A에서 생성한 plan을 workspace B에 apply하면 에러. 이전 버전에서는 검증 불충분 |
| `terraform import`가 workspace 변수 전체 조회 | 1.14.0 | variable set에서 상속된 변수 포함. CLI workspace보다 TF Cloud 관련 |
| `terraform query` (list resources) | 1.14.0 | 신규 기능. workspace와 직접 관련 없으나 `*.tfquery.hcl` 파일 추가 |

workspace 기본 동작 자체에 대한 breaking change는 TF 1.14.x에서 없다. 핵심 개념과 사용법은 이전 버전과 동일하다.

## 버전 요구사항

| 항목 | 요구 버전 | 비고 |
|------|----------|------|
| Terraform CLI | >= 1.14.0 | workspace 기본 기능은 0.10+에서 존재. 1.14의 plan 교차 검증 개선 활용 |
| hashicorp/aws provider | ~> 6.0 | workspace 동작과 무관 (provider는 state 저장 방식을 모름) |
| S3 backend | Terraform 내장 | 별도 provider 불요 |

---

## 주의사항

- `terraform.workspace`를 backend 블록에서 사용하려는 시도가 빈번한 실수다. backend 블록은 정적이므로 변수·locals·표현식 일체 불가
- workspace 전환을 잊고 `apply`하는 실수 방지를 위해 prompt에 workspace 표시를 권장 (`terraform workspace show`)
- `workspace_key_prefix` 변경 시 기존 state 경로가 달라지므로 반드시 `terraform init -reconfigure` 실행
- workspace 삭제 전 반드시 `terraform destroy`로 리소스 정리. `-force` 삭제는 dangling 리소스 발생
