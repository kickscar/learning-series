# Research: 02.03 — resource 블록

> 조사일: 2026-04-15
> 기준: Terraform 1.14.x, hashicorp/aws provider >= 6.0

## 검증 결과

| 항목 | 상태 | 비고 |
|------|------|------|
| resource 블록 구조: 레이블 2개 (타입, 이름) | ✓ 정확 | 공식 문서 resource block reference에서 확인 |
| 리소스 주소: type.name 형식 | ✓ 정확 | plan, state, -target 등 모든 곳에서 사용 |
| 참조 표현식: type.name.attribute | ✓ 정확 | 변경 없음 |
| 암묵적 의존성 (참조 표현식 기반) | ✓ 정확 | 참조가 의존성 그래프를 자동 생성 |
| depends_on 문법: 리소스 참조 리스트 | ✓ 정확 | `depends_on = [resource_type.name]` 형식. 리스트이며 여러 리소스/모듈 참조 가능. 임의 표현식 불가 |
| depends_on 사용 가능 블록 | ⚠ 주의 | draft에서는 resource에서만 언급. 실제로는 resource, module, data, check, ephemeral, output, stack component 블록에서 모두 사용 가능. Ch02 수준에서는 resource만 다루면 충분 |
| meta-argument 목록: depends_on, count, for_each, lifecycle | ⚠ 주의 | `provider` meta-argument 누락. 공식 문서에 provider도 meta-argument로 명시 (alias된 provider 지정용). draft 표에 추가 권장 |
| lifecycle: create_before_destroy, ignore_changes, prevent_destroy | ✓ 정확 | 모두 유효. 추가로 `precondition`, `postcondition`, `replace_triggered_by`도 존재하나 Ch02 수준에서는 불필요 |
| aws_security_group inline ingress/egress 블록 | ⚠ 주의 | AWS Provider 6.x에서 여전히 **지원됨** (deprecated 아님). 다만 공식 문서에서 `aws_vpc_security_group_ingress_rule` / `aws_vpc_security_group_egress_rule` 별도 리소스 사용을 권장. inline 규칙과 별도 규칙 리소스를 혼용하면 충돌 발생. draft의 lab01/lab02에서 inline 사용은 Ch02 학습 목적상 허용 가능하나, 주의 문구 추가 권장 |
| egress protocol = -1 (숫자) vs "-1" (문자열) | ⚠ 주의 | draft lab01 main.tf에서 `protocol = -1` (숫자), lab02에서도 동일. 동작은 하지만 공식 문서 예시에서는 `"-1"` (문자열) 사용. 일관성을 위해 `"-1"` 권장 |
| aws_iam_role assume_role_policy + jsonencode() | ✓ 정확 | 여전히 유효하고 권장되는 패턴. `aws_iam_policy_document` data source도 대안이지만 jsonencode()는 간결함의 장점 |
| aws_iam_role_policy_attachment 사용 | ✓ 정확 | `aws_iam_policy_attachment`보다 권장됨. 후자는 exclusive attachment로 다른 attachment를 해제할 위험. `aws_iam_role_policy_attachment`는 non-exclusive |
| AmazonSSMManagedInstanceCore policy ARN | ✓ 정확 | `arn:aws:iam::aws:policy/AmazonSSMManagedInstanceCore` — 이중 콜론(::)은 AWS 관리형 정책의 정상 ARN 형식. account-id 필드가 비어있는 것이 맞음 |
| SSM Session Manager 접속 패턴 | ✓ 정확 | SSH 포트 없이 IAM Role + AmazonSSMManagedInstanceCore 정책으로 Session Manager 접속 가능. ingress 규칙 불필요 |
| lab01: plan → apply → destroy 사이클 | ✓ 정확 | terraform.tfstate 생성 확인, destroy 후 빈 상태 |
| lab02: SG → EC2 암묵적 의존성 | ✓ 정확 | vpc_security_group_ids 참조가 의존성 생성 |
| lab03: depends_on으로 policy attachment 대기 | ✓ 정확 | EC2가 instance profile은 참조하지만 policy attachment는 참조하지 않으므로 depends_on 필요 |
| lab02: associate_public_ip_address = true | ✓ 정확 | default VPC에서 퍼블릭 IP 할당에 유효 |

## 주의사항

- **meta-argument 표에 `provider` 추가 권장**: 공식 문서에서 resource 블록의 meta-argument로 `depends_on`, `count`, `for_each`, `lifecycle`, **`provider`** 5가지를 명시한다. draft의 meta-argument 개요 표에 `provider`가 빠져 있다. `provider = aws.us_east` 형식으로 alias된 provider를 지정하는 용도이며, 02.02에서 alias를 다루고 있으므로 교차 참조 가능하다.

- **inline ingress/egress 사용 관련 안내 추가 권장**: AWS Provider 공식 문서에서 inline 규칙 대신 `aws_vpc_security_group_ingress_rule` / `aws_vpc_security_group_egress_rule` 별도 리소스를 권장한다. Ch02에서는 학습 목적으로 inline을 사용하되, "실무에서는 별도 리소스를 사용한다" 정도의 한 줄 안내가 있으면 좋다. 별도 리소스로 전환하는 실습은 이후 챕터에서 다룰 수 있다.

- **protocol 값 타입 일관성**: lab01 main.tf에서 `protocol = -1` (숫자)을 사용한다. HCL이 자동 변환하므로 동작은 하지만, AWS Provider 문서 예시에서는 `"-1"` (문자열)을 사용한다. 코드 일관성을 위해 문자열로 통일하는 것이 좋다: `protocol = "-1"`.

- **lifecycle 옵션 추가 존재**: draft에서 `create_before_destroy`, `ignore_changes`, `prevent_destroy` 3가지를 소개한다. 실제로는 `precondition`, `postcondition`, `replace_triggered_by`도 존재한다. Ch02 수준에서 모두 다룰 필요는 없으나, "이 외에도 옵션이 있으며 Ch09에서 다룬다" 정도의 언급이 있으면 좋다 (draft에 이미 Ch09 참조가 있으므로 문제 없음).

- **lab02의 key_name = "key-tf-core"**: 이 키 페어가 독자 환경에 존재한다고 가정한다. 사전 준비 또는 주석에서 키 페어 생성을 안내하거나, 대안으로 키 페어 없이 SSM으로 접속하는 패턴(lab03)을 미리 언급하면 좋다.
