# State 변경 명령어

## 1. terraform state mv

State에서 리소스 주소를 변경한다. 코드에서 리소스 이름을 바꾸거나 모듈로 이동할 때 사용한다.

```bash
$ terraform state mv <SOURCE> <DESTINATION>
```

코드에서 `aws_security_group.web`을 `aws_security_group.instance_web`으로 이름을 변경했다고 하자. 코드만 바꾸면 Terraform은 `web`을 삭제하고 `instance_web`을 새로 만들려 한다. `state mv`로 State의 주소를 먼저 변경하면 같은 리소스로 인식한다.

```bash
# 사전 확인
$ terraform state mv -dry-run aws_security_group.web aws_security_group.instance_web

# 실행
$ terraform state mv aws_security_group.web aws_security_group.instance_web
```

```text
Move "aws_security_group.web" to "aws_security_group.instance_web"
Successfully moved 1 object(s).
```

`-dry-run`으로 먼저 확인하는 것을 권장한다. 모듈 간 이동도 가능하다.

```bash
# 루트 → 모듈
$ terraform state mv aws_instance.web module.ec2.aws_instance.web

# 모듈 → 루트
$ terraform state mv module.ec2.aws_instance.web aws_instance.web
```

### ① 선언형 대안: moved 블록

`state mv`는 CLI 명령어다. TF 1.1+에서는 코드에 `moved` 블록을 선언해 같은 효과를 얻을 수 있다.

```hcl
moved {
  from = aws_security_group.web
  to   = aws_security_group.instance_web
}
```

`moved` 블록은 코드에 남아있으므로 팀원이 `terraform apply`를 실행하면 자동으로 State가 갱신된다. CLI `state mv`는 실행한 사람의 State만 변경하므로 팀 환경에서는 `moved` 블록이 더 안전하다.

## 2. terraform state rm

State에서 리소스를 제거한다. **실제 인프라는 삭제하지 않는다** — Terraform의 추적 대상에서만 빠진다.

```bash
# 사전 확인
$ terraform state rm -dry-run aws_security_group.instance_web

# 실행
$ terraform state rm aws_security_group.instance_web
```

```text
Removed aws_security_group.instance_web
Successfully removed 1 resource instance(s).
```

확인 프롬프트 없이 즉시 실행된다. 반드시 `-dry-run`으로 먼저 확인한다.

제거 후 `terraform plan`을 실행하면 해당 리소스를 "새로 생성"하려 한다 — State에서 빠졌으므로 Terraform이 존재를 모르기 때문이다. 코드에서 해당 `resource` 블록도 함께 제거해야 plan이 깨끗해진다.

### ① 선언형 대안: removed 블록

TF 1.7+에서는 `removed` 블록으로 같은 효과를 얻을 수 있다.

```hcl
removed {
  from = aws_security_group.instance_web

  lifecycle {
    destroy = false
  }
}
```

`destroy = false`는 실제 인프라를 삭제하지 않고 State에서만 제거한다. `moved` 블록과 마찬가지로 코드에 선언되므로 팀 환경에서 안전하다.

# terraform import

## 1. CLI import

AWS 콘솔이나 다른 도구로 생성한 기존 리소스를 Terraform State로 가져온다.

```bash
$ terraform import <ADDRESS> <ID>
```

```bash
$ terraform import aws_security_group.instance_web sg-0abc1234567890def
```

```text
aws_security_group.instance_web: Importing from ID "sg-0abc1234567890def"...
aws_security_group.instance_web: Import prepared!
aws_security_group.instance_web: Refreshing state...

Import successful!
```

import는 **State에만 기록**한다. 코드를 자동 생성하지 않는다. import 후 해당 리소스의 `resource` 블록을 직접 작성해야 한다. 코드가 없으면 다음 plan에서 삭제 대상이 된다.

import 후 작업 흐름:

1. `terraform import`로 State에 등록
2. `terraform state show`로 속성 확인
3. 확인한 속성을 기반으로 `resource` 블록 작성
4. `terraform plan`으로 "No changes" 확인

## 2. import 블록 (선언형)

TF 1.5+에서 `import` 블록을 코드에 선언할 수 있다.

```hcl
import {
  to = aws_security_group.instance_web
  id = "sg-0abc1234567890def"
}
```

`terraform plan`을 실행하면 import 계획이 포함된다. `-generate-config-out` 플래그로 코드를 자동 생성할 수도 있다.

```bash
$ terraform plan -generate-config-out="generated.tf"
```

생성된 코드는 검토 후 정리가 필요하지만, 속성을 하나씩 `state show`로 확인해 작성하는 것보다 효율적이다. `for_each`와 함께 사용하면 여러 리소스를 한 번에 import할 수 있다.

---

# 명령어 정리

| 명령어 | 역할 | 실제 인프라 | 선언형 대안 |
|--------|------|-----------|-----------|
| `state list` | 관리 리소스 목록 | 변경 없음 | — |
| `state show` | 리소스 상세 조회 | 변경 없음 | — |
| `state mv` | 리소스 주소 변경 | 변경 없음 | `moved` 블록 (TF 1.1+) |
| `state rm` | State에서 제거 | **유지** (삭제 안 함) | `removed` 블록 (TF 1.7+) |
| `import` (CLI) | 기존 리소스 → State | 변경 없음 | `import` 블록 (TF 1.5+) |

`state mv`, `state rm`, `import`에는 선언형 대안이 있다. CLI 명령어는 실행한 사람의 State만 변경하지만, 선언형 블록은 코드에 남아 팀원에게도 적용된다.

---

# 핵심 정리

- `terraform state list`로 관리 중인 리소스를 조회하고, `state show`로 상세 속성을 확인한다.
- `terraform state mv`로 리소스 이름 변경이나 모듈 이동 시 불필요한 삭제-재생성을 방지한다. `-dry-run`으로 먼저 확인한다.
- `terraform state rm`은 State에서만 제거하고 실제 인프라는 유지한다. 확인 프롬프트 없이 즉시 실행되므로 주의한다.
- `terraform import`로 기존 인프라를 State에 등록한다. State만 기록하므로 코드를 별도로 작성해야 한다.
- `moved`, `removed`, `import` 블록은 각각 `state mv`, `state rm`, CLI `import`의 선언형 대안이다. 팀 환경에서는 선언형이 더 안전하다.

# [실습] lab02: Gallery 인스턴스 이름 변경 → state rm → 문제 발생

Gallery의 `aws_instance.web`을 다른 이름으로 변경하고, State에서 제거해 문제 상황을 만든다.

### 실습 목표

- `state mv`로 리소스 이름 변경 후 "No changes" 확인
- `state rm`으로 State에서 제거 — 서비스는 계속 동작
- `terraform plan`에서 "to be created" 확인 — 이미 존재하는 인스턴스를 또 만들려는 문제 상황
- lab03에서 해결할 문제를 남겨둔다

---

# 1. state mv — 이름 변경

`aws_instance.web`을 `aws_instance.spring`으로 변경한다. 먼저 State를 변경한다:

```bash
# 사전 확인
$ terraform state mv -dry-run aws_instance.web aws_instance.spring

# 실행
$ terraform state mv aws_instance.web aws_instance.spring
```

```text
Move "aws_instance.web" to "aws_instance.spring"
Successfully moved 1 object(s).
```

코드의 `main.tf`에서도 리소스 이름을 변경한다:

```hcl
# 변경 전
resource "aws_instance" "web" { ... }

# 변경 후
resource "aws_instance" "spring" { ... }
```

`outputs.tf`에서 `aws_instance.web` 참조도 `aws_instance.spring`으로 변경한다.

```bash
$ terraform plan
```

```text
No changes. Your infrastructure matches the configuration.
```

State와 코드가 일치한다. 서비스도 계속 동작 중이다. 이름 변경 자체는 성공이다.

---

# 2. state rm — State에서 제거

```bash
# 사전 확인
$ terraform state rm -dry-run aws_instance.spring

# 실행
$ terraform state rm aws_instance.spring
```

```text
Removed aws_instance.spring
Successfully removed 1 resource instance(s).
```

확인 프롬프트 없이 즉시 실행된다. AWS에서 EC2 인스턴스는 계속 동작 중이다 — State에서만 빠졌다.

---

# 3. terraform plan — 문제 확인

```bash
$ terraform plan
```

```text
  # aws_instance.spring will be created
  + resource "aws_instance" "spring" {
      + ami                         = "ami-024ea438ab0376a47"
      + instance_type               = "t3.small"
      ...
    }

Plan: 1 to add, 0 to change, 0 to destroy.
```

이미 동작 중인 인스턴스가 있는데 Terraform은 새로 만들려 한다. State에서 빠졌으므로 Terraform이 존재를 모르기 때문이다. 이 상태에서 apply하면 **동일한 구성의 EC2가 중복 생성**된다.

**apply하지 않는다.** 이 문제를 lab03에서 `import`로 해결한다.

# 5. 정리

Gallery 인프라는 Ch06에서 계속 사용하므로 destroy하지 않는다. lab02~03을 통해 다음을 확인했다:

- `state rm`은 State에서만 제거하고 실제 인프라는 유지한다
- State에서 빠진 리소스는 plan에서 "to be created"로 표시된다 — 중복 생성 위험
- `import`로 실행 중인 인프라를 State에 다시 등록할 수 있다
- 이름 변경이 필요하면 `state mv` 또는 `moved` 블록을 사용하고, `state rm` + `import`는 복구 수단이다
