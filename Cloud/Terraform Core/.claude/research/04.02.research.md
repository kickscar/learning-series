## 질문별 상세 답변

### Q1. `version: 4`가 TF 1.14.x에서도 현행인가? version 5 계획은?

**현행 확인**: State format version 4는 TF 1.14.x에서도 유지된다. Go 소스의 `stateV4` struct가 현재 사용 중이며, 1.14.0 릴리스 노트에 state format 변경 언급이 없다.

**version 5 계획**: 2026년 4월 기준 공식적으로 확인된 version 5 계획은 없다. 다만 공식 문서에서 "the state format is subject to change"라고 명시하고 있으므로, 향후 변경 가능성은 열려 있다. version 4 내에서 새 필드가 추가되는 방식(check_results, identity 등)으로 확장이 이루어지고 있다.

04.01 research에서도 동일 결론을 확인했다.

### Q2. `serial` 충돌 시 동작

**serial의 역할**: 매 state 변경마다 1씩 증가하는 uint64 카운터다. 동시성 제어의 보조 수단으로 사용된다.

**충돌 시나리오 (local state, 두 사람 동시 apply)**:
- Local state에서는 OS-level file locking이 동작하므로 **같은 머신**에서는 동시 apply가 차단된다.
- **다른 머신**에서 같은 local state 파일을 공유하는 경우(NFS 등), lock이 작동하지 않아 후자의 apply가 전자의 변경을 덮어쓸 수 있다. 이것이 local state의 근본적 한계다.

**`terraform state push` 시 serial 보호**:
- 공식 문서: "If the 'serial' value in the destination state is higher than the state being pushed, Terraform will prevent the push."
- 즉, 원격에 더 높은 serial이 있으면 push가 거부된다. `-force` 플래그로 무시할 수 있으나 권장하지 않는다.

**Remote backend에서의 보호**:
- Remote backend(S3 등)는 locking 메커니즘으로 동시 apply를 차단한다.
- Lock 획득 실패 시 apply가 시작되지 않으므로 serial 충돌 자체가 발생하지 않는다.
- `terraform force-unlock`으로 lock을 해제할 수 있으나, lock ID가 필요하며 다른 사용자의 active lock 해제는 위험하다.

### Q3. Backend 마이그레이션 시 `lineage` 변경 여부

**결론: lineage는 변경되지 않는다.**

`terraform init -migrate-state`로 backend를 변경(예: local → S3)하면 state 파일의 내용이 그대로 새 backend로 복사된다. lineage는 state의 이력 체인을 식별하는 UUID이므로, 같은 state를 이동하는 것일 뿐 새로 생성하는 것이 아니다.

lineage가 달라지는 경우:
- `terraform init`으로 완전히 새로운 state를 생성할 때
- `terraform state push`로 다른 lineage의 state를 강제 push할 때 (`-force`)

공식 문서: "If a lineage is different, then it means the states were created at different times and its very likely you're modifying a different state. Terraform will not allow this." — state push 시 lineage 검증으로 실수를 방지한다.

### Q4. `sensitive_attributes` 사용 여부 및 `sensitive = true`와의 상호작용

**사용 중 확인**: Go 소스에서 `AttributeSensitivePaths json.RawMessage \`json:"sensitive_attributes,omitempty"\``로 정의되어 있다. TF 1.14.x에서도 현행이다.

**동작 방식**:
- `sensitive = true`로 표시된 variable/output을 참조하는 리소스 속성은 `sensitive_attributes` 필드에 해당 속성의 JSON path가 기록된다.
- 이 필드는 CLI 출력(plan, apply, state show)에서 값을 `(sensitive value)`로 마스킹하는 데 사용된다.
- **중요**: sensitive 마킹은 **출력 마스킹(redaction)만** 수행한다. 실제 값은 `attributes` 필드에 plaintext로 저장된다.
- 공식 문서: "Terraform stores values with the sensitive argument in both state and plan files, and anyone who can access those files can access your sensitive values."

**`sensitive_attributes` 형식 예시**:
```json
"sensitive_attributes": [
  [{"type": "get_attr", "value": "password"}]
]
```
JSON path 배열로, 어떤 속성이 sensitive인지 기록한다.

### Q5. `check_results` 필드

**도입 시점**: TF 1.5.0 (check 블록과 함께 도입)

**Go struct 정의**:
```go
type checkResultsV4 struct {
    ObjectKind string                   `json:"object_kind"`
    ConfigAddr string                   `json:"config_addr"`
    Status     string                   `json:"status"`
    Objects    []checkResultsObjectV4   `json:"objects"`
}

type checkResultsObjectV4 struct {
    ObjectAddr      string   `json:"object_addr"`
    Status          string   `json:"status"`
    FailureMessages []string `json:"failure_messages,omitempty"`
}
```

**역할**: 이전 run의 check 블록 실행 결과를 state에 저장한다. check 블록이 없는 구성에서는 이 필드가 비어 있거나 null이다. TF 1.5 이전 버전으로 생성된 state에는 이 필드 자체가 없을 수 있다(Go 소스 주석: "We differentiate absence from an empty array here so that we can recognize if the previous run was with a version of Terraform that didn't support checks yet").

**draft 작성 시 권장**: top-level 필드 소개 시 간략히 언급하되, check 블록 자체는 Ch09에서 다루므로 깊이 있는 설명은 불필요하다.

### Q6. `terraform.tfstate.backup` 생성 시점

**생성 시점**: state를 변경하는 모든 작업에서 생성된다.

- `terraform apply`: state 변경이 발생하므로 **생성된다**
- `terraform plan`: state를 변경하지 않으므로 **생성되지 않는다** (단, `-refresh=true` 기본값이지만 refresh 결과는 plan 중에는 state에 기록되지 않는다)
- `terraform state mv/rm`: state 수정 명령이므로 **생성된다**. 공식 문서: "all state-modifying commands automatically write backup files that can not be disabled"
- `terraform import`: state에 리소스를 추가하므로 **생성된다**

**동작 방식**: 현재 `terraform.tfstate`를 `terraform.tfstate.backup`으로 복사한 후, 새로운 state를 `terraform.tfstate`에 기록한다. 즉, backup은 **직전** state의 스냅샷이다.

**비활성화**: `-backup=-` 플래그로 backup 생성을 비활성화할 수 있다. 단, state 수정 명령(mv, rm 등)에서는 비활성화할 수 없다.

**참고**: `-backup` 플래그는 legacy feature로 분류되어 있다. Remote backend 사용 시에는 backend 자체의 versioning(S3 versioning 등)이 backup 역할을 대신한다.

### Q7. `terraform state pull` / `terraform state push` 현행 여부

**현행 확인**: 두 명령 모두 사용 가능하며 deprecation 예정 없다.

**`terraform state pull`**:
- 현재 backend에서 state를 다운로드하여 stdout에 JSON으로 출력한다.
- 자동으로 현재 TF 버전에 호환되는 state format으로 변환한다.
- 주 용도: state 내용 확인, `jq` 등과 조합한 분석, disaster recovery
- 예: `terraform state pull | jq '.resources'`

**`terraform state push`**:
- 로컬 state 파일을 현재 backend에 업로드한다.
- **안전 장치 2가지**:
  1. **lineage 검증**: 다른 lineage이면 push 거부
  2. **serial 검증**: 원격 serial이 더 높으면 push 거부
- `-force` 플래그로 두 검증을 모두 무시할 수 있으나 비권장
- stdin에서 읽기: `terraform state push -` (파이프 입력)

**분류**: 공식 문서에서 "Disaster Recovery" 섹션에 배치. 일상적 사용보다는 긴급 상황용이다.

## 추가 발견: `instances` 필드 전체 구조 (version4.go 기준)

Go 소스에서 확인한 `instanceObjectStateV4`의 전체 필드:

| JSON 필드 | Go 타입 | 설명 | omitempty |
|-----------|---------|------|-----------|
| `index_key` | interface{} | count/for_each 사용 시 인스턴스 키 | ✓ |
| `status` | string | 인스턴스 상태 (tainted 등) | ✓ |
| `deposed` | string | deposed 인스턴스 키 | ✓ |
| `schema_version` | uint64 | provider의 리소스 schema 버전 | |
| `attributes` | json.RawMessage | 리소스의 모든 속성 값 (JSON) | ✓ |
| `attributes_flat` | map[string]string | legacy 평면화 속성 (거의 사용 안 함) | ✓ |
| `sensitive_attributes` | json.RawMessage | sensitive로 표시된 속성의 JSON path 목록 | ✓ |
| `identity_schema_version` | uint64 | identity schema 버전 (TF 1.12+) | |
| `identity` | json.RawMessage | 리소스 identity 데이터 (TF 1.12+) | ✓ |
| `private` | []byte | provider 내부 사용 데이터 (base64 인코딩) | ✓ |

### `identity` / `identity_schema_version` (TF 1.12+)

TF 1.12에서 import 블록에 `identity` 속성이 도입되었고, state에도 `identity` 필드가 추가되었다. TF 1.13에서 "Fixes resource identity being dropped from state in certain cases" 버그 수정이 있었다. 이 필드는 리소스의 고유 식별 정보를 `id`와 별도로 저장하는 메커니즘이다.

**draft 작성 시 권장**: instances 구조 설명에서 핵심 필드(schema_version, attributes, sensitive_attributes, dependencies)를 중심으로 설명하고, identity는 "TF 1.12+에서 추가된 필드"로 간략히 언급하는 수준이 적절하다.

## 주의사항

### state 파일 직접 수정의 위험성 — draft 서술 포인트

공식 문서에서 직접 수정을 경고하는 이유:
1. **포맷 변경 가능성**: "the state format is subject to change" — 새 버전에서 구조가 바뀔 수 있다
2. **내부 일관성 파괴**: serial, lineage, dependencies 등 내부 참조가 깨질 수 있다
3. **JSON 포맷 오류**: 수동 편집 시 JSON 문법 오류로 state 자체가 읽히지 않을 수 있다
4. **안전한 대안 존재**: `terraform state` 명령이 내부 일관성을 유지하면서 수정을 수행한다

### `terraform plan` 시 backup이 생성되지 않는 이유

`terraform plan`은 state를 변경하지 않는 읽기 전용 작업이다. `-refresh=true`(기본값)로 실제 인프라를 조회하지만, refresh 결과를 state 파일에 기록하지 않는다. 따라서 backup이 불필요하다.

단, `terraform apply -refresh-only`는 refresh 결과를 state에 기록하므로 backup이 생성된다.

### Ephemeral resources와 write-only attributes — State 비저장

TF 1.10+(ephemeral resources)과 TF 1.11+(write-only attributes)는 State에 저장되지 않는다.
- 공식 문서: "Terraform does not store information about ephemeral resources in state or plan files."
- 04.01에서도 확인한 사항. 04.02 draft에서 "State에 저장되는 것과 저장되지 않는 것"을 구분하는 맥락에서 간략히 언급할 수 있다.
