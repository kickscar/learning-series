07.02~03에서 Workspace와 디렉토리 기반 환경 분리를 실습했다. 두 방식 모두 환경별로 **다른 변수 값**을 주입하는 것이 핵심이었다. 이번 섹션에서는 Terraform의 변수 입력 메커니즘 전체를 정리한다: 우선순위 체계, 환경별 분리 전략, CI/CD 파이프라인에서의 활용, sensitive 변수 처리.

# 변수 입력 전략

## 1. 우선순위 체계

Terraform은 같은 변수에 여러 값이 주어지면 **우선순위**로 결정한다. 높은 쪽이 이긴다.

```text
1. -var / -var-file          ← 명령��� (가장 높음)
2. *.auto.tfvars(.json)      ← lexical 순서 자동 로드
3. terraform.tfvars.json     ← 자동 로드
4. terraform.tfvars          ← 자동 로드
5. TF_VAR_<name>             ← 환경 변수
6. variable { default }      ← 코드 기본값 (가장 낮음)
```

### ① -var과 -var-file

명령줄에서 나열 순서대로 처리된다. 같은 변수를 여러 번 지정하면 **마지막 값**이 적용된다.

```bash
# -var-file 먼저, -var 나중 → -var 승
$ terraform apply -var-file="base.tfvars" -var="instance_type=t3.large"

# -var 먼저, -var-file 나중 → -var-file 승
$ terraform apply -var="instance_type=t3.large" -var-file="override.tfvars"
```

### ② 자동 로드 파일

| 파일 | 로드 조건 |
|------|----------|
| `terraform.tfvars` | 존재하면 자동 |
| `terraform.tfvars.json` | 존재하면 자동 (`.tfvars`보다 우선) |
| `*.auto.tfvars` | 존재하면 lexical 순서로 자동 |
| `*.auto.tfvars.json` | 존재하면 lexical 순서로 자동 |

`terraform.tfvars`와 `terraform.tfvars.json`이 모두 존재하면 **양쪽 모두 로드**된다. 같은 변수가 충돌하면 `.json`이 이긴다.

`*.auto.tfvars`는 사전순(lexical order)으로 로드된다. `00-base.auto.tfvars` → `99-override.auto.tfvars` 순서다. 뒤에 로드된 파일이 앞 파일의 같은 변수를 덮어쓴다.

## 2. terraform.tfvars — 공통 설정

프로젝트의 **기본 설정값**을 담는다. `-var-file` 플래그 없이 자동 로드된다.

```hcl
# terraform.tfvars
aws_region    = "ap-northeast-2"
instance_type = "t3.micro"
env           = "dev"
```

디렉토리 기반 환경 분리(07.03)에서는 각 환경 디렉토리에 `terraform.tfvars`를 배치했다. 이때 `terraform.tfvars`는 "공통 설정"이 아니라 "해당 환경의 기본 설정"이 된다.

## 3. -var-file — 환경별 분리

단일 디렉토리에서 환경별 설정을 분리할 때 사용한다.

```text
project/
├── main.tf
├── variables.tf
├── terraform.tfvars       ← 공통 설정 (자동 로드)
├── dev.tfvars             ← dev 전용 override
└── prod.tfvars            ← prod 전용 override
```

```bash
$ terraform apply -var-file="dev.tfvars"     # dev 배포
$ terraform apply -var-file="prod.tfvars"    # prod 배포
```

`terraform.tfvars`의 공통 값을 기반으로, `-var-file`이 환경별 차이를 덮어쓴다. 우선순위에서 `-var-file`이 `terraform.tfvars`보다 높기 때문이다.

> 디렉토리 기반 분리(07.03)와 비교: 디렉토리 방식은 각 환경이 자체 `terraform.tfvars`를 가지므로 `-var-file` 플래그가 불필요하다. `-var-file` 방식은 단일 디렉토리에서 환경을 분리할 때 사용하며, backend까지 분리하기는 어렵다.

## 4. TF_VAR_* — CI/CD 파이프라인

환경 변수 `TF_VAR_<name>`으로 값을 주입한다. CI/CD 파이프라인에서 보안 민감 값을 전달할 때 주로 사용한다.

```bash
$ export TF_VAR_db_password="s3cr3t"
$ export TF_VAR_instance_type="t3.small"
$ terraform apply
```

변수 이름에서 `TF_VAR_` 접두사를 제거한 나머지가 Terraform 변수명과 매칭된다. 대소문자 구분한다.

**CI/CD 활용 예** (GitHub Actions):

```yaml
env:
  TF_VAR_db_password: ${{ secrets.DB_PASSWORD }}
  TF_VAR_env: "prod"

steps:
  - run: terraform apply -auto-approve
```

Secrets에 저장된 값을 `TF_VAR_*`로 주입하면 코드나 tfvars 파일에 민감 값을 남기지 않는다.

> 우선순위 주의: `TF_VAR_*`는 `terraform.tfvars`보다 **낮다**. tfvars 파일에 같은 변수가 있으면 tfvars가 이긴다. CI/CD에서 확실히 override하려면 `-var` 플래그를 사용한다.

## 5. default 전략

`variable` 블록의 `default` 값을 설정하는 두 가지 전략이 있다.

### ① Sane Defaults

대부분의 경우 적절한 기본값을 제공한다. 명시하지 않으면 기본값이 적용된다.

```hcl
variable "instance_type" {
  type    = string
  default = "t3.micro"    # 개발 환경에 적합한 기본값
}
```

개발자가 변수를 하나하나 지정하지 않아도 `apply`가 동작한다. 프로덕션에서만 `-var-file`로 override한다.

### ② No Default (필수 입력)

보안 민감 값이나 환경에 따라 반드시 달라야 하는 값은 default를 설정하지 않는다.

```hcl
variable "db_password" {
  type      = string
  sensitive = true
  # default 없음 → apply 시 반드시 값 필요
}
```

default가 없으면 Terraform이 값을 요구한다. 이것이 "실수로 기본값이 적용되는 것"을 방지하는 안전장치다.

**조합 원칙:**

| 변수 성격 | default 전략 | 입력 방법 |
|---------|-------------|----------|
| 환경 공통 (region, project) | sane default | terraform.tfvars |
| 환경별 차이 (instance_type) | dev용 default | -var-file 또는 tfvars |
| 보안 민감 (password, key) | no default | TF_VAR_* 또는 -var |
| 인프라 식별 (env) | no default 또는 "dev" | tfvars 또는 workspace |

## 6. sensitive 변수

`sensitive = true`로 선언하면 `plan`/`apply` CLI 출력���서 값이 마스킹된다.

```hcl
variable "db_password" {
  type      = string
  sensitive = true
}
```

```text
# terraform plan 출력
  + resource "aws_db_instance" "main" {
      + password = (sensitive value)
    }
```

### ① 마스킹 범위

| 단계 | 동작 |
|------|------|
| `terraform plan` | `(sensitive value)` |
| `terraform apply` | `(sensitive value)` |
| `terraform output` | `<sensitive>` |
| `terraform output -json` | **평문 노출** |
| State 파일 | **평문 저장** |

> **State에는 평문으로 저장된다.** `sensitive = true`는 CLI 출력 마스킹일 뿐, State 파일의 보안을 보장하지 않는다. State를 S3 + 암호화(`encrypt = true`)로 관리하고, 접근을 IAM으로 제한해야 한다.

### ② ephemeral 변수 (TF 1.10+)

Terraform 1.10부터 `ephemeral = true` 옵션이 추가되었다. sensitive와 달리 State와 plan 파일에 **아예 저장하지 않는다**. 민감 값이 어디에도 영속되지 않아야 하는 경우에 사용한다. 이 시리즈에서는 상세히 다루지 않는다.

## 7. 환경 분리와의 조합

Sec01~03에서 다룬 환경 분리 방식과 변수 입력 전략을 조합한다.

| 환경 분리 방식 | 변수 입력 패턴 |
|-------------|-------------|
| **Workspace** | `terraform.workspace`로 locals 분기. tfvars 불필요 |
| **디렉토리** | 환경별 `terraform.tfvars` 자동 로드. `-var-file` 불필요 |
| **단일 디렉토리 + -var-file** | `terraform.tfvars`(공통) + `prod.tfvars`(override) |
| **CI/CD** | `TF_VAR_*`(secrets) + `-var-file`(환경 설정) 조합 |

CI/CD 파이프라인에서 가장 일반적인 조합:

```bash
# 디렉토리 기반 + TF_VAR_* (secrets)
$ cd envs/prod
$ export TF_VAR_db_password="${DB_PASSWORD_FROM_VAULT}"
$ terraform apply -auto-approve
```

`terraform.tfvars`가 환경 기본값을, `TF_VAR_*`가 보안 민감 값을 각각 담당한다.

---

# 핵심 정리

- 변수 우선순위: `-var`/`-var-file` > `*.auto.tfvars` > `terraform.tfvars` > `TF_VAR_*` > default
- `-var`과 `-var-file`은 명령줄 나열 순서대로 처리 — 마지막 값이 적용
- 디렉토리 분리에서는 각 환경의 `terraform.tfvars`가 자동 로드 — `-var-file` 불필요
- CI/CD에서 민감 값은 `TF_VAR_*`로, 환경 설정은 `terraform.tfvars` 또는 `-var-file`로
- `sensitive = true`는 CLI 마스킹일 뿐 — State에는 평문 저장된다
- default 전략: 안전한 기본값(sane default) vs 필수 입력(no default)

# 4. 우선순위 실험

## 실험 1: default만 (tfvars 없이)

```bash
$ terraform plan -var-file=/dev/null
```

> `/dev/null`을 지���하면 `terraform.tfvars` 자동 로드를 무시할 수 없다. 대신 변수별 기본값 확인은 variable 블록에서 직접 읽는다.

실제로는 `terraform.tfvars`가 항상 자동 로드되므로, default만으로 동작하는 케이스는 `terraform.tfvars`가 없을 때다.

## 실험 2: terraform.tfvars 자동 로드

```bash
$ terraform plan
```

```text
  + resource "aws_vpc" "this" {
      + tags = {
          + "Environment"  = "dev"
          + "InstanceType" = "t3.micro"
          + "Name"         = "tf-core-lab01-dev-vpc-main"
        }
    }
```

`terraform.tfvars`의 `instance_type = "t3.micro"`, `env = "dev"`가 적용되었다. `project`는 default(`"lab01"`)가 적용된다.

## 실험 3: -var-file override

```bash
$ terraform plan -var-file="prod.tfvars"
```

```text
  + resource "aws_vpc" "this" {
      + tags = {
          + "Environment"  = "prod"
          + "InstanceType" = "t3.small"
          + "Name"         = "tf-core-lab01-prod-vpc-main"
        }
    }
```

`prod.tfvars`가 `terraform.tfvars`를 override했다. `-var-file`은 우선순위 1위다.

## 실험 4: -var 최종 override

```bash
$ terraform plan -var-file="prod.tfvars" -var="instance_type=t3.large"
```

```text
  + resource "aws_vpc" "this" {
      + tags = {
          + "Environment"  = "prod"
          + "InstanceType" = "t3.large"
          + "Name"         = "tf-core-lab01-prod-vpc-main"
        }
    }
```

`-var`이 `-var-file`까지 override했다. `env`는 `prod.tfvars`에서, `instance_type`은 `-var`에서 온다. **마지막에 지정된 값이 이긴다.**

## 실험 5: 환경 변수 (TF_VAR_*)

```bash
$ export TF_VAR_instance_type="t3.medium"
$ terraform plan
```

```text
  + resource "aws_vpc" "this" {
      + tags = {
          + "InstanceType" = "t3.micro"
        }
    }
```

`terraform.tfvars`(`t3.micro`)가 `TF_VAR_*`(`t3.medium`)를 이겼다. **환경 변수는 tfvars보다 우선순위가 낮다.**

```bash
$ unset TF_VAR_instance_type
```

```text
"MyS3cr3tP@ss"
```

> **보안 주의**: `sensitive = true`는 실수로 터미널에 값이 노출되는 것을 방지할 뿐이다. State 파일 보안은 S3 암호화 + IAM 접근 제어로 별도 확보해야 한다.

---

# 6. -var로 TF_VAR_* override

```bash
$ terraform plan -var="instance_type=t3.large"
```

```text
  ~ resource "aws_vpc" "this" {
      ~ tags = {
          ~ "InstanceType" = "t3.micro" -> "t3.large"
        }
    }
```

`-var`(`t3.large`)가 `TF_VAR_*`(`t3.micro`)를 override했다. 우선순위: `-var` > `TF_VAR_*`.

---

# 7. 정리

```bash
$ terraform destroy -auto-approve
$ unset TF_VAR_instance_type
$ unset TF_VAR_db_password
```

# 핵심 정리

- `TF_VAR_*`는 코드·파일에 값을 남기지 않는 주입 방식 — CI/CD secrets 전달에 적합
- `TF_VAR_*`는 `terraform.tfvars`보다 **낮은** 우선순위 — 확실한 override는 `-var` 사용
- `sensitive = true`는 CLI 출력 마스킹 — State에는 평문 저장된다
- `terraform output -raw`로 sensitive 값 확인 가능 — 진정한 보안은 State 암호화 + 접근 제어
- default 없는 변수는 Terraform이 값을 요구한다 — 실수 방지 안전장치
