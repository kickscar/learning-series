# Research: 04.01 — State란 무엇인가

> 조사일: 2026-04-15
> 기준: Terraform 1.14.x, hashicorp/aws provider ~> 6.0

## 주의사항

### `terraform refresh` deprecated 상태 — 정확한 표현

공식 문서 원문: "This command is deprecated. Instead, add the `-refresh-only` flag to `terraform apply` and `terraform plan` commands."

deprecated이지만 1.14.x에서 **제거되지 않았다**. 여전히 실행 가능하다. draft에서는 다음과 같이 서술하는 것이 정확하다:

- `terraform refresh`는 deprecated 상태다
- 기능적으로 `terraform apply -refresh-only -auto-approve`와 동일하다
- 권장 대체: `terraform apply -refresh-only` (승인 프롬프트 포함)
- deprecated 이유: auto-approve가 위험하기 때문. credential 설정 오류 시 모든 리소스가 삭제된 것으로 오인하여 State에서 전부 제거할 수 있다

### Local backend locking — "Locking 부재"는 부정확

draft에서 Local State의 한계로 "Locking 부재"를 언급할 경우 주의가 필요하다.

**사실**: local backend는 시스템 API(OS-level file locking)를 사용하여 locking을 지원한다. 같은 머신에서 동시에 두 terraform 프로세스를 실행하면 lock 충돌이 발생한다.

**진짜 한계**: local backend의 locking은 **같은 머신** 내에서만 유효하다. 팀원 각자의 머신에 State가 있으면 서로의 lock을 인식할 수 없다. 따라서 "locking 부재"가 아니라 **"팀 단위 locking 불가"** 또는 **"분산 환경에서의 locking 불가"**로 표현해야 정확하다.

### State 파일 포맷 — version 4 안정성

State 파일 format version 4는 장기간 안정적으로 유지되고 있다. 2026년 4월 기준 version 5 계획은 확인되지 않는다. State 파일은 항상 JSON 형식이며, 바이너리 포맷 옵션은 존재하지 않는다.

참고: plan 파일(`terraform plan -out=tfplan`)은 바이너리 형식이다. State 파일과 혼동하지 않도록 주의.

### `terraform plan` 기본 refresh 동작

`terraform plan`은 기본적으로 `-refresh=true`로 동작한다. 즉, plan 실행 시 자동으로 실제 인프라 상태를 읽어 State를 갱신한 후 diff를 계산한다.

- `-refresh=false`: refresh를 건너뛴다. 대규모 인프라에서 속도 향상 목적. 외부 변경 감지 불가.
- `-refresh-only`: refresh만 수행하고 인프라 변경 계획은 생성하지 않는다. drift 확인 전용.

### Ephemeral resources & write-only attributes — State와의 관계

TF 1.10+(ephemeral resources)과 TF 1.11+(write-only attributes)에서 도입된 기능으로, **State에 저장되지 않는 값**을 다룰 수 있다. 04.01 이론 섹션에서 State의 역할을 설명할 때 "모든 것이 State에 저장되는 것은 아니다"는 맥락에서 간략히 언급할 수 있다. 단, 상세 설명은 이 시리즈 범위를 벗어날 수 있으므로 깊이 조절 필요.

### State의 민감 데이터 문제

State 파일은 plaintext JSON이며, 리소스의 모든 속성 값을 포함한다. 데이터베이스 비밀번호, API 토큰 등 민감한 값이 그대로 저장된다. 공식 문서 권장사항:

- State 파일을 Git에 커밋하지 않는다
- Remote backend 사용 시 암호화 옵션 활용 (S3: `encrypt`, GCS: Cloud KMS, HCP Terraform: 자동 암호화)
- `.gitignore`에 `*.tfstate`, `*.tfstate.*` 추가

### drift 감지 및 해소 방법

**감지**: `terraform plan` 실행 시 refresh 단계에서 자동 감지. 실제 인프라 상태와 State의 차이를 보여준다.

**해소 방법**:
1. `terraform apply`: 코드 기준으로 인프라를 되돌린다 (코드가 진실)
2. 코드를 수정하여 현재 인프라 상태에 맞춘다 → `terraform apply` (인프라가 진실)
3. `terraform apply -refresh-only`: State만 갱신하고 인프라는 변경하지 않는다 (외부 변경 수용)

### S3 backend native locking (TF 1.10+)

S3 backend에서 DynamoDB 없이 S3 native locking을 사용할 수 있는 기능이 TF 1.10에서 도입되고 TF 1.11에서 GA가 되었다 (`use_lockfile` 인수). Ch04 Sec03(Remote Backend)에서 다룰 때 참고할 사항이다. 04.01 이론 섹션에서는 직접 다루지 않지만 알아두면 좋다.
