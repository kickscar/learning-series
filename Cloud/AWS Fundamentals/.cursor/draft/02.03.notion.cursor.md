# 1. Policy가 의미하는 것

## 1. Policy는 Authorization 규칙이다

IAM Policy는 "권한"을 정의하는 JSON 문서다. User/Group/Role 자체가 권한을 갖는 것이 아니라, **Policy가 무엇을 할 수 있는지(또는 못하게 할지)** 를 정의하고 그것이 Identity에 연결되면서 최종 권한이 결정된다.

인증(Authentication)이 "누구인가"를 확인하는 과정이라면, 인가(Authorization)는 "무엇을 할 수 있는가"를 결정하는 과정이다. Policy는 Authorization을 위한 규칙 집합이다.

### ① Policy는 API 호출 단위로 권한을 정의한다

Console에서 하는 클릭도 내부적으로는 AWS API 호출로 환산된다. Policy의 `Action`은 "어떤 API를 허용/거부하는가"를 의미한다.

예: `s3:GetObject`, `ec2:DescribeInstances`

### ② Policy는 아키텍처의 보안 경계다

Policy는 편의 기능이 아니라 보안 경계와 책임 분리를 만드는 도구다.

- 어떤 서비스에 접근할 수 있는가
- 어떤 리소스에만 접근할 수 있는가
- 어떤 조건에서만 허용되는가(예: 특정 IP, 특정 MFA, 특정 Tag)

---

## 2. Policy가 인프라 운영에서 갖는 의미

Policy를 제대로 이해하면, "어디에서 권한이 생겼고(상속 경로), 왜 막혔는지(평가 로직)"를 설명할 수 있게 된다. 이는 이후 모든 실습(EC2/VPC/S3/RDS/ECS)에서 발생하는 권한 이슈를 빠르게 정리하는 기반이 된다.

---

# 2. Policy 문서 구조

## 1. Policy의 기본 형태

Policy는 대체로 `Version` + `Statement[]` 구조를 가진다. `Statement`는 규칙 단위이며 여러 개가 동시에 평가된다.

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": ["s3:ListBucket"],
      "Resource": ["arn:aws:s3:::example-bucket"]
    }
  ]
}
```

이 예시는 특정 S3 Bucket에 대해 `ListBucket`을 허용하는 최소 형태다.

### ① Effect / Action / Resource

- `Effect`: `Allow` 또는 `Deny`
- `Action`: 어떤 API를 허용/거부하는가
- `Resource`: 어떤 리소스에 적용되는가(ARN)

### ② Condition(선택)

`Condition`은 "추가 조건이 만족될 때만" 허용/거부를 적용한다.

- MFA로 로그인한 경우에만 허용
- 특정 Source IP에서만 허용
- 특정 Tag가 붙은 리소스에만 허용

이 시리즈는 Console 기반 기본 흐름에 집중하므로, Condition은 "존재 의미"를 이해하는 수준으로만 다룬다.

# 3. Policy 유형

## 1. AWS Managed / Customer Managed / Inline

Policy는 어디에서 관리되고 어떻게 재사용되는지에 따라 유형이 나뉜다.

### ① AWS Managed Policy

AWS가 제공하는 사전 정의 Policy다. 빠르게 권한을 부여할 수 있지만, 범위가 넓을 수 있으므로 실무에서는 그대로 쓰지 않고 기준으로 삼아 커스텀하는 경우가 많다.

예: `AdministratorAccess`, `AmazonS3ReadOnlyAccess`

### ② Customer Managed Policy

사용자가 직접 작성해 계정에서 재사용하는 Policy다. 실무에서 가장 흔한 형태이며 권한 표준화에 유리하다.

`lab04`에서 이 유형을 직접 만든다.

### ③ Inline Policy

특정 User/Group/Role에만 직접 붙는 Policy다. 재사용성이 낮고 운영 추적이 어려워지는 경우가 많아, 특별한 이유가 아니면 Customer Managed Policy를 우선한다.

---

# 4. Policy 평가 로직

## 1. Explicit Deny > Explicit Allow > Implicit Deny

Policy 평가는 "허용 목록 합산"이 아니라, 규칙을 평가해 최종 결정을 내리는 과정이다. 핵심 규칙은 다음이다.

### ① 명시적 Deny가 최우선이다

어떤 Policy에서든 `Effect: Deny`가 적용되면, 다른 Allow가 있어도 최종 결과는 Deny가 된다.

### ② Allow가 없으면 기본은 거부다(Implicit Deny)

아무 Policy에서도 Allow가 나오지 않으면 기본은 거부다. 따라서 "필요한 것만 허용"이 가능한 구조가 된다.

이 평가 로직은 최소 권한(Least Privilege)의 기반이다.

---

# 5. 최소 권한(Least Privilege)

## 1. 최소 권한이 의미하는 것

최소 권한은 "작게 준다"가 아니라, **업무에 필요한 행동(Action)과 대상(Resource)을 정확히 제한한다**는 원칙이다.

### ① 실무에서의 최소 권한 패턴

- 서비스 단위로 제한: S3만, RDS만
- 리소스 단위로 제한: 특정 Bucket만, 특정 DB만
- 조건으로 제한: 특정 IP, 특정 Tag, MFA, VPC Endpoint 등

### ② 이 Lab에서의 범위

이 Section의 실습은 S3를 대상으로 "ReadOnly" 권한을 직접 만들어, 권한이 실제로 어떻게 제한되는지 Console에서 체감하는 것이 목표다. S3 상세(버킷 정책, 퍼블릭 접근 제어)는 Ch06에서 다룬다.

---

# 핵심 정리

- Policy는 Authorization 규칙이며, 권한은 Policy가 Identity(User/Group/Role)에 연결되면서 결정된다.
- Policy의 핵심 필드는 Effect/Action/Resource(필수)와 Condition(선택)이다.
- Policy 유형은 AWS Managed, Customer Managed, Inline으로 나뉘며 실무에서는 Customer Managed가 표준에 가깝다.
- 평가 로직은 `Explicit Deny > Explicit Allow > Implicit Deny`이며 최소 권한 설계의 기반이다.

# [실습] lab04: Custom Policy 생성과 연결

S3 ReadOnly Custom Policy를 생성해 Group에 연결하고, 제한된 권한을 가진 IAM User로 Console에 로그인해 동작을 검증한다. ReadOnly는 성공하고, Upload/Delete 같은 변경 작업은 거부되는 것을 확인한다.

---

### 실습 목표

- Customer Managed Policy를 생성한다(S3 ReadOnly).
- Group을 만들고 Policy를 연결한다.
- 테스트용 IAM User를 생성해 Group에 추가한다.
- Console에서 ReadOnly 동작과 Deny 동작을 확인한다.

⚠️ 비용 주의: 본 실습에서는 S3 Bucket/Object를 테스트 용도로 생성할 수 있으므로 소액 과금이 발생할 수 있다.

---

# 1. 전체 아키텍처

```mermaid
flowchart LR
  Admin[Admin(관리자)] --> S3[S3 Bucket/Object\n(테스트 리소스)]
  Admin --> CP[Customer Managed Policy\nS3 ReadOnly]
  CP --> G[Group]
  G --> U[Test IAM User]
  U --> Console[AWS Console]
  Console -->|List/Get OK| S3
  Console -.->|Put/Delete Deny| S3
```

이 아키텍처는 "정의(Policy) → 적용(Group) → 사용(User) → 검증(Console)" 흐름을 보여준다. ReadOnly 권한은 조회(List/Get)는 허용하지만 변경(Put/Delete)은 차단해야 한다.

# 2. 사전 준비

- 관리자 권한으로 Console 로그인(예: Root 또는 Admin Group User)
- 테스트용 S3 리소스 준비
  - 이 Lab에서는 권한 검증을 위해 최소한의 Bucket/Object를 만든다.
  - S3 상세 설정은 Ch06에서 다룬다.

# 4. 실행 및 결과 검증

설명: 테스트 User로 Console 로그인한 뒤, S3에서 ReadOnly/변경 작업을 시도해 권한이 의도대로 제한되는지 확인한다.

- 테스트 User는 Bucket/Object를 조회(List/Get)할 수 있다.
- 테스트 User는 Upload/Delete 같은 변경 작업을 수행할 수 없다.
- Policy는 User에 직접 붙지 않고 Group을 통해 적용된다.
