## 주의사항

### plan "3단계" 표현에 대해

draft에서 "Refresh -> Diff -> Plan 생성"으로 3단계를 명시했다. 공식 문서는 이를 명시적으로 "3단계"라고 부르지는 않지만, 실제 내부 동작을 설명할 때 다음 세 가지 작업을 언급한다:

1. "Reads the current state of any already-existing remote objects to make sure that the Terraform state is up-to-date" (= Refresh)
2. "Compares the current configuration to the prior state and noting any differences" (= Diff)
3. "Proposes a set of change actions that should, if applied, make the remote objects match the configuration" (= Plan 생성)

따라서 draft의 3단계 모델은 공식 문서의 설명을 정확히 구조화한 것이다. 다만, "공식 문서에서 3단계로 명명한다"는 뉘앙스는 피하는 것이 좋다. draft에서는 "내부적으로 세 단계를 거친다"로 서술하고 있어 적절하다.

### Plan 기호 — `-/+` 와 `+/-` 차이

공식 문서에서 plan 기호는 다음과 같다:

| 기호 | 의미 |
|------|------|
| `+` | create |
| `-` | destroy |
| `~` | update in-place |
| `-/+` | destroy and then create replacement |
| `+/-` | create replacement and then destroy |

draft에서 `-/+`를 "삭제 후 재생성"으로 설명한 것은 맞지만, `+/-` (새 리소스 먼저 생성 후 기존 삭제 — `create_before_destroy` 동작)도 존재한다. Ch03 Sec03에서 `lifecycle > create_before_destroy`를 다루므로 그때 `+/-`를 소개하면 된다. 이 Sec01에서는 `-/+`만 다루는 것이 적절하다.

### `-refresh=false` 제약

`-refresh=false`는 `-refresh-only` 모드와 함께 사용할 수 없다. `-refresh-only`는 refresh만 수행하는 특수 모드이므로 refresh를 끄면 모순이다. draft에서 이 제약을 별도로 언급하지 않는 것은 괜찮다 — 이 시리즈 범위에서 `-refresh-only`를 다루지 않기 때문이다.

### `-target` 사용 주의

공식 문서에서 `-target`은 "exceptional circumstances only"로 제한한다. 인프라 일관성을 깨뜨릴 수 있기 때문이다. draft에서 "의존 관계를 무시할 수 있어 주의가 필요하다"고 경고한 것은 적절하다. 정확히는 `-target`이 의존 관계를 "무시"하는 것이 아니라, **대상 리소스와 그 의존 대상만 처리**하므로 나머지 리소스가 불일치 상태가 될 수 있다.

### State version 4 안정성

State 파일 format version 4는 장기간 안정적으로 유지되고 있다. 2026년 4월 현재까지 version 5로의 변경 계획은 확인되지 않는다. Terraform은 "a new version of Terraform requires a change to the state file's format" 시에만 버전을 올린다.

### `terraform apply`에서 내부 plan 재실행

draft에서 "`terraform apply`를 실행하면 내부적으로 plan을 다시 계산한다"는 설명은 정확하다. 저장된 plan 파일 없이 `terraform apply`를 실행하면 plan 단계가 먼저 수행된다. 저장된 plan 파일을 전달하면 (`terraform apply tfplan`) plan을 다시 계산하지 않고 파일의 계획을 바로 실행한다.

### init이 필요한 시점 표

draft의 표에서 "Backend 설정 변경"(Ch04)과 "모듈 추가/변경"(Ch06)은 정확하다. 추가로, provider configuration이 변경되어도 init이 필요할 수 있다 (새 provider alias 등). 하지만 Ch02 수준에서는 draft의 4가지 시점이 충분하다.
