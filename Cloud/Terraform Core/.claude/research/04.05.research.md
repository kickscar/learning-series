# Research: 04.05 — State 명령어

> 조사일: 2026-04-16
> 기준: Terraform 1.14.x (로컬 확인: v1.14.7), hashicorp/aws provider ~> 6.0

## 검증 결과

| 항목 | 상태 | 비고 |
|------|------|------|
| `terraform state list` 기본 동작 | ✓ 정확 | 모든 관리 리소스 주소를 출력한다 |
| `terraform state list` 주소 필터 | ✓ 정확 | `terraform state list module.vpc` 형태로 모듈/리소스 주소 필터 가능 |
| `terraform state list -id=` | ✓ 정확 | `-id=sg-1234abcd` 형태로 provider ID 기준 역조회 가능 |

| `terraform import` CLI 문법 | ✓ 정확 | `terraform import <address> <id>` |
| `import` block (HCL 기반) | ✓ 정확 | TF 1.5+에서 도입, 1.14.x에서 완전 지원. `import { to = ... id = ... }` |
| `import` block `for_each` | ✓ 정확 | 지원됨. map 또는 list 사용 가능 |

| `moved` block (state mv 대안) | ✓ 정확 | TF 1.1+에서 도입. HCL 선언형으로 state mv 대체 |
| `removed` block (state rm 대안) | ✓ 정확 | `removed { from = ... lifecycle { destroy = false } }` 형태 |
| `state pull` / `state push` | ✓ 정확 | 존재하나 Remote Backend 운용 시 필요, 이 섹션에서는 언급 수준 |
| `state identities` (신규 서브커맨드) | ⚠ 주의 | TF 1.14.x에서 추가된 서브커맨드. identity 기반 import와 연계 |
| `state replace-provider` | ✓ 정확 | Provider FQN 변경용. 이 섹션 범위 밖이나 존재 인지 필요 |

## 상세 조사

### Q1. `terraform state mv` — 모듈 간 이동 가능한가?

**가능하다.** 공식 문서 예시:

```bash
# 루트 → 모듈
terraform state mv packet_device.worker module.worker.packet_device.worker

# 모듈 → 모듈 (전체 모듈 이동)
terraform state mv module.app module.parent.module.app

# count/for_each 인스턴스 간 이동
terraform state mv 'packet_device.worker[0]' 'packet_device.helper[0]'
terraform state mv 'packet_device.worker["example123"]' 'packet_device.helper["example456"]'
```

제약: SOURCE와 DESTINATION은 동일한 리소스 타입이어야 한다.

---

### Q2. `terraform state rm` — 확인 프롬프트?

**프롬프트 없이 즉시 실행된다.** 공식 문서에 확인 절차 언급이 없으며, 실행 전 자동으로 backup 파일을 생성한다. 안전한 사용을 위해 `-dry-run` 플래그를 먼저 사용하는 것을 권장한다.

```bash
# 사전 확인
terraform state rm -dry-run 'aws_instance.web'

# 실제 제거
terraform state rm 'aws_instance.web'
```

---

### Q3. `import` block (HCL 기반) — TF 1.14.x 지원 현황

TF 1.5에서 도입, 1.14.x에서 완전 지원된다. CLI `terraform import`와의 비교:

| 구분 | CLI `terraform import` | HCL `import` block |
|------|----------------------|-------------------|
| 선언 방식 | 명령형 (CLI 실행) | 선언형 (HCL 코드) |
| 실행 시점 | 즉시 | `terraform plan` / `terraform apply` 시 |
| 코드 자동 생성 | 불가 | `-generate-config-out` 플래그로 가능 |
| `for_each` | 불가 | 지원 (map, list) |
| CI/CD 연동 | 스크립트 필요 | 코드 기반으로 자연스러움 |
| 반복 실행 | 멱등하지 않음 (이미 있으면 에러) | 멱등 (이미 있으면 무시) |

```hcl
# 단일 리소스
import {
  to = aws_instance.web
  id = "i-0abc1234567890def"
}

# for_each 사용
locals {
  buckets = {
    "staging" = "bucket-staging-123"
    "prod"    = "bucket-prod-456"
  }
}

import {
  for_each = local.buckets
  to       = aws_s3_bucket.this[each.key]
  id       = each.value
}
```

### Q4. `terraform import` — 코드 자동 생성?

- **CLI `terraform import`**: State에만 기록, 코드를 생성하지 않는다. 사용자가 직접 resource block을 작성해야 한다.
- **HCL `import` block + `-generate-config-out`**: `terraform plan -generate-config-out="generated.tf"` 실행 시 provider에서 attribute를 조회하여 HCL 코드를 자동 생성한다.

주의사항:
- 생성된 코드는 "best guess" — 반드시 수동 검토/편집이 필요하다
- 상충하는 인수(예: `ipv6_address_count` vs `ipv6_addresses`)가 동시에 생성될 수 있다
- 기존 파일 경로를 지정하면 에러 — 새 파일 경로만 허용

---

### Q5. `state pull` / `state push` — 언급 가치?

- `state pull`: 현재 State를 stdout으로 출력 (JSON). Remote Backend 디버깅, 수동 검사 시 유용
- `state push`: 로컬 State 파일을 Remote Backend에 밀어넣기. `-force` 플래그 존재
- **이 섹션에서의 위치**: "기타 State 명령어" 정도로 간단히 언급하되, 상세 실습은 Ch08 (Backend) 또는 해당 섹션에서 다루는 것이 적절하다

---

### Q6. `terraform state mv -dry-run` 존재 여부

**존재한다.** 1.14.x 기준 확인 완료:

```bash
terraform state mv -dry-run 'aws_instance.old' 'aws_instance.new'
```

실제 이동 없이 어떤 리소스가 이동될지 출력한다.

---

### Q7. `terraform state list` — 패턴 필터링

**주소 기반 필터를 지원한다.** glob 패턴은 아니며, 리소스 주소를 직접 인수로 전달한다.

```bash
# 모듈 내 모든 리소스
terraform state list module.vpc

# 특정 리소스 (count 포함)
terraform state list aws_instance.web
# 출력: aws_instance.web[0], aws_instance.web[1], ...

# ID 기반 역조회
terraform state list -id=sg-1234abcd
```

여러 주소를 동시에 전달 가능: `terraform state list module.vpc aws_instance.web`

---

### Q8. TF 1.14.x 신규 State 서브커맨드

**`terraform state identities`** — 1.14.x에서 추가된 새로운 서브커맨드다.
- `-json` 플래그 필수
- 리소스 identity 정보를 JSON 형태로 출력
- `import` block의 `identity` attribute와 연계되는 기능
- 주소 필터 및 `-id=` 플래그 지원

```bash
terraform state identities -json
terraform state identities -json module.vpc
```

> 이 섹션의 핵심 주제는 아니지만, State 명령어 전체 목록에서 존재를 언급할 가치가 있다.

---

### Q9. `import` block — `for_each` 지원

**지원된다.** map 또는 list(set of string) 사용 가능:

```hcl
# map 사용
import {
  for_each = local.buckets
  to       = aws_s3_bucket.this[each.key]
  id       = each.value
}

# list 사용 (인덱스 기반 리소스)
import {
  for_each = toset(["bucket-a", "bucket-b"])
  to       = aws_s3_bucket.this[each.key]
  id       = each.value
}
```

### Q10. `terraform state rm` 후 plan 결과

**plan에서 "to be created"로 표시된다.** 공식 문서 원문:

> "a subsequent `terraform plan` will include an action to create a new object for each of the 'forgotten' instances."

실제 인프라는 그대로 존재하지만, Terraform은 더 이상 관리하지 않으므로 코드에 해당 resource block이 남아있으면 새로 생성하려 시도한다. 이를 방지하려면:
1. resource block을 코드에서 삭제하거나
2. `removed` block으로 대체한다

## 섹션 구성 제안사항

### HCL 선언형 대안 언급 수준

`moved`, `removed`, `import` block은 각각 `state mv`, `state rm`, CLI `terraform import`의 HCL 선언형 대안이다. 이 섹션은 State 명령어(CLI) 중심이므로:

- 각 CLI 명령어 설명 후 "HCL 선언형 대안으로 `moved` block이 있다" 정도로 존재를 알리고
- `import` block은 Lab03에서 CLI import와 비교하여 실습에 포함할 수 있다
- `moved`, `removed` block 상세는 별도 섹션(Ch05 리팩토링 또는 해당 챕터)에서 다루는 것이 적절하다

### Lab 구성 적합성

| Lab | 내용 | 적합성 |
|-----|------|--------|
| Lab01 | `state list` / `state show` | ✓ 적합 — 기존 State가 있는 프로젝트에서 조회 |
| Lab02 | `state mv` (rename) / `state rm` + plan 확인 | ✓ 적합 — `-dry-run` 사용 후 실행, rm 후 plan 확인 |
| Lab03 | CLI `import` + 코드 동기화 | ✓ 적합 — 수동 생성 리소스 → import → resource block 작성 |

Lab03에서 `import` block 방식도 함께 비교하면 학습 효과가 높다.

## 주의사항

- `terraform state rm`은 확인 프롬프트 없이 즉시 실행 — Lab에서 반드시 `-dry-run` 먼저 사용하는 패턴을 보여줘야 한다
- `terraform import` CLI는 코드를 생성하지 않으므로 import 후 `terraform plan`에서 diff가 나올 수 있다 — resource block을 정확히 맞춰야 no-change가 된다
- `import` block의 `-generate-config-out`으로 생성된 코드는 항상 수동 검토가 필요하다 (conflicting arguments 가능)
- `terraform state mv`의 SOURCE/DESTINATION은 동일한 리소스 타입이어야 한다
- Shell에서 `for_each` 키를 포함한 주소는 따옴표로 감싸야 한다: `'aws_instance.web["key"]'`
- `state identities`는 1.14.x 신규 기능 — 아직 초기 단계이므로 간단히 언급만 권장
