# Research: 03.03 — Destroy와 리소스 생명주기

> 조사일: 2026-04-15
> 기준: Terraform 1.14.x, hashicorp/aws provider ~> 6.0

## 검증 결과

| 항목 | 상태 | 비고 |

| destroy 역순 삭제 (의존 관계 그래프 역전) | ✓ 정확 | 03.01/03.02에서 검증 완료. `terraform destroy`는 내부적으로 `terraform apply -destroy`와 동일 |
| `lifecycle` 블록 — resource 전용 | ✓ 정확 | `create_before_destroy`, `prevent_destroy`, `ignore_changes`, `replace_triggered_by`는 resource 블록에서만 사용 |

## 주의사항

### `prevent_destroy`는 전체 destroy를 차단한다

`prevent_destroy = true`가 설정된 리소스가 하나라도 있으면 `terraform destroy` 명령 전체가 실패한다. 해당 리소스만 건너뛰는 것이 아니라, plan 단계에서 에러를 반환하여 어떤 리소스도 삭제되지 않는다.

해제 방법:
1. `prevent_destroy = false`로 변경 후 `terraform destroy` 실행
2. 또는 resource 블록 자체를 코드에서 제거 — 이 경우 prevent_destroy도 함께 사라지므로 삭제가 진행됨

공식 문서 원문: "Terraform rejects plans that would destroy the infrastructure object"

**draft에서 주의할 점**: "prevent_destroy를 설정하면 해당 리소스를 삭제할 수 없다"가 아니라, "해당 리소스를 포함하는 destroy plan 전체가 거부된다"로 정확히 서술해야 한다.

### `create_before_destroy`의 고유 이름 제약

공식 문서에서 명시적으로 경고한다: "This is an opt-in behavior because many remote object types have unique name requirements." AWS 예시:
- `aws_security_group`에 `name`을 지정하면 같은 이름의 SG가 동시에 존재할 수 없어 생성 실패
- `aws_instance`는 `Name` 태그만 사용하므로 이 제약이 없음

또한 의존 리소스에 `create_before_destroy`가 암묵적으로 전파된다. 리소스 A가 `create_before_destroy = true`이고 리소스 B가 A에 의존하면, B에도 create_before_destroy가 암묵적으로 적용된다. B에서 이를 `false`로 재정의하면 cycle이 발생할 수 있다.

### `create_before_destroy`와 destroy provisioner

`create_before_destroy = true`를 설정하면 destroy-time provisioner가 실행되지 않는다. 이 시리즈에서는 provisioner를 본격적으로 다루지 않으므로 간략히 언급만 하면 충분하다.

### `ignore_changes` — 중첩 속성 지원

공식 문서에서 다음 표기를 지원한다고 확인:
- 단순 속성: `tags`
- Map 요소: `tags["Name"]`
- List 요소: `list[0]`

`ignore_changes`는 meta-argument 자체(`lifecycle`, `depends_on` 등)에는 적용할 수 없다. 리소스 타입이 정의한 속성만 대상이다.

### `replace_triggered_by` — 허용 표현식 정리

| 표현식 유형 | 예시 | 동작 |
|------------|------|------|
| managed resource 참조 | `aws_ecs_service.svc` | 해당 리소스의 update/replace 시 교체 트리거 |
| 리소스 인스턴스 참조 | `aws_instance.web[0]` | 특정 인스턴스의 변경 시 트리거 |
| 인스턴스 속성 참조 | `aws_ecs_service.svc.id` | 속성값 변경 시 트리거 |

**사용 불가**: `var.*`, `local.*`, `data.*` — plain values는 `terraform_data` 리소스로 감싸야 한다.

`terraform_data` 리소스의 `triggers_replace` 인수를 사용하면 plain value 변경을 리소스 교체로 변환할 수 있다:

```hcl
resource "terraform_data" "replacement" {
  triggers_replace = var.revision
}

resource "aws_instance" "web" {
  # ...
  lifecycle {
    replace_triggered_by = [terraform_data.replacement]
  }
}
```

### `terraform taint` — deprecated이지 removed 아님

TF 0.15.2에서 `terraform apply -replace` 도입과 함께 deprecated 선언. 1.14.x에서도 여전히 실행 가능하나 공식 문서에서 "This command is deprecated"라고 명시한다.

`-replace`가 우월한 이유: taint는 state를 즉시 수정하므로 다른 팀원이 tainted 상태의 리소스로 plan을 실행할 위험이 있다. `-replace`는 plan에 교체 의도를 포함하므로 리뷰 후 apply할 수 있다.

### `precondition` / `postcondition` — lifecycle 내부 but 별개 목적

`precondition`과 `postcondition`은 lifecycle 블록 내부에 위치하지만, 리소스 생명주기 제어(`create_before_destroy` 등)와는 성격이 다르다. 검증(validation) 목적이다.

- **precondition**: plan 시 리소스 생성/변경 전에 평가. 실패 시 plan 중단
- **postcondition**: apply 후 리소스 생성/변경 후에 평가. 실패 시 의존 리소스 처리 중단
- 도입 버전: TF 1.2.0

이 시리즈에서는 Ch09 Sec02 (Lifecycle Guards)에서 다룬다. Ch03 Sec03에서는 존재만 언급하고 넘어가는 것이 적절하다.

### `action_trigger` — TF 1.14.x의 새 lifecycle 인수

공식 문서(v1.14.x)에 `action_trigger`가 새 lifecycle 인수로 등장한다. events 기반으로 리소스 생성/수정 전후에 action을 트리거하는 기능이다.

- **events**: `before_create`, `after_create`, `before_update`, `after_update`
- **condition**: 선택적 boolean 조건
- **actions**: 실행할 action 목록

정확한 도입 버전과 action 유형에 대한 추가 조사가 필요하다. Ch03 Sec03에서 이 기능을 본격적으로 다루기보다는, lifecycle 블록의 인수 목록에서 "존재"만 언급하고 향후 심화로 미루는 것이 적절하다. 이 시리즈의 scope (lifecycle 기본 제어)를 벗어나는 고급 기능이다.

### `removed` 블록의 `lifecycle { destroy }` — 일반 lifecycle과 혼동 주의

TF 1.7에서 도입된 `removed` 블록은 자체적으로 `lifecycle { destroy = false }` 인수를 갖는다. 이는 일반 resource의 lifecycle 인수가 아니다. "코드에서 리소스를 제거하되 실제 인프라는 유지"하는 용도이다.

Ch03 Sec03에서는 언급하지 않는 것이 좋다. State Management (Ch04) 또는 별도 시점에서 다루는 것이 자연스럽다.

### `terraform apply -replace` 복수 지정

공식 문서 확인: "Include this option multiple times to replace several objects at once."

```bash
$ terraform apply -replace=aws_instance.web -replace=aws_security_group.main
```

`-replace`는 `-destroy` 옵션과 함께 사용할 수 없다 (상호 배타적 planning mode).

### Plan 기호 정리 — `-/+` vs `+/-`

03.01 research에서 확인한 내용 재정리:

| 기호 | 의미 | lifecycle 설정 |
|------|------|---------------|
| `-/+` | destroy and then create (기본) | `create_before_destroy` 미설정 |
| `+/-` | create and then destroy | `create_before_destroy = true` |

draft에서 `+/-`를 이번 Sec03에서 소개하는 것이 적절하다 (03.01 research에서 "Ch03 Sec03에서 `+/-`를 소개하면 된다"고 권고한 바 있음).

## Sec03 draft에서 다룰 범위 권고

### 핵심 (반드시 포함)

1. destroy 역순 삭제 원리 — 03.01/03.02 내용 환기 후 간결하게
2. `lifecycle` 블록 4대 인수: `create_before_destroy`, `prevent_destroy`, `ignore_changes`, `replace_triggered_by`
3. `terraform apply -replace=<address>` — taint 대체
4. `terraform destroy -target=<address>` — 특정 리소스 삭제

### 언급만 (상세 미포함)

- `precondition` / `postcondition` — Ch09 Sec02에서 본격 다룸
- `action_trigger` — 고급 기능, 이 시리즈 범위 외 또는 향후 추가
- `terraform taint` — deprecated 상태 언급, `-replace` 권장

### 제외

- `removed` 블록 — Ch04 State Management 영역
- provisioner 관련 세부사항 — 이 시리즈에서 provisioner를 본격적으로 다루지 않음
