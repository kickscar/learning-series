# 1. AWS를 보는 관점

## 1. AWS의 성격

AWS(Amazon Web Services)는 Compute, Storage, Database, Networking 같은 인프라 구성 요소를 "서비스"로 제공하는 Public Cloud다. 사용자는 장비를 구매하고 설치하는 대신, Console에서 리소스를 생성하고 연결해 원하는 인프라 상태를 만든다.

이 시리즈는 AWS 서비스를 기능 목록으로 외우는 것이 아니라, 서비스들이 **하나의 배포 가능한 구조**로 연결되는 흐름을 이해하는 것을 목표로 한다.

### ① 주요 서비스 카테고리(학습 관점)

- Compute: EC2, ECS
- Networking: VPC, Subnet, Route Table, IGW, NAT Gateway, ALB
- Storage/CDN: S3, CloudFront
- Database: RDS
- Security/Identity: IAM, Security Group, Network ACL

카테고리는 "서비스 분류"이면서 동시에 아키텍처를 구성하는 블록이다. 뒤의 Chapter들은 이 블록들을 실제로 연결하면서, 요청 흐름과 보안 경계를 설명할 수 있는 상태를 만든다.

### ② Console 기반 학습의 의미

AWS는 API 기반으로 자동화할 수도 있지만, 이 시리즈는 IaC/CLI를 다루지 않는다. Console에서 리소스를 직접 만들고 연결하면서 "무엇이 어디에 붙어 있고, 어디에서 트래픽이 흐르는지"를 눈으로 확인하는 경험을 우선한다.

자동화는 다음 시리즈(AWS IaC with CloudFormation)에서 다룬다.

# 2. AWS Global Infrastructure

## 1. Region / Availability Zone / Edge Location

AWS Global Infrastructure는 "어디에 리소스를 생성하고, 어디에서 사용자에게 서비스되는가"를 결정하는 물리적 기반이다. 이 구조를 이해하지 못하면, 실습 중 리소스가 보이지 않거나(Region mismatch), 성능/가용성/비용 의사결정이 흔들린다.

### ① Region

Region은 지리적으로 분리된 데이터센터 클러스터다. 대부분의 리소스(EC2, VPC, RDS 등)는 특정 Region에 생성된다.

Region은 다음을 결정한다.

- 사용자의 지연 시간(Latency)
- 데이터 위치와 규정 준수(Compliance)
- 서비스 가용성(Region별 제공 서비스)
- 비용(Region별 가격 차이)

### ② Availability Zone(AZ)

Availability Zone(AZ)은 Region 내부에서 물리적으로 분리된 데이터센터 묶음이다. Multi-AZ 아키텍처는 같은 Region 안에서 AZ를 분산해 장애 범위를 줄이는 방식으로 구현된다.

이 시리즈에서 중요한 결론은 두 가지다.

- Subnet은 AZ 단위로 만들어진다(Ch04).
- 고가용성은 "AZ 분산 + 상위 레이어(Load Balancer/Auto Scaling)"로 완성된다(Ch05).

### ③ Edge Location

Edge Location은 CloudFront 같은 CDN이 콘텐츠를 전달하기 위한 배포 거점이다. 사용자는 가까운 Edge에서 콘텐츠를 받아 지연 시간을 줄일 수 있다.

Edge Location은 "리소스를 생성하는 장소"가 아니라, "콘텐츠를 전달하는 장소"에 가깝다. 이 시리즈에서는 Ch06에서 S3 정적 웹 호스팅과 CloudFront를 다룰 때 다시 연결한다.

## 2. Global Infrastructure를 인프라 관점으로 묶기

```mermaid
flowchart TB
  User[사용자] --> Edge[Edge Location\n(CloudFront)]
  User --> Region[Region]
  Region --> AZ1[AZ-a]
  Region --> AZ2[AZ-b]
  AZ1 --> Subnet1[Subnet]
  AZ2 --> Subnet2[Subnet]
  Subnet1 --> R1[리소스\n(EC2/RDS 등)]
  Subnet2 --> R2[리소스\n(EC2/RDS 등)]
```

이 다이어그램은 "사용자 접근"이 항상 Region으로 직접 들어오는 것이 아니라, 콘텐츠는 Edge로 먼저 전달될 수 있음을 보여준다. 또한 대부분의 인프라 리소스는 Region 안에서 AZ와 Subnet을 통해 배치되며, Multi-AZ는 같은 Region 내부에서 구현된다는 점이 핵심이다.

# 3. Region 선택 기준

## 1. 선택 기준(실무 기준)

Region 선택은 기술적 선택이면서 동시에 비용/규정/운영 선택이다.

### ① Latency

사용자와 가까운 Region을 선택하면 지연 시간이 줄어든다. 특히 API 응답이나 DB 왕복이 많은 시스템에서는 체감이 크다.

### ② Compliance / Data Residency

산업/국가 규제에 따라 데이터가 저장되는 물리적 위치가 중요할 수 있다. 이 경우 Region 선택은 제약 조건이 된다.

### ③ Service Availability

모든 AWS 서비스가 모든 Region에 동일하게 제공되지는 않는다. 특정 기능이 필요한 경우, Region별 지원 여부가 선택을 제한한다.

### ④ Cost

동일 서비스라도 Region에 따라 비용이 달라질 수 있다. 실습과 운영에서 비용은 항상 고려 대상이다.

## 2. 본 시리즈의 Region 기준

이 시리즈는 **`ap-northeast-2 (Seoul)`** Region을 기준으로 진행한다.

이 기준은 다음을 단순화하기 위한 선택이다.

- 실습 중 리소스가 보이지 않는 문제(Region mismatch)를 줄인다.
- Chapter 전반에서 동일 Region을 사용해 리소스가 누적되는 흐름을 만든다.

다만 조직/개인 환경에 따라 다른 Region을 사용할 수 있다. 중요한 것은 "모든 Lab을 동일 Region에서 일관되게 수행한다"는 점이다.

# 핵심 정리

- AWS는 인프라 구성 요소를 서비스로 제공하는 Public Cloud이며, 이 시리즈는 서비스를 연결해 배포 가능한 구조를 만드는 흐름을 학습한다.
- Global Infrastructure의 핵심 단위는 Region(리소스 생성 단위), AZ(고가용성 분산 단위), Edge Location(콘텐츠 전달 단위)이다.
- Region 선택은 Latency/Compliance/Service Availability/Cost를 함께 고려하는 실무 의사결정이다.
- 본 시리즈는 `ap-northeast-2 (Seoul)`을 기준으로 진행하며, Lab은 동일 Region에서 일관되게 수행해야 한다.
