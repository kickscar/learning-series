# Research: 07.03 — 디렉토리 기반 환경 분리

> 조사일: 2026-04-29
> 기준: Terraform >= 1.14.0, hashicorp/aws provider ~> 6.0

## 검증 결과

| 항목 | 상태 | 비고 |
|------|------|------|
| `envs/dev/`, `envs/prod/` 패턴 — 각 env가 독립 root module | ✓ 정확 | HashiCorp 공식 튜토리얼에서 이 구조 사용 |
| 각 env 디렉토리에서 독립 `terraform init` 실행 | ✓ 정확 | 디렉토리별 `.terraform/` 생성 |
| 공유 모듈 참조 (`../../modules/`) | ✓ 정확 | 상대경로로 참조 가능 |
| Partial backend configuration (`-backend-config`) | ✓ 정확 | 파일 기반, key-value 기반 모두 지원 |
| Backend config 파일 형식 (HCL, `.tfbackend` 권장) | ✓ 정확 | `terraform` / `backend` 블록 감싸지 않음 |
| Backend 블록에서 변수·표현식 사용 불가 | ✓ 정확 | 공식 문서 명시: "cannot refer to named values" |
| `terraform.tfvars` 자동 로드 (디렉토리별) | ✓ 정확 | 해당 디렉토리의 `terraform.tfvars`만 자동 로드 |
| `*.auto.tfvars` lexical 순서 자동 로드 | ✓ 정확 | |
| 동일 S3 버킷, 다른 key prefix로 환경 분리 가능 | ✓ 정확 | `key` 인수에 환경 prefix 포함 |
| Workspace보다 디렉토리 분리가 강한 격리 제공 | ✓ 정확 | HashiCorp 공식 권장 |
| TF 1.14.x 디렉토리 분리 관련 변경사항 | — 없음 | 기본 동작에 변경 없음 |

## 상세 조사

### 1. 디렉토리 구조 패턴

#### 기본 구조

환경별 디렉토리를 root module로 사용하고, 공통 인프라 코드를 모듈로 분리한다.

```text
project/
├── modules/                    # 공유 모듈
│   ├── vpc/
│   │   ├── main.tf
│   │   ├── variables.tf
│   │   └── outputs.tf
│   ├── sg/
│   └── instance/
└── envs/
    ├── dev/                    # dev 환경 root module
    │   ├── main.tf             # module 호출
    │   ├── variables.tf
    │   ├── outputs.tf
    │   ├── backend.tf          # dev 전용 backend 설정
    │   └── terraform.tfvars    # dev 전용 변수값
    └── prod/                   # prod 환경 root module
        ├── main.tf
        ├── variables.tf
        ├── outputs.tf
        ├── backend.tf
        └── terraform.tfvars
```

#### 각 환경 디렉토리의 역할

각 `envs/{env}/` 디렉토리는 **독립된 root module**이다. 각 디렉토리에서:

- `terraform init`을 독립 실행한다 (자체 `.terraform/` 생성)
- `terraform plan` / `apply`를 독립 실행한다
- 자체 state를 가진다 (backend 설정에 따라 경로 결정)
- 자체 `.terraform.lock.hcl`을 생성한다

#### 모듈 참조 경로

`envs/dev/main.tf`에서 공유 모듈을 상대경로로 참조한다.

```hcl
module "vpc" {
  source = "../../modules/vpc"

  cidr_block  = var.vpc_cidr
  environment = var.env
}

module "instance" {
  source = "../../modules/instance"

  instance_type = var.instance_type
  subnet_id     = module.vpc.public_subnet_id
  sg_id         = module.sg.instance_sg_id
}
```

`source`의 상대경로는 **호출하는 파일이 위치한 디렉토리** 기준이다. `envs/dev/main.tf`에서 `../../modules/vpc`는 `project/modules/vpc`를 가리킨다.

### 2. Backend 설정 분리

#### 방법 A: 환경별 `backend.tf` 직접 작성

각 환경 디렉토리에 `backend.tf`를 배치한다. 가장 단순한 방식이다.

**`envs/dev/backend.tf`:**

```hcl
terraform {
  backend "s3" {
    bucket = "tf-core-tfstate"
    key    = "dev/gallery/terraform.tfstate"
    region = "ap-northeast-2"
  }
}
```

**`envs/prod/backend.tf`:**

```hcl
terraform {
  backend "s3" {
    bucket = "tf-core-tfstate"
    key    = "prod/gallery/terraform.tfstate"
    region = "ap-northeast-2"
  }
}
```

**장점**: 코드만 보면 어디에 state가 저장되는지 즉시 확인 가능
**단점**: backend 설정이 환경 디렉토리마다 중복

#### 방법 B: Partial backend configuration

`backend.tf`에는 최소 backend 타입만 선언하고, 환경별 값은 별도 파일로 분리한다.

**`envs/dev/backend.tf` (공통, 동일 내용):**

```hcl
terraform {
  backend "s3" {}
}
```

**`backend-dev.hcl`:**

```hcl
bucket = "tf-core-tfstate"
key    = "dev/gallery/terraform.tfstate"
region = "ap-northeast-2"
```

**`backend-prod.hcl`:**

```hcl
bucket = "tf-core-tfstate"
key    = "prod/gallery/terraform.tfstate"
region = "ap-northeast-2"
```

**init 실행:**

```bash
$ cd envs/dev
$ terraform init -backend-config="../../backend-dev.hcl"
```

```bash
$ cd envs/prod
$ terraform init -backend-config="../../backend-prod.hcl"
```

#### Partial backend config 파일 형식

- **HCL 형식**이다 (JSON이 아님)
- `terraform` / `backend` 블록으로 감싸지 않는다 — 키-값 쌍만 작성
- 공식 권장 확장자: `*.backendname.tfbackend` (예: `config.s3.tfbackend`)
- 관례적 확장자: `.hcl` 도 널리 사용됨

#### Partial backend config — 커맨드라인 key-value

파일 대신 인라인으로 전달할 수도 있다.

```bash
$ terraform init \
  -backend-config="bucket=tf-core-tfstate" \
  -backend-config="key=dev/gallery/terraform.tfstate" \
  -backend-config="region=ap-northeast-2"
```

CI/CD 파이프라인에서 환경변수로 조립할 때 유용하다.

#### Backend 블록 제약사항 (재확인)

Backend 블록은 정적이다. 공식 문서:

> "A backend block cannot refer to named values (like input variables, locals, or data source attributes)"

즉, 다음은 **불가능**하다:

```hcl
# 불가 — backend 블록에서 변수 참조
terraform {
  backend "s3" {
    bucket = var.state_bucket    # ERROR
    key    = "${var.env}/terraform.tfstate"  # ERROR
  }
}
```

이 제약 때문에 환경별 backend 분리에는 (A) 디렉토리별 하드코딩 또는 (B) partial config가 필요하다.

#### S3 Key 패턴 — 환경 분리

동일 S3 버킷에서 key prefix로 환경을 분리하는 패턴이다.

| 환경 | S3 Key |
|------|--------|
| dev | `dev/gallery/terraform.tfstate` |
| prod | `prod/gallery/terraform.tfstate` |

별도 버킷을 사용하는 패턴도 있다:

| 환경 | Bucket | Key |
|------|--------|-----|
| dev | `tf-core-tfstate-dev` | `gallery/terraform.tfstate` |
| prod | `tf-core-tfstate-prod` | `gallery/terraform.tfstate` |

동일 버킷 + key prefix 방식이 관리가 단순하다. 버킷 분리는 환경별 AWS 계정이 다를 때 사용한다.

### 3. tfvars 환경별 관리

#### 디렉토리 구조에서의 tfvars

각 환경 디렉토리에 `terraform.tfvars`를 배치하면 해당 디렉토리에서 `terraform plan/apply` 실행 시 **자동 로드**된다.

**`envs/dev/terraform.tfvars`:**

```hcl
env           = "dev"
instance_type = "t3.micro"
```

**`envs/prod/terraform.tfvars`:**

```hcl
env           = "prod"
instance_type = "t3.small"
```

자동 로드 파일 목록 (Terraform이 해당 디렉토리에서 자동 탐색):

| 파일 | 로드 조건 |
|------|----------|
| `terraform.tfvars` | 존재하면 자동 로드 |
| `terraform.tfvars.json` | 존재하면 자동 로드 (`terraform.tfvars`보다 우선) |
| `*.auto.tfvars` | 존재하면 lexical 순서로 자동 로드 |
| `*.auto.tfvars.json` | 존재하면 lexical 순서로 자동 로드 |

#### `-var-file` 방식 (대안)

디렉토리별 `terraform.tfvars` 대신, 프로젝트 루트에 환경별 tfvars를 두고 `-var-file`로 지정할 수도 있다.

```text
project/
├── dev.tfvars
├── prod.tfvars
├── main.tf
├── variables.tf
└── backend.tf
```

```bash
$ terraform plan -var-file="dev.tfvars"
$ terraform plan -var-file="prod.tfvars"
```

이 방식은 **단일 root module + tfvars 분리** 패턴이다. 디렉토리 기반 분리와는 다른 접근이며, backend까지 분리하기 어렵다는 한계가 있다 (backend는 `-backend-config`로 별도 처리 필요).

#### 변수 우선순위 (전체 — 높은 쪽 → 낮은 쪽)

```text
1. -var / -var-file (명령줄 나열 순서대로, 마지막 값 적용)
2. *.auto.tfvars / *.auto.tfvars.json (lexical 순서)
3. terraform.tfvars.json
4. terraform.tfvars
5. TF_VAR_<name> 환경 변수
6. variable 블록의 default 값
```

디렉토리 기반 분리에서는 각 환경 디렉토리의 `terraform.tfvars`가 자동 로드되므로 별도 `-var-file` 플래그 없이 환경별 값이 적용된다. 이것이 디렉토리 분리의 편리함이다.

### 4. Workspace 대비 장점

#### ① 완전한 코드 격리

각 환경이 독립된 root module이다. dev에만 필요한 리소스를 추가하거나, prod에만 있는 리소스를 구성할 때 `count` 조건식이 필요 없다. 코드 자체가 다를 수 있다.

```hcl
# envs/prod/main.tf — prod에만 RDS 추가
module "rds" {
  source = "../../modules/rds"
  # ...
}
```

Workspace에서는 동일 코드를 공유하므로 이런 차이를 `count = terraform.workspace == "prod" ? 1 : 0` 같은 조건으로 처리해야 한다.

#### ② 환경별 다른 Backend 자격 증명

각 환경 디렉토리가 독립 backend를 가지므로 다른 AWS 계정, 다른 IAM Role을 사용할 수 있다.

```hcl
# envs/dev/backend.tf — dev 계정
terraform {
  backend "s3" {
    bucket  = "dev-account-tfstate"
    key     = "gallery/terraform.tfstate"
    region  = "ap-northeast-2"
    profile = "dev-account"
  }
}
```

```hcl
# envs/prod/backend.tf — prod 계정
terraform {
  backend "s3" {
    bucket  = "prod-account-tfstate"
    key     = "gallery/terraform.tfstate"
    region  = "ap-northeast-2"
    profile = "prod-account"
  }
}
```

Workspace에서는 단일 backend를 공유하므로 환경별 자격 증명 분리가 불가능하다.

#### ③ Workspace 전환 실수 방지

Workspace에서는 `terraform workspace select` 전환을 잊으면 잘못된 환경에 `apply`할 수 있다. 디렉토리 분리에서는 `cd envs/prod`로 이동해야 하므로 물리적으로 구분된다.

HashiCorp 공식 문서:

> "Workspaces are not appropriate for system decomposition or deployments requiring separate credentials and access controls."

#### ④ CI/CD 파이프라인 명확성

디렉토리 경로가 파이프라인 트리거와 직결된다.

```yaml
# GitHub Actions 예시
jobs:
  deploy-dev:
    if: github.ref == 'refs/heads/develop'
    defaults:
      run:
        working-directory: envs/dev

  deploy-prod:
    if: github.ref == 'refs/heads/main'
    defaults:
      run:
        working-directory: envs/prod
```

Workspace 기반에서는 동일 디렉토리에서 `terraform workspace select`를 파이프라인 내에서 실행해야 한다.

#### ⑤ 환경별 독립 lock 파일

각 환경 디렉토리가 자체 `.terraform.lock.hcl`을 가진다. Provider 버전 업그레이드를 환경별로 독립 관리할 수 있다 (dev에서 먼저 업그레이드 테스트 후 prod 적용).

### 5. 단점

#### ① 코드 중복 위험

각 환경 디렉토리의 `main.tf`가 유사한 module 호출을 포함한다. 모듈 인터페이스가 변경되면 모든 환경 디렉토리를 업데이트해야 한다.

**완화 방법:**

- 공유 모듈에 인프라 로직을 집중 → 환경 디렉토리는 "모듈 호출 + 변수 주입" 역할만
- 환경별 `main.tf`를 최소화하면 drift 가능성 감소

```hcl
# envs/dev/main.tf — 최소한의 호출만
module "infra" {
  source = "../../modules/infra"

  env           = var.env
  instance_type = var.instance_type
}
```

#### ② 모듈 동기화 관리

공유 모듈의 인터페이스(변수, 출력)가 바뀌면 모든 환경 디렉토리에서 해당 변경을 반영해야 한다. 환경이 많아질수록 동기화 부담이 증가한다.

**완화 방법:**

- 모듈 버전 태깅 (Git tag 기반 `source`) — 각 환경이 다른 모듈 버전을 사용 가능
- CI에서 모든 환경 디렉토리를 `terraform validate`로 검증

#### ③ 디렉토리 수 증가

환경 수 x 레이어 수만큼 디렉토리가 생긴다. 3 환경(dev, stg, prod) x 3 레이어(network, platform, workload)면 9개 디렉토리다.

### 6. HashiCorp 공식 입장

#### Workspace vs 디렉토리 — 공식 권장

HashiCorp 공식 문서에서 명시적으로 언급한 내용:

> "Workspaces are not appropriate for system decomposition or deployments requiring separate credentials and access controls."

> "Use workspaces for environments that do not greatly deviate from one another, to avoid duplicating your configurations."

정리하면:

| 상황 | 권장 방식 |
|------|----------|
| 환경 간 인프라가 거의 동일, 파라미터만 다름 | Workspace |
| 빠른 프로토타이핑, 개인 테스트 환경 | Workspace |
| 환경별 인프라 구성이 다를 수 있음 | 디렉토리 |
| 환경별 다른 자격 증명/접근 제어 필요 | 디렉토리 |
| 프로덕션 운용, 팀 간 분리 필요 | 디렉토리 |
| CI/CD 파이프라인 환경별 독립 배포 | 디렉토리 |

#### Terraform Cloud workspace와의 관계

Terraform Cloud(HCP Terraform)의 "workspace"는 CLI workspace와 **다른 개념**이다. Terraform Cloud workspace는 디렉토리 분리에 가까운 개념으로, 각 workspace가 독립된:

- 코드 (VCS 연결 또는 API 업로드)
- 변수 (workspace-level variables)
- 자격 증명 (workspace-level credentials)
- State

를 가진다. CLI workspace는 "동일 코드 + state 분리"인 반면, Terraform Cloud workspace는 "완전 독립 단위"다.

공식 문서:

> "HCP Terraform workspaces behave more like completely separate working directories."

### 7. 디렉토리 기반 분리 — 운용 패턴

#### 초기화 및 적용 흐름

```bash
# dev 환경
$ cd envs/dev
$ terraform init
$ terraform plan
$ terraform apply

# prod 환경
$ cd envs/prod
$ terraform init
$ terraform plan
$ terraform apply
```

각 디렉토리에서 독립적으로 `init` → `plan` → `apply`를 실행한다. `init`은 해당 디렉토리에 `.terraform/`을 생성하고 backend를 설정한다.

#### destroy 순서

환경별 독립 destroy:

```bash
$ cd envs/dev
$ terraform destroy

$ cd envs/prod
$ terraform destroy
```

디렉토리가 분리되어 있으므로 다른 환경에 영향 없이 destroy할 수 있다.

#### `path.module` 주의사항

모듈 내에서 `path.module`을 사용하면 **모듈 소스 디렉토리**를 가리킨다. 환경 디렉토리에서 상대경로로 에셋(파일 등)을 참조할 때 경로가 달라질 수 있으므로 주의한다.

```hcl
# modules/instance/main.tf
resource "aws_instance" "web" {
  user_data = file("${path.module}/scripts/init.sh")
  # path.module = "../../modules/instance" (호출자 위치 무관)
}
```

`path.module`은 호출하는 환경 디렉토리가 아닌 모듈 자체의 위치를 반환한다. 환경별로 다른 스크립트가 필요하면 변수로 경로를 주입한다.

## 버전 요구사항

| 항목 | 요구 버전 | 비고 |
|------|----------|------|
| Terraform CLI | >= 1.14.0 | 디렉토리 분리 기능 자체는 초기 버전부터 존재. 특별한 버전 제약 없음 |
| hashicorp/aws provider | ~> 6.0 | 디렉토리 분리와 무관 |
| S3 backend | Terraform 내장 | 별도 provider 불요 |

TF 1.14.x에서 디렉토리 기반 분리와 관련된 breaking change나 신규 기능은 없다. 이 패턴은 Terraform의 기본 동작(디렉토리 = root module)을 활용하는 것이므로 버전 의존성이 낮다.

## 주의사항

- 각 환경 디렉토리에서 `terraform init`을 실행해야 한다. `.terraform/` 디렉토리는 root module별로 독립이다
- 공유 모듈의 `source` 상대경로가 올바른지 확인한다. `envs/dev/`에서 `../../modules/`는 프로젝트 루트의 `modules/`를 가리킨다
- Backend config 파일(`.hcl`, `.tfbackend`)에 자격 증명을 포함하면 `.terraform/`에 평문 저장된다. 자격 증명은 환경변수로 전달하는 것이 안전하다
- `terraform.tfvars`는 해당 디렉토리에서만 자동 로드된다. 상위 디렉토리의 `terraform.tfvars`는 자동 로드되지 않는다
- 환경별 모듈 호출 동기화 — 모듈 인터페이스 변경 시 모든 환경 디렉토리 업데이트 필요. CI `terraform validate`로 검증 권장
