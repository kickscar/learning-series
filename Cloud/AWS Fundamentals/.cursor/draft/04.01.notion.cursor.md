# 1. VPC(Virtual Private Cloud)

## 1. VPC가 의미하는 것

VPC(Virtual Private Cloud)는 AWS 클라우드 안에서 만드는 논리적으로 격리된 네트워크 공간이다. EC2, RDS 같은 리소스는 항상 "어딘가의 네트워크" 안에 배치되며, 그 네트워크의 기본 단위가 VPC다.

VPC는 "서버를 하나 만든다" 수준이 아니라, 배포 가능한 환경의 네트워크 경계를 정의한다.

### ① Region 단위의 네트워크 경계

VPC는 하나의 Region에 속한다. AZ는 Region 내부의 여러 데이터센터 묶음이고, Subnet은 AZ 단위로 배치된다. 따라서 VPC는 Region 내부에서 Multi-AZ 아키텍처를 만들기 위한 상위 컨테이너 역할을 한다.

---

# 2. CIDR 블록과 IP 주소 설계

## 1. VPC CIDR과 Subnet CIDR

VPC는 CIDR(Classless Inter-Domain Routing) 블록으로 IP 주소 공간을 정의한다. VPC CIDR은 "이 VPC 안에서 사용할 사설 IP 주소 범위"다.

예:

- VPC: `10.0.0.0/16`
- Subnet: `10.0.1.0/24`, `10.0.2.0/24`, ...

[이미지: 네트워크 구조 - VPC CIDR에서 Subnet CIDR로 분할 - /16을 /24 여러 개로 나누는 그림]

이 구조는 VPC 전체 주소 공간을 Subnet 단위로 분할해 "배치할 장소"를 만든다는 관점을 보여준다. Subnet CIDR은 서로 겹치면 안 된다.

---

# 3. Subnet

## 1. Subnet이 의미하는 것

Subnet은 VPC 안에서 리소스를 배치하는 구역이다. Subnet을 나누면 라우팅과 보안 경계를 구역별로 다르게 적용할 수 있다.

### ① Subnet은 AZ 단위다

Subnet은 하나의 AZ에 속한다. Multi-AZ 아키텍처를 만들려면 같은 역할의 Subnet을 AZ별로 만든다.

예:

- Public Subnet A (AZ-a), Public Subnet B (AZ-b)
- Private Subnet A (AZ-a), Private Subnet B (AZ-b)

### ② Public vs Private는 라우팅으로 결정된다

Public/Private는 이름이 아니라 라우팅과 IP 할당 조건으로 결정된다.

- Public Subnet 조건: `0.0.0.0/0 → IGW` 라우트 + Instance에 Public IP 가능
- Private Subnet 조건: 외부에서 직접 접근 경로 없음, 필요 시 NAT로 Outbound만 허용

이 Section에서는 Subnet "자리"와 Route Table 기본을 먼저 정리하고, 인터넷 경로(IGW + Public Route Table)는 다음 Section(04.02)에서 완성한다.

# 4. Route Table(기본)

## 1. Route Table은 "Destination + Target" 규칙이다

Route Table은 "어떤 목적지(Destination)로 가는 패킷을 어떤 Target으로 보낼 것인가"를 정의한다.

예:

- `10.0.0.0/16 -> local` (VPC 내부 통신)
- `0.0.0.0/0 -> igw-...` (인터넷 방향 통신)

[이미지: AWS Console - VPC - Route tables - Routes 탭 - Destination/Target/Status 확인 포인트]

여기서 중요한 점은 "인터넷이 된다/안 된다"를 추상적으로 보지 않고, "어디로 보낼지"를 규칙으로 선언한다는 것이다.

## 2. local route는 기본으로 존재한다

VPC를 생성하면 Route Table에는 VPC CIDR로 향하는 local route가 기본으로 들어간다. 이 라우트가 의미하는 것은 "같은 VPC CIDR 내부는 VPC 라우터가 처리한다"는 것이다.

즉, VPC 내부 통신은 기본적으로 가능하고, 인터넷 통신은 별도 라우트(IGW/NAT)가 필요하다.

## 3. Main Route Table과 Subnet association

VPC를 만들면 Main Route Table이 자동으로 생성된다. Subnet이 특정 Route Table과 명시적으로 association되지 않으면, 그 Subnet은 Main Route Table을 사용한다.

[이미지: AWS Console - VPC - Route tables - Main 표시 확인]
[이미지: AWS Console - VPC - Subnets - Route table 컬럼으로 연결 상태 확인]

실무에서는 Subnet 역할(Public/Private)을 "Subnet 이름"이 아니라 "Subnet이 어떤 Route Table을 쓰는가"로 분리한다. 이 관점을 기반으로 04.02에서 Public Subnet의 인터넷 경로(IGW + Public Route Table)를 완성한다.

# 5. Default VPC vs Custom VPC

## 1. Custom VPC를 써야 하는 이유

Default VPC는 빠르게 시작할 수 있지만, 실무 설계 관점에서는 제약이 많다. 반면 Custom VPC는 CIDR/Subnet/라우팅/보안 경계를 설계 의도대로 쌓을 수 있다.

Ch03에서 Default VPC로 EC2를 만들고 SSH로 접속할 수 있었던 이유는, Default VPC가 이미 "인터넷으로 나갈 수 있는 경로"를 갖추고 있기 때문이다.

이 Chapter의 목표는 "요청이 어디로 흘러가고, 어디에서 막히는가"를 설명할 수 있는 상태다. Custom VPC는 그 출발점이다.

---

# 핵심 정리

- VPC는 Region 단위의 논리적으로 격리된 네트워크 경계다.
- CIDR은 주소 공간이며, Subnet CIDR은 그 공간을 분할해 리소스를 배치할 구역을 만든다.
- Subnet은 AZ 단위 리소스이며, Multi-AZ를 위해 같은 역할의 Subnet을 AZ별로 쌍으로 만든다.
- Public/Private는 라우팅으로 결정되며, Route Table 관점에서 "Subnet 역할을 분리"할 수 있다.
- 인터넷 경로(IGW + Public Route Table)는 04.02에서 완성한다.
