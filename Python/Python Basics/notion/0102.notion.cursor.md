# 1.2 REPL vs 스크립트 vs m 옵션으로 모듈 실행하기

## 개요

파이썬 코드를 실행하는 방법은 크게 세 가지가 있다. 대화형 REPL(Read-Eval-Print Loop), 파일 경로로 스크립트를 직접 실행, 그리고 `python -m`으로 모듈을 실행하는 방식이다. 이 세 가지는 겉보기에는 비슷해 보이지만, 내부적으로 `sys.path`와 `__name__`을 다르게 설정한다. 이 차이를 모르면 "왜 import가 안 되지?", "왜 `__name__`이 `'__main__'`이 아니지?" 같은 문제에 부딪힌다.

특히 `-m` 옵션은 패키지 구조에서 모듈을 실행할 때 표준 방식이지만, `sys.path[0]`에 현재 작업 디렉터리(CWD)가 들어가면서 모듈 검색 경로가 달라진다. 이 차이를 이해하면 이후 모듈·패키지 시스템(Ch8)과 `__main__` 진입점(Ch1.5)에 대한 논의가 자연스럽게 이어진다.

---

## 핵심 개념

### 세 가지 실행 모드

#### 1) REPL (대화형 셸)

`python` 또는 `python3`를 인자 없이 실행하면 대화형 셸이 시작된다. 프롬프트(`>>>`)에서 한 줄씩 입력하면 즉시 실행되고 결과가 출력된다.

- **`sys.path[0]`**: 빈 문자열 `''`로 시작하지만, 실제로는 **현재 작업 디렉터리(CWD)**를 의미한다. `import` 시 CWD가 모듈 검색 경로의 첫 번째가 된다.
- **`__name__`**: `'__main__'`이다.
- **특징**: 한 줄씩 입력·실행·출력이 반복되는 루프. `exit()` 또는 `Ctrl+D`(Unix)/`Ctrl+Z`(Windows)로 종료.

#### 2) 스크립트 직접 실행

`python script.py`처럼 파일 경로를 인자로 주면, 그 파일이 스크립트로 실행된다.

- **`sys.path[0]`**: **스크립트 파일이 있는 디렉터리**의 절대 경로가 들어간다. 예를 들어 `/home/user/proj/script.py`를 실행하면 `sys.path[0]`는 `/home/user/proj`가 된다.
- **`__name__`**: `'__main__'`이다.
- **특징**: 스크립트 파일 전체가 한 번에 컴파일·실행된다. 스크립트 내에서 `import`를 하면, 그 모듈은 `sys.path`를 따라 검색된다.

#### 3) `-m` 옵션으로 모듈 실행

`python -m pkg.mod` 또는 `python -m pkg`처럼 `-m` 뒤에 모듈/패키지 이름을 주면, 그 모듈을 `__main__`으로 실행한다.

- **`sys.path[0]`**: **현재 작업 디렉터리(CWD)**의 절대 경로가 들어간다. 스크립트 실행과 달리, 실행된 모듈의 위치가 아니라 **어디서 명령을 실행했는가**가 중요하다.
- **`__name__`**: `'__main__'`이다. 다만 모듈이 `import`될 때는 원래 모듈 이름(예: `'pkg.mod'`)이 된다.
- **특징**: 모듈 검색 경로가 CWD 기준이 되어, 같은 프로젝트 내 다른 모듈을 상대적으로 쉽게 찾을 수 있다. 패키지 구조에서 CLI 진입점을 만드는 표준 방식이다.

### sys.path의 구성과 검색 순서

`sys.path`는 리스트이며, 파이썬이 모듈을 찾을 때 이 순서대로 검색한다.

1. **`sys.path[0]`**: 실행 모드에 따라 다르다(위 참고).
2. **`PYTHONPATH` 환경 변수**: 설정되어 있으면 그 경로들이 추가된다.
3. **표준 라이브러리 경로**: 파이썬 설치 디렉터리의 `lib/python3.x` 등.
4. **`site-packages`**: 서드파티 패키지가 설치되는 경로. 가상환경을 쓰면 venv의 `site-packages`가 들어간다.

`import foo`를 하면, `sys.path`의 각 디렉터리에서 `foo.py` 또는 `foo/__init__.py`를 찾는다. `sys.path[0]`가 다르면 같은 `import` 문이라도 다른 모듈을 찾을 수 있다.

### __name__의 의미

모듈에는 `__name__`이라는 내장 속성이 있다. 이 값은 **그 모듈이 어떻게 로드되었는가**에 따라 달라진다.

- **직접 실행될 때**: `__name__`은 `'__main__'`이다. REPL, 스크립트 실행, `-m` 실행 모두 여기에 해당한다.
- **import될 때**: `__name__`은 그 모듈의 **정규화된 이름**이다. 예를 들어 `import pkg.mod`로 로드되면 `pkg.mod`가 된다.

이 차이를 이용해 `if __name__ == "__main__":` 블록으로 "직접 실행될 때만 실행되는 코드"를 만들 수 있다. 라이브러리로 쓰일 때는 그 블록이 실행되지 않고, CLI로 실행될 때만 실행된다.

### -m 옵션의 동작 원리

`python -m pkg.mod`를 실행하면, 내부적으로는 다음과 같은 과정을 거친다.

1. `sys.path[0]`에 **현재 작업 디렉터리**를 넣는다.
2. `pkg.mod`를 **import**한다. 이때 `pkg.mod`의 `__name__`은 `'pkg.mod'`가 된다.
3. `pkg.mod`를 `__main__`으로 **재실행**한다. 이 과정에서 `runpy.run_module('pkg.mod', run_name='__main__')`가 호출된다고 보면 된다. 이때 `pkg.mod`의 `__name__`은 `'__main__'`으로 바뀐다.

만약 `pkg`가 패키지이고 `pkg/__main__.py`가 있으면, `python -m pkg`는 `pkg/__main__.py`를 `__main__`으로 실행한다. 이는 패키지에 CLI 진입점을 붙이는 표준 방식이다(Ch1.5에서 상세히 다룬다).

## 코드 예제

### 1) 세 가지 모드에서 sys.path[0] 확인

**REPL에서:**

```python
$ python
>>> import sys
>>> sys.path[0]
'/home/user/proj'  # 현재 작업 디렉터리
```

**스크립트 실행 (`/home/user/proj/script.py`):**

```python
# script.py
import sys
print(sys.path[0])
```

```bash
$ python script.py
/home/user/proj  # 스크립트가 있는 디렉터리
```

**-m 옵션 (`/home/user/proj/pkg/mod.py`):**

```python
# pkg/mod.py
import sys
print(sys.path[0])
```

```bash
$ cd /home/user/proj
$ python -m pkg.mod
/home/user/proj  # 현재 작업 디렉터리 (스크립트 실행과 같아 보이지만...)
```

하지만 다른 디렉터리에서 실행하면:

```bash
$ cd /tmp
$ python -m pkg.mod
/tmp  # 현재 작업 디렉터리! (스크립트 실행과 다름)
```

**아하 모먼트**: `-m` 실행 시 `sys.path[0]`는 **어디서 명령을 실행했는가**에 따라 달라진다. 스크립트 실행은 **스크립트 파일 위치**에 따라 달라진다.

### 2) __name__의 차이

**스크립트로 직접 실행 (`script.py`):**

```python
# script.py
print(f"__name__ = {__name__}")

if __name__ == "__main__":
    print("직접 실행됨")
```

```bash
$ python script.py
__name__ = __main__
직접 실행됨
```

**import될 때:**

```python
# other.py
import script  # script.py를 import
```

```bash
$ python other.py
__name__ = script  # import될 때는 모듈 이름
# "직접 실행됨"은 출력되지 않음
```

### 3) -m 옵션과 패키지 구조

**프로젝트 구조:**

```
proj/
├── pkg/
│   ├── __init__.py
│   ├── mod.py
│   └── __main__.py
└── main.py
```

**`pkg/mod.py`:**

```python
# pkg/mod.py
def func():
    return "mod.func()"

if __name__ == "__main__":
    print("mod.py가 직접 실행됨")
```

**`pkg/__main__.py`:**

```python
# pkg/__main__.py
from .mod import func

if __name__ == "__main__":
    print("패키지 pkg가 -m으로 실행됨")
    print(func())
```

**실행:**

```bash
$ python -m pkg.mod
mod.py가 직접 실행됨

$ python -m pkg
패키지 pkg가 -m으로 실행됨
mod.func()
```

**아하 모먼트**: `python -m pkg`는 `pkg/__main__.py`를 찾아 실행한다. 패키지에 CLI를 붙이는 표준 방식이다.

### 4) sys.path[0] 차이로 인한 import 문제 재현

**프로젝트 구조:**

```
proj/
├── utils.py
└── subdir/
    └── script.py
```

**`utils.py`:**

```python
# utils.py
def helper():
    return "helper"
```

**`subdir/script.py`:**

```python
# subdir/script.py
import utils  # utils.py를 import하려고 함
print(utils.helper())
```

**스크립트 실행 (실패):**

```bash
$ python subdir/script.py
ModuleNotFoundError: No module named 'utils'
```

`sys.path[0]`가 `subdir/`이 되어, `proj/utils.py`를 찾지 못한다.

**해결 방법 1: -m 옵션 사용**

```bash
$ cd proj
$ python -m subdir.script
helper
```

`sys.path[0]`가 `proj/`가 되어 `utils.py`를 찾을 수 있다.

**해결 방법 2: PYTHONPATH 설정**

```bash
$ PYTHONPATH=/path/to/proj python subdir/script.py
helper
```

**아하 모먼트**: 같은 `import utils`라도 `sys.path[0]`가 다르면 다른 모듈을 찾는다. `-m` 옵션은 CWD를 `sys.path[0]`에 넣어 프로젝트 루트 기준으로 import를 쉽게 만든다.

## 주의사항 (Pitfalls)

### 1) 스크립트 실행과 -m 실행의 sys.path[0] 차이

**스크립트 실행**: `sys.path[0]`는 **스크립트 파일이 있는 디렉터리**.  
**-m 실행**: `sys.path[0]`는 **현재 작업 디렉터리(CWD)**.

같은 파일이라도 실행 방식에 따라 `sys.path[0]`가 달라져, `import`가 실패하거나 다른 모듈을 찾을 수 있다. 특히 하위 디렉터리의 스크립트를 실행할 때 주의해야 한다.

### 2) C/Java와의 차이: "진입점" 개념이 다르다

C/Java는 `main()` 함수가 진입점이고, 실행 파일의 위치가 클래스패스에 자동으로 포함된다. 파이썬은:

- **진입점이 명시적이지 않다**: 파일 전체가 실행된다.
- **모듈 검색 경로가 실행 방식에 따라 달라진다**: `sys.path[0]`가 실행 모드에 따라 다르게 설정된다.

이 차이를 모르면 "왜 다른 언어에서는 되는데 파이썬에서는 안 되지?"라는 혼란이 생긴다.

### 3) -m 옵션 사용 시 CWD 의존성

`python -m pkg.mod`를 실행할 때, `sys.path[0]`에 CWD가 들어간다. 따라서 **어디서 명령을 실행했는가**가 중요하다. 같은 명령이라도 다른 디렉터리에서 실행하면 다른 모듈을 찾을 수 있다.

재현 가능한 실행을 위해서는:
- 프로젝트 루트에서 실행하거나
- `PYTHONPATH`를 명시적으로 설정하거나
- 상대 import를 올바르게 사용해야 한다(Ch8에서 다룬다).

### 4) REPL에서의 import 경로

REPL에서 `import foo`를 하면, `sys.path[0]`가 CWD이므로 CWD에 있는 `foo.py`를 찾는다. 스크립트 실행과 달리 "현재 파일"이라는 개념이 없어, CWD 기준으로만 검색한다.

### 5) __name__ == "__main__" 체크의 필요성

스크립트에 `if __name__ == "__main__":` 블록이 없으면, 그 스크립트를 `import`할 때도 전체 코드가 실행된다. 라이브러리로 쓰일 때는 원하지 않는 부작용(예: 즉시 실행되는 코드)이 생길 수 있다.

## Deep Insight

### runpy.run_module()의 역할

`python -m pkg.mod`는 내부적으로 `runpy.run_module('pkg.mod', run_name='__main__')`를 호출한다고 볼 수 있다. 이 함수는:

1. `pkg.mod`를 **import**한다. 이때 `pkg.mod`의 `__name__`은 `'pkg.mod'`가 된다.
2. 그 모듈의 코드 객체를 다시 `__main__`이라는 이름으로 실행한다. 이 과정에서 `pkg.mod`의 `__name__`이 `'__main__'`으로 바뀐다.

이렇게 하면 모듈이 "import된 것처럼" 로드되어(의존성 해결 등이 일어남) 동시에 "직접 실행된 것처럼" `__name__ == "__main__"` 블록이 실행된다.

### sys.path[0]의 특수성

`sys.path[0]`는 다른 경로들과 달리 **동적으로 설정**된다. 스크립트 실행 시에는 스크립트 디렉터리, `-m` 실행 시에는 CWD, REPL에서는 CWD가 들어간다. 이는 파이썬이 "어떤 컨텍스트에서 실행되는가"를 `sys.path[0]`에 반영하는 방식이다.

이 특수성 때문에, `sys.path`를 수정해 모듈 검색 경로를 바꿀 수 있지만, `sys.path[0]`는 실행 모드에 따라 덮어써질 수 있다는 점을 염두에 두어야 한다.

### 모듈 검색과 패키지 구조

`-m` 옵션은 패키지 구조에서 모듈을 실행할 때 특히 유용하다. 프로젝트 루트를 `sys.path[0]`에 넣어, 패키지 내 모듈들이 서로를 상대적으로 쉽게 import할 수 있게 한다. 이는 이후 Ch8(모듈과 패키지)에서 다루는 상대 import, `__init__.py`, 패키지 구조와 밀접하게 연관된다.
