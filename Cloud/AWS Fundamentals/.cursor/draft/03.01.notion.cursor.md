# 1. EC2(Elastic Compute Cloud)

## 1. EC2가 의미하는 것

EC2(Elastic Compute Cloud)는 AWS에서 가상 서버를 생성해 실행하는 서비스다. 핵심은 "서버 한 대"라는 결과물이 아니라, **필요할 때 생성하고 필요 없으면 제거할 수 있는 Compute 리소스**를 서비스로 소비한다는 점이다.

EC2를 이해할 때는 두 관점을 분리해서 본다.

- EC2 서비스: Instance를 만들고 실행/중지/종료하는 관리 영역
- EC2 Instance: 실제로 OS가 부팅되고 애플리케이션이 실행되는 단위

### ① EC2 Instance는 단독 리소스가 아니다

EC2 Instance는 보통 아래 리소스들과 묶여서 "실행 가능한 서버"가 된다.

- AMI(Amazon Machine Image): OS + 기본 소프트웨어 이미지
- Instance Type: CPU/Memory/Network 성능 프로파일
- Network Interface(ENI): VPC 내 네트워크에 붙는 인터페이스
- Security Group: Instance 수준 보안 경계(방화벽)
- EBS Volume: Root disk 및 추가 스토리지
- Key Pair: SSH 접속을 위한 공개키/개인키

---

# 2. Instance Type

## 1. Instance Type이 결정하는 것

Instance Type은 Instance의 하드웨어 성격을 정의한다. CPU/Memory/Network 성능이 묶여 있는 프로파일이라고 보면 된다.

### ① Family / Generation / Size

Instance Type 이름은 보통 다음 요소를 포함한다.

- Family: 범용(t, m), 컴퓨팅 최적화(c), 메모리 최적화(r) 등
- Generation: 세대(3, 5, 7 등)
- Size: micro, small, medium, large, xlarge 등

예:

- `t3.micro`: 범용(Burst) 성격의 3세대, micro 사이즈
- `m5.large`: 범용 성격의 5세대, large 사이즈

이 시리즈의 목표는 성능 튜닝이 아니라 인프라 연결 구조 이해이므로, 실습은 최소 사양을 기준으로 한다.

### ② Free Tier와 비용 감각

Free Tier는 "아무거나 공짜"가 아니라 조건이 있는 무료 사용량이다. Region/계정 상태/리소스 종류에 따라 달라질 수 있으므로, Console에서 "Free tier eligible" 표기와 예상 비용을 습관적으로 확인한다.

[이미지: AWS Console - EC2 - Launch instance - Instance type 선택 화면 - Free tier eligible 표기 확인 포인트]

# 3. AMI(Amazon Machine Image)

## 1. AMI가 의미하는 것

AMI(Amazon Machine Image)는 Instance가 부팅될 때 사용하는 이미지다. OS뿐 아니라 기본 패키지, 초기 설정이 포함될 수 있다. 같은 Instance Type이라도 AMI가 다르면 서버 내부 환경이 달라진다.

### ① AMI 선택 기준

- OS 계열: Amazon Linux, Ubuntu 등
- 패키지/런타임 포함 여부
- 업데이트/보안 지원 정책

이 시리즈는 Console 기반 학습을 위해 범용 AMI를 사용한다. 커스텀 AMI 빌드/자동화는 본 시리즈 범위를 벗어난다.

# 4. Key Pair

## 1. Key Pair가 의미하는 것

Key Pair는 SSH 접속을 위한 공개키/개인키 쌍이다. Instance 생성 시 Key Pair를 선택하면, Instance 내부에 해당 공개키가 등록된 상태로 부팅된다. 이후 로컬에 있는 개인키(`.pem`)가 없으면 SSH 접속이 불가능하다.

### ① Key Pair와 IAM Credential은 역할이 다르다

- IAM Credential: AWS 리소스 제어 권한(콘솔/API)
- Key Pair: Instance OS 레벨 로그인 권한(SSH)

즉 IAM으로 "Instance를 만들 수 있는 권한"이 있어도, Key Pair가 없으면 "그 Instance에 SSH로 들어갈 수 있는 권한"은 없다.

# 5. EC2가 인프라에서 위치하는 곳

## 1. 네트워크와 보안 경계 안에서 이해한다

EC2는 Compute이지만, 실무에서는 네트워크와 보안 경계 안에서 이해해야 한다. "서버가 실행된다"는 것은 결국 VPC 안에 네트워크 엔드포인트가 생겼다는 뜻이기 때문이다.

```mermaid
flowchart LR
  U[사용자] --> Internet[Internet]
  Internet --> IGW[Internet Gateway]
  IGW --> VPC[VPC]
  VPC --> Subnet[Subnet]
  Subnet --> ENI[ENI]
  ENI --> SG[Security Group]
  SG --> EC2[EC2 Instance]
  EC2 --> EBS[EBS Volume]
```

이 구조는 외부 트래픽이 EC2 Instance에 도달하기 위해 "VPC 경로(IGW/Subnet)"와 "보안 경계(Security Group)"를 동시에 통과해야 한다는 점을 보여준다. VPC/Subnet/Routing/SG/NACL의 상세는 Ch04에서 다룬다.

# 핵심 정리

- EC2는 가상 서버 서비스이며, Instance는 여러 리소스(AMI/Type/ENI/SG/EBS/Key Pair)의 결합 결과다.
- Instance Type은 비용과 성능을 동시에 결정하므로 Console에서 Free Tier/예상 비용을 확인한다.
- AMI는 부팅 이미지이며, Key Pair는 OS 레벨 SSH 접속 자격 증명이다(IAM Credential과 다름).
- EC2는 VPC/보안 경계 안에서 이해해야 하며, 네트워크 설계는 Ch04에서 확장한다.
