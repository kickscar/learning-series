# 1. RDS가 의미하는 것

## 1. RDS(Relational Database Service)는 Managed DB다

RDS는 AWS가 운영 책임의 일부를 대신 가져가는 관리형 관계형 데이터베이스 서비스다. 사용자는 DB 엔진과 용량, 네트워크 경계, 접근 제어를 설정하고, AWS는 백업/패치/장애 복구 같은 운영 작업을 서비스 형태로 제공한다.

### ① EC2에 직접 DB를 설치하는 방식과의 차이

EC2에 직접 MariaDB/MySQL을 설치하면 OS 수준까지 포함해 모든 운영 책임이 사용자에게 있다.

- 패치 적용, 보안 업데이트
- 백업/복원 자동화
- 장애 복구와 고가용성 구성

RDS는 이 운영 부담을 줄이지만, 그만큼 제약도 있다.

- OS 접근 불가(루트 권한 없음)
- 커스터마이징 범위 제한(엔진/파라미터 단위)
- 네트워크/보안은 AWS 모델에 맞춰야 한다

이 Chapter의 목적은 "DB를 만들었다"가 아니라, Private Subnet의 DB 계층을 구성하고 애플리케이션 계층(EC2)에서 안전하게 연결하는 구조를 완성하는 것이다.

# 2. RDS 엔진과 본 시리즈의 선택

## 1. MariaDB를 기준으로 진행한다

RDS는 MariaDB, MySQL, PostgreSQL 등 여러 엔진을 지원한다. 이 시리즈에서는 Gallery가 MariaDB JDBC를 포함하고 있고, 이후 실습 흐름이 MariaDB 기준으로 설계되어 있으므로 MariaDB를 사용한다.

본 시리즈 실습에서 선택하는 **엔진 버전 기준은 MariaDB 11.8.5**다. Region/시점에 따라 Console에 동일 번호가 없을 수 있으므로, 그 경우에는 **11.8.x 계열에서 가장 가까운 패치 버전**을 선택하고, 문서·스크린샷에는 실제로 선택한 버전을 남긴다.

Aurora는 AWS 전용 엔진이며 실무에서 자주 쓰지만, 본 시리즈에서는 "존재와 방향"만 알고 넘어간다.

# 3. RDS의 핵심 리소스

## 1. DB Instance

DB Instance는 "DB 서버"다. 인스턴스 클래스(db.t3.micro 등)와 스토리지(gp3 등)를 가진다. RDS는 인스턴스 단위로 백업, 파라미터, 유지보수 창 같은 운영 설정을 가진다.

## 2. Endpoint

애플리케이션은 RDS Endpoint로 접속한다. Endpoint는 다음 특성이 중요하다.

- 고정된 접속 문자열의 기준점이다
- Multi-AZ, 장애 복구 구성에서 Endpoint가 변경/전환될 수 있다

[이미지: AWS Console - RDS - Databases - Connectivity & security - Endpoint/Port 확인 포인트]

이 화면에서 Endpoint와 Port(3306)를 확인하고, 이후 JDBC URL을 구성한다.

## 3. Storage와 성능(개요)

스토리지는 단순 용량만이 아니라 성능과 비용을 결정한다. 이 시리즈에서는 기본 gp3로 시작하고, 성능 튜닝은 다루지 않는다.

# 4. 고가용성 옵션(개요)

## 1. Multi-AZ Deployment

Multi-AZ는 장애 시 자동 복구를 제공한다. 운영에서는 흔한 선택이지만, 실습에서는 비용과 복잡도 때문에 단일 AZ 또는 기본 설정으로 진행할 수 있다.

## 2. Read Replica

Read Replica는 읽기 부하 분산을 위한 구조다. 이 시리즈의 목표(기본 배포 가능한 인프라 구성) 범위를 넘어가므로 개념만 소개한다.

---

# 핵심 정리

- RDS는 관리형 관계형 데이터베이스 서비스이며, 운영 부담(백업/패치/복구)을 줄이는 대신 OS 수준 커스터마이징이 제한된다.
- 이 시리즈는 MariaDB를 기준으로 진행하며, 실습에서의 **엔진 버전 기준은 MariaDB 11.8.5**다(Console 가용 버전에 맞춰 11.8.x로 조정 가능).
- RDS 연결의 기준점은 Endpoint이며, 애플리케이션은 이 Endpoint로 접속한다.
- 고가용성(Multi-AZ, Read Replica)은 운영 확장 개념이며, 실습에서는 비용/범위를 고려해 최소 구성으로 시작한다.
