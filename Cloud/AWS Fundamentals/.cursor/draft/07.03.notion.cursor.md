# 1. EC2 -> RDS 연결 구조

## 1. 같은 VPC 내부의 Private 통신이다

RDS가 Private Subnet에 있어도, 같은 VPC 안의 EC2(Private 또는 Public)에서는 Private IP 경로로 접근할 수 있다. 중요한 것은 "인터넷을 타지 않는다"는 점이다.

### ① 연결을 허용하는 것은 SG다

RDS는 DB SG로 접근을 통제한다. 표준 패턴은 다음이다.

- DB SG inbound: 3306 from App SG
- App EC2 SG outbound: 3306 to DB SG(기본 outbound가 열려 있으면 별도 조치 없음)

[이미지: 네트워크 구조 - App EC2(SG) -> RDS(SG) 3306 - Source는 App SG]

이 구조에서 연결 실패가 나면 먼저 SG 참조가 올바른지 확인한다. NACL은 추가 계층이므로, 기본은 SG에서 원인이 나오는 경우가 많다.

---

# 2. Endpoint와 Connection String

## 1. Endpoint는 접속 주소다

RDS는 Endpoint와 Port를 제공한다.

[이미지: AWS Console - RDS - DB - Connectivity & security - Endpoint/Port 확인]

## 2. JDBC URL 구성(MariaDB)

Gallery 등 애플리케이션은 JDBC URL로 DB에 연결한다.

형태:

- `jdbc:mariadb://{endpoint}:3306/{db_name}`
- Gallery 실행 시에는 전송 암호화를 켜되 URL을 짧게 유지하기 위해 `?sslMode=trust` 정도만 붙이는 방식으로 둘 수 있다(인증서 경로를 URL에 넣지 않음).

여기에 username/password가 추가된다. DB 엔진은 `lab20`에서 만든 RDS와 동일하게 **MariaDB 11.8.5**(또는 선택한 11.8.x)를 전제로 한다. 이 Chapter에서는 "연결이 되는가"를 먼저 확인하고, 프로젝트 Lab에서 애플리케이션 설정으로 전환한다.

---

# 3. 연결 테스트의 목적

## 1. 네트워크/보안 문제를 앱 문제와 분리한다

애플리케이션을 바로 연결하면 "앱 설정 문제인지, 네트워크/보안 문제인지"가 섞인다. 따라서 먼저 EC2에서 DB 클라이언트로 접속해 다음을 확인한다.

- Endpoint가 DNS로 해석되는가
- 3306이 열려 있는가
- 인증 정보가 맞는가
- 기본 쿼리가 동작하는가

이 검증이 끝나면, 앱 설정으로 넘어가도 원인 범위가 좁아진다.

---

# 핵심 정리

- EC2 -> RDS 연결은 같은 VPC 내부의 Private 통신이며, 접근 허용은 SG가 결정한다.
- RDS Endpoint/Port는 연결 문자열의 기준점이며, 엔진은 **MariaDB 11.8.5** 기준으로 맞춘다.
- 먼저 DB 클라이언트로 연결 테스트를 수행해 네트워크/보안 문제를 앱 문제와 분리한다.

# [실습] lab21: EC2에서 RDS 연결

EC2에 DB 클라이언트를 설치하고, RDS용 CA 번들(`global-bundle.pem`)을 내려받은 뒤 `mysql` CLI로 TLS 연결(`--ssl`)을 시도한다. 데이터베이스/테이블을 만들고 간단한 쿼리를 수행해 연결이 정상인지 검증한다.

---

### 실습 목표

- RDS TLS 검증에 사용하는 CA 번들(`global-bundle.pem`)을 EC2에 내려받는다(Gallery Lab·Launch Template user data와 동일 출처).
- `mysql` CLI로 RDS Endpoint에 접속한다(`--ssl`). 사용자명은 `lab20`에서 지정한 Master username(**{db-admin}**)을 쓴다.
- 연결이 되지 않으면 SG/서브넷/NACL 관점에서 원인을 찾는다.
- 데이터베이스/테이블을 생성하고 기본 쿼리를 수행한다.

⚠️ 비용 주의: RDS는 시간당 비용이 발생한다. 이 실습은 기존 RDS를 전제로 하며, 학습 종료 시점에 프로젝트 전체 정리 기준을 적용한다.

# 3. 리소스 생성 및 설정 (생성 + 연결)

각 단계에서 AWS Console 화면 스냅샷을 반드시 명시한다.

## 1. DB SG inbound 확인(3306 from App SG)

설명: DB가 App 계층에서만 접근 가능하도록 SG가 구성되어 있어야 한다.

[이미지: AWS Console - EC2 - Security Groups - DB SG Inbound - 3306 Source=App SG 확인]

## 2. EC2에서 DB 클라이언트 설치

설명: MariaDB/MySQL 클라이언트로 접속을 시도한다.

[이미지: 터미널 - EC2 - 패키지 설치 로그]

예시:

```bash
sudo dnf install -y mariadb105 || sudo yum install -y mariadb
```

⚠️ 주의:

- 패키지 설치가 필요하면 Private Subnet의 경우 NAT Gateway가 필요할 수 있다.
- 설치가 어려우면 Public Subnet EC2(베스천)에서 먼저 연결 테스트를 수행하고, 이후 Private로 옮기는 방식도 가능하다.

## 3. RDS CA 번들 다운로드

설명: Gallery Lab의 user data와 **동일한 AWS PKI 번들**을 받아 둔다. 이후 `mysql` 에서 **서버 인증서를 CA로 검증**하려면 `--ssl-ca` 로 이 파일을 지정하면 된다. 아래 `mysql` 단계에서는 우선 **`--ssl` 만**으로 암호화 연결을 시도한다.

[이미지: 터미널 - curl - global-bundle.pem 저장 확인]

예시(`ec2-user` 홈에 저장):

```bash
curl -fsSL -o ~/global-bundle.pem https://truststore.pki.rds.amazonaws.com/global/global-bundle.pem
chmod 644 ~/global-bundle.pem
```

⚠️ 주의:

- Private Subnet EC2에서 `curl` 이 동작하려면 NAT Gateway 등 Outbound 경로가 있어야 한다.

## 4. RDS Endpoint로 접속(mysql CLI, TLS)

설명: Endpoint/계정 정보로 접속해 네트워크와 인증을 확인한다. **`--ssl`** 로 TLS 연결을 켠다(Gallery JDBC 의 `sslMode=trust` 와 같이, 실습에서는 명령을 짧게 유지하는 쪽을 우선한다).

[이미지: 터미널 - mysql --ssl 접속 - 성공 또는 에러 메시지]

예시:

```bash
mysql -h **{rds-endpoint}** -P 3306 -u **{db-admin}** -p --ssl
```

보조(선택): CA로 서버 인증서까지 검증하려면 다음처럼 `--ssl-ca` 를 추가한다. 명령이 길어지므로 필요할 때만 쓴다.

```bash
mysql -h **{rds-endpoint}** -P 3306 -u **{db-admin}** -p --ssl --ssl-ca=$HOME/global-bundle.pem
```

# [실습] Gallery: RDS(MariaDB 11.8.5) 연동

Gallery의 datasource를 RDS(MariaDB **11.8.5** 기준 인스턴스)로 전환한다. 운영형 흐름을 만들기 위해 Launch Template user data를 업데이트하고 ASG Instance Refresh로 롤링 교체해 변경을 반영한다. 요청(업로드/삭제)을 수행했을 때 DB 레코드가 생성/삭제되는지 확인하고 Phase 1(EC2 + S3 + RDS) 구성을 완성한다.

---

### 실습 목표

- Gallery의 DB 연결을 RDS(MariaDB 11.8.5)로 전환한다.
- RDS SG는 App SG에서만 접근되도록 유지한다.
- Launch Template(v3)로 datasource 파라미터를 반영하고 Instance Refresh로 적용한다.
- 업로드/삭제 동작이 DB 레코드 생성/삭제로 반영되는지 확인한다.
- S3 연동과 함께 전체 흐름이 동작하는지 확인한다.

⚠️ 비용 주의: 이 프로젝트 Lab은 RDS 인스턴스 비용과 S3 비용이 누적될 수 있다. 학습 종료 시점에 프로젝트 전체 정리 기준을 적용한다.

⚠️ 주의:

- 이 실습에서 다루는 "버전"은 Gallery 앱 버전이 아니라 Launch Template 버전이다.

⚠️ 시리즈 스코프(자격 증명·후속 학습):

- 실무에서는 DB 비밀번호 등을 **AWS Secrets Manager** 또는 **SSM Parameter Store**(SecureString) 등에 두고, 부팅 시 주입하거나 런타임에 조회하는 패턴이 일반적이다.
- 본 시리즈(AWS Fundamentals)에서는 **Console·user data·연결 동작** 이해에 집중하기 위해, Gallery **datasource를 JVM `ExecStart` 인자로 직접 넘기는 예시**를 사용한다. 이 방식은 **프로세스 목록 등에 노출될 수 있어 운영 환경에 그대로 적합하지 않다**는 점을 전제로 한다.
- **비밀 관리·코드/파이프라인과의 연동**은 후속 시리즈 **AWS IaC with CloudFormation** 등에서 다루는 것을 목표로 한다.

**프로덕션 프로필**의 구조화 로그(JSON, `logback-prod`)는 **CloudWatch Logs Insights**에서 필드 단위 검색·분석에 유리한 형태이나, 본 시리즈에서는 **인스턴스 내 파일 경로 확인** 위주로 둔다(에이전트 연동·대시보드는 후속 범위).

# 3. 리소스 생성 및 설정 (생성 + 연결)

각 단계에서 AWS Console 화면 스냅샷을 반드시 명시한다.

## 1. Launch Template v3 생성(user data 전체: S3 + RDS + datasource)

설명: RDS 전환을 "운영형"으로 반영한다. Ch06에서 만든 Launch Template **v2**와 같이 **user data 전체 스크립트**를 한 블록으로 넣되, **v3**에서는 v2 대비 **`global-bundle.pem` 다운로드 단계**와 **datasource용 짧은 JDBC URL·자격 증명 인자**가 추가된다.

[이미지: AWS Console - EC2 - Launch Templates - aws-fund-gallery-lt - Versions - Create new version - Source version=v2 선택]
[이미지: AWS Console - EC2 - Launch Templates - Create new version - User data 전체 붙여넣기]

다음은 **v3 기준 user data 전체 예시**다. `**{bucket-name}**`, `**{rds-endpoint}**`, `**{db-name}**`, `**{db-admin}**`, `**{db-password}**` 는 실습 환경 값으로 바꾼다.

- **RDS CA 번들**: `curl` 로 AWS PKI의 `global-bundle.pem` 을 `/opt/gallery/global-bundle.pem` 에 받아 둔다. **Gallery의 JDBC URL에는 넣지 않는다.** `lab21` 의 `mysql` CLI 등에서 `--ssl-ca` 로 쓰거나, 이후 운영에서 verify-ca 로 바꿀 때 참고할 파일로 남긴다.
- **JDBC URL**: 앱 파라미터는 짧게 `jdbc:mariadb://**{rds-endpoint}**:3306/**{db-name}**?sslMode=trust` 만 둔다. TLS 세션은 켜되, CA 파일 경로를 URL에 붙이지 않아 user data 가 길어지지 않게 한다.
- **systemd `ExecStart`**: 특수문자를 안전하게 넘기려고 **`/bin/sh -c 'exec /usr/bin/java ...'`** 형태로 감싼다.

```bash
#!/bin/bash
set -euo pipefail

APP_DIR=/opt/gallery
REPO_DIR=/home/ec2-user/workspace
JAR_PATH=/opt/gallery/gallery.jar
RDS_CA_PATH=/opt/gallery/global-bundle.pem

mkdir -p "${APP_DIR}"
chown -R ec2-user:ec2-user "${APP_DIR}"

sudo -u ec2-user bash -lc "
set -euo pipefail
cd /home/ec2-user
rm -rf workspace
git clone --filter=blob:none --sparse https://github.com/kickscar/learning-series.git workspace
cd workspace
git sparse-checkout init --no-cone
git sparse-checkout set Cloud/Workloads/gallery-spring-boot
cd Cloud/Workloads/gallery-spring-boot
./mvnw clean package -DskipTests -Dbuild.finalName=gallery
"

cp "${REPO_DIR}/Cloud/Workloads/gallery-spring-boot/target/gallery.jar" "${JAR_PATH}"
chown ec2-user:ec2-user "${JAR_PATH}"

curl -fsSL -o "${RDS_CA_PATH}" https://truststore.pki.rds.amazonaws.com/global/global-bundle.pem
chown ec2-user:ec2-user "${RDS_CA_PATH}"
chmod 644 "${RDS_CA_PATH}"

cat >/etc/systemd/system/gallery.service <<'UNITEOF'
[Unit]
Description=Gallery Spring Boot
After=network.target

[Service]
Type=simple
User=ec2-user
WorkingDirectory=/opt/gallery
ExecStart=/bin/sh -c 'exec /usr/bin/java -jar /opt/gallery/gallery.jar --server.port=8080 --spring.profiles.active=dev --app.storage.type=s3 --app.storage.s3.bucket=**{bucket-name}** --spring.datasource.url="jdbc:mariadb://**{rds-endpoint}**:3306/**{db-name}**?sslMode=trust" --spring.datasource.username=**{db-admin}** --spring.datasource.password=**{db-password}**'
Restart=always
RestartSec=5
SuccessExitStatus=143

[Install]
WantedBy=multi-user.target
UNITEOF

systemctl daemon-reload
systemctl enable --now gallery.service
```

⚠️ 주의:

- 비밀번호를 user data에 넣으면 노출될 수 있다. 실습에서는 흐름 이해를 위해 허용하되, 운영에서는 Secrets Manager/외부 설정으로 분리한다.
- `global-bundle.pem` 은 **인스턴스당 한 번 받아 두고 재사용**하면 된다. 위 스크립트는 부팅마다 `curl` 을 실행하므로 중복 다운로드가 되지만, 실습 단순화를 위해 동일 흐름을 유지한다. 운영에서는 이미지에 포함하거나 초기화 한정으로만 받는 방식을 검토한다.
- `sslMode=trust` 는 운영에서 권장되는 최종 형태가 아니라, **실습에서 URL을 짧게 유지하면서 암호화 연결을 켜기 위한 타협**으로 이해한다. CA 검증까지 하려면 `verify-ca` 와 인증서 경로 등을 추가로 설계한다.

## 2. Launch Template 기본 버전(Default version)을 v3로 설정

설명: 이후 생성되는 인스턴스가 v3를 기본으로 사용하도록 기준선을 올린다.

[이미지: AWS Console - EC2 - Launch Templates - Versions - Set default version - v3 선택]
