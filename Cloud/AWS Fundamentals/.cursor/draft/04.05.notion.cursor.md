# 1. Private Subnet이 의미하는 것

## 1. Public/Private는 "이름"이 아니라 Route의 차이다

Public/Private Subnet은 Subnet 이름으로 결정되지 않는다. 핵심은 Subnet이 사용하는 Route Table과, 그 Route Table의 `0.0.0.0/0` Target이다.

- Public: `0.0.0.0/0 -> IGW` + 인스턴스 Public IPv4 가능
- Private: 인터넷으로 향하는 직접 경로가 없다

즉, "Private은 격리"이며, 격리는 라우팅과 IP 할당 조건으로 만들어진다.

## 2. 실무 관점: Private에는 "의도적인 라우트 테이블"을 붙인다

Private Subnet을 만들 때, 실무에서는 다음을 자주 한다.

- Private Subnet 전용 Route Table을 만든다
- 초기에 local route만 가진 "비어 보이는" 상태로 둔다
- 필요해지면 NAT Gateway, VPC Endpoint 같은 Target을 점진적으로 추가한다

이 방식은 "기본은 격리, 필요할 때만 Outbound를 연다"는 운영 원칙과 잘 맞는다.

---

# 2. Private Subnet이 필요한 이유

## 1. 공격 표면을 줄인다

애플리케이션 서버와 DB를 Public Subnet에 두면, SG로 막아도 노출 면적이 커진다. 표준 패턴은 외부 엔드포인트는 ALB 같은 진입점으로 제한하고, 실제 서버는 Private Subnet으로 숨기는 것이다.

## 2. 계층 분리(Web/App/DB)를 강제한다

Private Subnet은 "어떤 계층을 인터넷에서 분리할 것인가"를 네트워크 경계로 강제한다. 이후 Chapter에서 ALB, RDS가 이 패턴 위에 자연스럽게 얹힌다.

---

# 3. VPC Endpoint

## 1. Private Subnet에서 "필요한 AWS 서비스"에 연결하는 방법

Private Subnet은 기본이 격리다. 그런데 운영에서는 Private에 있는 리소스가 "인터넷 전체"가 아니라, 특정 AWS 서비스에만 안전하게 연결해야 하는 경우가 많다.

이때 쓰는 도구가 VPC Endpoint다. VPC Endpoint는 "Private Subnet에서 AWS 서비스로 나가는 경로"를 VPC 내부로 끌어들이는 방식이다.

## 2. VPC Endpoint의 유형: Gateway vs Interface

- Gateway Endpoint: Route Table Target으로 붙는다(S3/DynamoDB)
- Interface Endpoint: Subnet에 ENI를 만들고, Security Group으로 443을 제어한다(PrivateLink 기반)

이 Section의 목표는 "Private에서 필요한 서비스에 닿는 방법이 NAT만은 아니다"를 이해하는 것이다.

## 3. Interface Endpoint의 핵심 포인트

- Endpoint는 Subnet(보통 AZ별) 안에 ENI 형태로 생성된다
- Endpoint에도 Security Group이 붙고, 일반적으로 443을 허용한다
- (권장) Private DNS를 활성화해, 서비스 도메인을 Private IP로 해석하게 만든다

⚠️ 주의:

- Interface Endpoint를 쓸 때 "EC2 SG inbound 443"을 여는 것이 아니다.
- 보통 흐름은 `EC2(SSM Agent) -> (outbound 443) -> Endpoint ENI` 이다.
  - EC2 SG: outbound 443 허용(기본 outbound allow면 OK)
  - Endpoint SG: inbound 443을 EC2 SG(source)에서 허용

[이미지: 네트워크 구조 - Private EC2 -> Interface VPC Endpoint(443) -> AWS Service - 인터넷 경유 없음]

## 4. 제한과 다음 Section 연결

VPC Endpoint는 "해당 서비스"에만 연결된다. 즉 다음은 해결하지 못한다.

- OS 패키지 설치/업데이트(외부 리포지토리)
- Git clone 같은 외부 네트워크 접근
- 외부 API 호출

이 문제를 풀기 위한 범용 해법이 NAT Gateway이며, 다음 Section(04.06)에서 완성한다.

---

# 핵심 정리

- Public/Private는 라우팅과 Public IPv4 조건으로 결정된다.
- Private Subnet은 기본 격리(local route만)로 시작하고, 필요 시 NAT/VPC Endpoint로 Outbound를 연다.
- VPC Endpoint는 Private Subnet에서 특정 AWS 서비스로 안전하게 연결하는 방법이다.
- VPC Endpoint가 해결하지 못하는 범용 Outbound은 NAT Gateway로 해결한다.

# [실습] lab15: Private Subnet 격리 확인과 VPC Endpoint(SSM 운영 접속)

Private Subnet에 EC2를 배치해 외부에서 직접 접속이 불가능함을 확인한다. 이후 Interface VPC Endpoint를 구성해, Private 환경에서 필요한 AWS 서비스에 연결하는 패턴을 만든다. 실습 시나리오는 SSM(Session Manager)을 사용한다.

⚠️ 주의:

- 이 Lab에서 SSM(Session Manager)은 "운영 접속" 자체가 핵심이라기보다, Interface VPC Endpoint를 통해 Private에서 AWS 서비스로 안전하게 연결하는 패턴을 보여주는 시나리오다.

---

### 실습 목표

- Private Subnet 전용 Route Table을 구성(local route만)해 격리를 만든다.
- Private Subnet EC2는 Public IPv4 없이 외부에서 직접 접근이 안 되는 것을 확인한다.
- EC2용 IAM Role(SSM)을 생성하고 Instance Profile로 적용한다.
- VPC Endpoint로 SSM 경로를 구성하고, Console에서 Session Manager로 접속한다.

⚠️ 비용 주의: 이 Lab에서 사용하는 Interface VPC Endpoint는 시간당 비용이 발생할 수 있다. 실습 종료 시 Endpoint를 유지할지/삭제할지 기준을 명확히 한다.

---

# 1. 전체 아키텍처

```mermaid
flowchart LR
  User[내 PC] --> AWSConsole[AWS Console]
  AWSConsole --> SSM[SSM Session Manager]
  SSM --> VPCE[VPC Endpoint\n(ssm/ssmmessages/ec2messages)]
  VPCE --> PrivateSubnet[Private Subnet]
  PrivateSubnet --> PrivateEC2[Private EC2]
```

이 실습은 "Private EC2는 직접 접근 불가"라는 격리를 만들고, "운영 접속은 SSM(Session Manager)로"라는 경로를 구성한다.

---

# 2. 사전 준비

- 리전: `ap-northeast-2 (Seoul)`
- `lab12` 완료(IGW/Route Table로 Public Subnet 경로 구성)
- Private Subnet 2개가 존재해야 한다(`lab11`)
- (권장) AMI는 AWS 제공 기본 AMI(Amazon Linux 등) 사용(SSM Agent 기본 포함)

SSM(Session Manager) 접속이 되려면 다음이 필요하다.

- Instance에 SSM Agent가 설치/실행 중이어야 한다(대부분의 AWS 제공 AMI는 기본 포함)
- Instance에 IAM Role(Instance Profile)이 부여되어 있어야 한다
  - 최소 권한 시작점: `AmazonSSMManagedInstanceCore`
- Private Subnet에서도 SSM으로 연결될 네트워크 경로가 있어야 한다
  - 이 Lab은 Interface VPC Endpoint로 경로를 만든다

# 3. 리소스 생성 및 설정 (생성 + 연결)

각 단계에서 AWS Console 화면 스냅샷을 반드시 명시한다.

## 1. Private Route Table 생성(local route만)

설명: Private Subnet을 기본 격리 상태로 시작하기 위해, 인터넷 라우트가 없는 Route Table을 만든다.

[이미지: AWS Console - VPC - Route tables - Create route table - VPC 선택/Name 입력]

## 2. Private Subnet association

설명: Private Subnet이 이 Route Table을 사용하도록 association한다.

[이미지: AWS Console - VPC - Route tables - Subnet associations - Private Subnet 선택]

## 3. Private EC2 SG 생성(운영 접속은 SSM, Inbound 최소화)

설명: SSM은 인바운드 포트(22)를 열지 않아도 된다. 이 Lab에서는 Inbound는 최소화한다.

[이미지: AWS Console - EC2 - Security Groups - Private EC2 SG - Inbound empty 또는 최소 규칙 확인]

⚠️ 주의:

- Session Manager를 쓰는 경우 EC2 SG에 SSH(22) inbound를 열지 않는다.
- EC2 SG inbound에 443을 추가하지 않는다(Endpoint SG에서 443을 제어한다).

## 4. Private EC2 생성(Private Subnet, Public IPv4 없음)

설명: Private Subnet에 EC2를 생성하고 Public IPv4가 없음을 확인한다.

[이미지: AWS Console - EC2 - Launch instance - Private Subnet/Auto-assign public IP Disabled 확인]

## 5. IAM Role 생성(SSM) 및 Instance Profile 적용

설명: EC2가 Systems Manager에 등록되고 Session Manager 접속이 가능하도록 IAM Role을 부여한다.

[이미지: AWS Console - IAM - Roles - Create role - Use case EC2 선택]
[이미지: AWS Console - IAM - Add permissions - AmazonSSMManagedInstanceCore 선택]
[이미지: AWS Console - EC2 - Instance - Modify IAM role - Role 연결 화면]

## 6. VPC Endpoint 생성(SSM)

설명: NAT 없이도 Private Subnet에서 SSM으로 연결되도록 Interface VPC Endpoint를 만든다.

[이미지: AWS Console - VPC - Endpoints - Create endpoint - Interface 선택]
[이미지: AWS Console - VPC Endpoint - Service name ssm/ssmmessages/ec2messages 선택]
[이미지: AWS Console - VPC Endpoint - Subnets에 Private Subnet 선택]
[이미지: AWS Console - VPC Endpoint - Enable Private DNS name 체크]
[이미지: AWS Console - EC2 - Security Groups - Endpoint SG 생성 - Inbound 443 Source=Private EC2 SG]
[이미지: AWS Console - VPC Endpoint - Security group - Endpoint SG 연결]

Endpoint SG 규칙(예시):

- Inbound: HTTPS(443) Source = **{private-ec2-sg}**
- Outbound: 기본값 유지(또는 필요 범위)

# 4. 실행 및 결과 검증

설명: Private 격리는 유지되면서도(Session Manager로) 운영 접속이 가능해야 한다.

## 1. Private EC2 격리 확인

[이미지: AWS Console - EC2 - Instance summary - Public IPv4 없음 확인]
[이미지: AWS Console - VPC - Route tables - Private Route Table에 0.0.0.0/0 라우트 없음 확인]

## 2. Session Manager 접속 확인(콘솔)

[이미지: AWS Console - EC2 - Connect - Session Manager 탭 - Connected 확인]
[이미지: AWS Console - Systems Manager - Session Manager - Start session 화면]

예시(세션에서 실행):

```bash
whoami
hostname
```

## 3. (선택) 인터넷 Outbound이 아직 안 되는지 확인

[이미지: Session Manager - curl 실패 확인]

예시:

```bash
curl -I https://aws.amazon.com
```

이 요청이 실패하면 "SSM 운영 접속은 되지만, 인터넷 Outbound은 닫힌 상태"라는 격리 목표를 달성한 것이다. Outbound은 다음 Lab(NAT Gateway)에서 연다.
