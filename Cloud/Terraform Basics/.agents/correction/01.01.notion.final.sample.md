# **개요**

본 Section에서는 Terraform 학습의 출발점이 되는 기초 이론을 설명한다. DevOps의 개념에서 출발하여, 자동화의 핵심 요소인 Infrastructure as Code(IaC), 그리고 이를 구현하는 대표 도구인 Terraform 까지의 흐름을 단계적으로 살펴보고 이해하도록 한다.

# **DevOps**

DevOps는 개발(Development)과 운영(Operations)을 통합하여 소프트웨어 전달 속도와 품질을 동시에 향상시키기 위한 접근 방식이다. 단순한 도구의 조합이 아니라, 조직 문화와 프로세스 전반을 개선하는 실천적 철학 또는 기술 체계에 가깝다. DevOps는 일반적으로 다음 네 가지 핵심 가치(CLAMS)를 중심으로 이해할 수 있다.

| 핵심 가치 | 설명 |
| --- | --- |
| 문화(Culture) | 개발팀과 운영팀 간 장벽을 허물고, 협업과 신뢰를 바탕으로 한 조직 문화를 만든다. |
| 린 (Lean) | 작은 단위(Small Batch)로 작업하여 복잡성을 줄이고, 프로세스상의 낭비를 제거해 가치 전달을 최적화한다. |
| 자동화(Automation) | 반복 작업을 도구로 대체하여 속도를 높이고, 휴먼 에러를 줄인다. |
| 측정(Measurement) | 지표를 수집·분석하여 개선점을 찾고, 의사결정을 데이터에 기반한다. |
| 공유(Sharing) | 지식, 도구, 프로세스를 팀 내외로 공유하여 학습과 개선을 가속한다. |

# 자동화와 배포

자동화는 DevOps를 구성하는 핵심 축 중 하나이다. 그 중, 소프트웨어 배포 자동화(Deployment Automation)는 코드를 빌드(Build)하고, 테스트(Test)하고, 인프라에 배포(Deploy)하는 전 과정을 자동으로 수행하는 것을 의미한다. 그러나 배포 자동화가 안정적으로 동작하기 위해서는 애플리케이션 코드뿐 아니라, 해당 코드가 실행되는 인프라 환경 역시 일관되고 반복 가능한 방식으로 준비되어야 한다. 실행 환경이 매번 달라진다면 자동화의 효과는 크게 감소하기 때문이다.

> 이 지점에서 등장하는 개념이 바로 I**nfrastructure as Code(IaC)** 이다.
> 

IaC는 인프라를 코드로 정의하고, 그 코드를 실행하여 인프라를 생성·변경·삭제하는 방식이다. 콘솔에서 수동으로 리소스를 생성하거나, 일회성 스크립트를 실행하는 대신, 인프라를 선언적으로 정의하고 자동화된 방식으로 관리한다. 이를 통해 인프라 역시 애플리케이션 코드와 동일하게 버전 관리, 재현성, 자동화의 대상이 된다.

# **IaC(Infrastructure as Code)**

Infrastructure as Code(IaC)는 서버, 네트워크, 스토리지와 같은 인프라 자원을 코드로 정의하고, 해당 코드를 실행하여 인프라를 생성·변경·삭제하는 방식을 의미한다. 인프라 구성을 사람이 직접 콘솔에서 수행하는 대신, 코드로 명세하고 자동화된 방식으로 관리한다는 점이 핵심이다. IaC를 도입하면 다음과 같은 이점을 얻을 수 있다.

### **① 일관성과 재현성**

동일한 코드를 실행하면 동일한 인프라가 반복적으로 생성된다. 이로 인해 개발(Development), 스테이징(Staging), 프로덕션(Production) 환경 간 구성 차이에서 발생하는 문제를 최소화할 수 있다.

### **② 버전 관리와 협업**

인프라 정의가 코드 형태이므로 Git과 같은 버전 관리 시스템에 저장할 수 있다. 이를 통해 변경 이력 추적, 코드 리뷰(Code Review), 롤백(Rollback) 등이 가능해지며, 팀 단위 협업이 용이해진다.

### **③ 문서화와 자동화**

코드 자체가 인프라의 명세서(Specification) 역할을 수행한다. 별도의 수동 문서 작성 부담이 줄어들며, CI/CD 파이프라인과 연계하여 인프라 변경을 자동화할 수 있다.

### **④ 속도와 안정성**

수동 작업 대비 프로비저닝(Provisioning) 시간이 단축되며, 반복 작업에서 발생하는 휴먼 에러(Human Error)를 줄일 수 있다. 특히 대규모 환경에서는 자동화의 효과가 더욱 크게 나타난다.

## IaC 도구의 유형

IaC를 구현하기 위한 도구와 방식은 목적과 적용 시점에 따라 다음과 같이 구분할 수 있다.

| 범주 | 설명 | 대표 도구/예시 |
| --- | --- | --- |
| 애드혹 스크립트 | 특정 작업을 위해 작성된 일회성 스크립트. 재사용성과 유지보수성이 낮다. | Bash, PowerShell, AWS CLI 스크립트 |
| 구성 관리 도구 | 이미 존재하는 서버의 설정(패키지, 서비스, 파일 등)을 코드로 관리한다. 서버 내부 구성을 다룬다. | Ansible, Chef, Puppet |
| 서버 템플릿 도구 | OS, 패키지, 설정이 포함된 이미지(템플릿)를 미리 만들어 두고, 그 이미지로 서버를 띄운다. | Packer, AMI, Docker 이미지 |
| 오케스트레이션 도구 | 컨테이너나 서비스를 배치·스케일링·로드밸런싱하는 등 실행 환경을 관리한다. | Kubernetes, Docker Swarm, ECS |
| 프로비전 도구 | 클라우드 API를 호출하여 서버, 네트워크, 스토리지 등 인프라 자원을 생성·변경·삭제한다. | Terraform, CloudFormation, Pulumi |

## IaC의 접근 방식 : Declarative vs Imperative

IaC는 코드 작성 방식과 인프라 관리 철학에 따라 선언적(Declarative) 방식과 명령형(Imperative) 방식으로 구분할 수 있다.

| 구분 | 설명 | 대표 도구 |
| --- | --- | --- |
| 선언적(Declarative) | "원하는 최종 상태"를 기술하고, 도구가 현재 상태와 비교하여 필요한 변경만 수행한다. | Terraform, CloudFormation, Pulumi |
| 명령형(Imperative) | "어떤 순서로 무엇을 실행할지" 단계별로 기술한다. | Ansible, Chef, Puppet |

# **Terraform**

Terraform은 선언적(Declarative) 방식의 프로비전(Provisioning) 도구로, HashiCorp에서 개발한 오픈 소스 Infrastructure as Code(IaC) 도구이다. 사용자는 인프라의 **목표 상태(Desired State)**를 **HCL(HashiCorp Configuration Language)**로 정의하고, Terraform은 이를 기반으로 실제 인프라를 생성·변경·삭제한다.

Terraform은 AWS, Azure, GCP와 같은 주요 클라우드뿐 아니라 온프레미스 환경까지 지원하는 멀티 클라우드(Multi-Cloud) 특성을 가진다. 하나의 코드베이스로 다양한 환경을 일관된 방식으로 관리할 수 있다는 점이 특징이다. Terraform은 사용자가 작성한 설정 파일을 읽고, 그 내용을 바탕으로 클라우드 API를 호출한다. 코드에 정의된 “원하는 상태”와 현재 인프라 상태를 비교한 후, 필요한 변경 사항만 선별적으로 적용한다.

## **Terraform 전체 동작 흐름**

Terraform의 실행 과정은 다음과 같이 구성된다.

1. 사용자가 `.tf` 파일에 원하는 리소스(VPC, EC2, S3 등)를 선언한다.
2. `terraform init`을 실행하여 Provider(AWS 등)를 다운로드하고 초기화한다.
3. `terraform plan`을 통해 코드에 정의된 상태와 현재 상태(State)를 비교한 변경 계획을 확인한다.
4. `terraform apply`를 실행하여 변경 계획을 실제 클라우드에 반영한다.
5. 적용 결과는 State 파일에 저장되며, 이후 실행 시 현재 상태로 사용된다.

Terraform은 코드(Configuration)와 State를 기준으로 변경 사항을 계산하고, 클라우드(AWS) API를 통해 실제 인프라를 제어한다.

## **Terraform의 핵심 구성 요소**

| 구성 요소 | 설명 |
| --- | --- |
| Configuration | HCL로 작성된 `.tf` 파일. 생성·변경할 리소스를 선언한다. 변수(Variables)와 출력(Output)도 포함된다. |
| Provider | AWS, Azure 등 대상 플랫폼과 통신하는 플러그인. `terraform init` 시 다운로드된다. |
| State | Terraform이 관리하는 리소스의 현재 상태를 기록한 파일. Plan과 Apply의 기준이 된다. |
| Plan | 코드와 State를 비교하여 변경 예정 사항을 미리 계산한 결과. 실제 변경은 수행하지 않는다. |
| Apply | Plan에서 계산된 변경 사항을 실제 클라우드 환경에 적용한다. |

## **동작 예시: S3 버킷 생성**

`.tf` 파일에 “S3 버킷 하나를 생성하라”고 정의한 경우, Terraform은 다음과 같은 절차로 동작한다.

1. **State 확인**: 동일한 리소스가 State에 존재하는지 확인한다.
2. **비교 분석**: 코드에 정의된 설정과 현재 상태(State 또는 실제 클라우드 상태)를 비교한다.
3. **계획 수립**: 리소스가 없다면 “생성(Create)”, 설정이 다르면 “수정(Update)”, 필요 없으면 “삭제(Delete)” 계획을 생성한다.
4. **적용**: `terraform apply` 실행 시 클라우드 API를 호출하여 실제 리소스를 생성·수정·삭제한다.
5. **State 갱신**: 변경 결과를 State 파일에 반영한다.

이 과정에서 Terraform은 항상 **코드(목표 상태)** 와 **State(현재 상태)** 를 비교하여 필요한 작업만 수행한다. 따라서 콘솔에서 수동으로 리소스를 생성하거나, 실행 순서를 맞춰 스크립트를 작성할 필요가 없다.

## **Terraform의 주요 특징**

### **① 멀티 클라우드 지원**

단일 코드베이스로 AWS, Azure, GCP 등 여러 클라우드를 동시에 다룰 수 있다. 하이브리드·멀티 클라우드 환경에서 일관된 방식으로 인프라를 관리할 수 있다.

### **② 선언적 문법**

"어떻게"가 아니라 "무엇을" 원하는지 기술한다. Terraform이 의존 관계를 분석하고 실행 순서를 결정한다.

### **③ 모듈과 재사용**

인프라 코드를 모듈로 캡슐화하여 재사용할 수 있다. Terraform Registry를 통해 공개·비공개 모듈을 공유하고 버전 관리할 수 있다.

### **④ State 기반 변경 관리**

State 파일을 통해 실제 인프라와 Terraform이 인식하는 상태를 동기화한다. 이를 바탕으로 plan 단계에서 정확한 변경 사항을 예측할 수 있다.
