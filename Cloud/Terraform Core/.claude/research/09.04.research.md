## 상세 조사

### 1. terraform test command

#### 도입 이력

- v1.6.0 (2023-10): **GA** (Generally Available). 실험적 단계에서 정식 전환. 파일 포맷과 실행 모델이 실험 버전에서 크게 변경됨.
- 실험적 단계(v1.5 이전)의 test 명령과는 호환되지 않음.

#### 기본 문법

```bash
terraform test [options]
```

#### CLI 플래그

| 플래그 | 설명 |
|--------|------|
| `-filter=testfile` | 특정 테스트 파일만 실행. 경로 포함 가능 (예: `tests/validations.tftest.hcl`) |
| `-verbose` | 각 run 블록의 plan/state 출력을 표시 |
| `-json` | 머신 리더블 JSON 포맷 출력 |
| `-junit-xml=<path>` | JUnit XML 포맷 리포트 저장 (v1.11.0 GA). `-cloud-run`과 비호환 |
| `-test-directory=<dir>` | 테스트 디렉토리 변경 (기본값: `tests`) |
| `-parallelism=<n>` | plan/apply 병렬 작업 수 (기본값: 10, v1.12.0 도입) |
| `-cloud-run=<module source>` | HCP Terraform에서 원격 실행 (private registry 모듈 소스만) |

> **참고:** `-run` 플래그는 공식 문서에서 확인되지 않음. `-filter`가 파일 단위 필터링을 담당한다.

#### terraform validate와의 차이

| 비교 항목 | `terraform validate` | `terraform test` |
|-----------|---------------------|-----------------|
| 검증 범위 | 구문(syntax) + 내부 일관성(consistency) | plan/apply 실행 후 결과 검증 |
| Provider API 호출 | 하지 않음 | apply 시 실제 호출 (mock 가능) |
| State 참조 | 하지 않음 | 테스트 전용 in-memory state 사용 |
| Variable 값 | 무시 | 테스트 파일에서 주입 |
| 인프라 생성 | 하지 않음 | `command = apply` 시 실제 생성 |
| 실행 안전성 | 완전 안전 (읽기 전용) | apply 시 실제 리소스 생성/삭제 |
| 용도 | CI에서 빠른 syntax check, 에디터 저장 후 자동 검증 | 모듈 단위/통합 테스트 |

### 2. .tftest.hcl 파일 포맷

#### 파일 위치 규칙

- **기본 테스트 디렉토리:** `tests/` (설정 루트 하위)
- **설정 루트 디렉토리:** `.tftest.hcl` 파일을 설정 파일과 같은 디렉토리에 배치 가능
- Terraform은 **항상** 설정 루트의 테스트 파일을 로드하고, 추가로 `-test-directory`로 지정된 디렉토리도 검색
- 테스트 디렉토리는 설정 루트 하위에 있어야 함 (중첩 가능)
- 파일 확장자: `.tftest.hcl` 또는 `.tftest.json`

#### 파일 루트 레벨 구조

```hcl
# (선택) test 블록 — 실행 동작 제어
test {
  parallel = true
}

# (선택) 파일 레벨 variables — 모든 run 블록에 적용
variables {
  instance_type = "t3.micro"
  region        = "ap-northeast-2"
}

# (선택) provider 블록 — provider 설정 override
provider "aws" {
  region = "us-west-2"
}

# (선택) mock_provider 블록 — provider mocking
mock_provider "aws" {}

# (선택) override_resource / override_data / override_module
override_resource {
  target = aws_instance.web
  values = { arn = "arn:aws:ec2:::i-mock" }
}

# (필수) 하나 이상의 run 블록
run "test_name" {
  command = plan
  # ...
}
```

#### run 블록 구조

| 속성/블록 | 타입 | 기본값 | 설명 |
|-----------|------|--------|------|
| `command` | string | `apply` | `plan` 또는 `apply` |
| `variables` | block | — | run 레벨 variable override |
| `assert` | block (복수 가능) | — | 검증 조건 |
| `expect_failures` | list | — | 예상 실패 대상 목록 |
| `module` | block | — | 대체 모듈 지정 (source, version) |
| `providers` | attribute | — | provider 매핑 |
| `plan_options.mode` | string | `normal` | `normal` 또는 `refresh-only` |
| `plan_options.refresh` | boolean | `true` | refresh 수행 여부 |
| `plan_options.replace` | list | — | 교체할 리소스 주소 |
| `plan_options.target` | list | — | 대상 리소스 주소 |
| `state_key` | string | — | 커스텀 state 파일 식별자 (v1.11.0+) |
| `parallel` | boolean | `false` | 병렬 실행 허용 |

#### expect_failures — 네거티브 테스팅

validation rule, check 블록, precondition/postcondition의 실패를 의도적으로 검증한다.

```hcl
run "invalid_instance_type" {
  command = plan

  variables {
    instance_type = "invalid-type"
  }

  expect_failures = [
    var.instance_type,    # variable validation 실패 예상
  ]
}
```

지원하는 checkable object:
- `var.<name>` — variable validation 블록
- `output.<name>` — output precondition
- `<resource_address>` — resource precondition/postcondition
- `data.<resource_address>` — data source precondition/postcondition
- `check.<name>` — check 블록 (복수 가능)

> `check` 블록만 `expect_failures`에 여러 checkable object를 허용한다.

#### provider 블록 override

테스트 파일에서 provider를 재정의할 수 있다. v1.7.0+에서는 변수와 run 블록 출력도 참조 가능하다.

```hcl
provider "aws" {
  region = "us-west-2"
}

provider "aws" {
  alias  = "secondary"
  region = "eu-central-1"
}
```

run 블록에서 provider 매핑:

```hcl
run "with_specific_provider" {
  providers = {
    aws = aws.secondary
  }
}
```

> **v1.9.0 주의:** 테스트 파일의 provider 블록에서 version constraint를 지정할 수 없다. 메인 설정의 `required_providers`에서 정의해야 한다.

### 3. assert 블록

```hcl
assert {
  condition     = aws_instance.web.instance_type == "t3.micro"
  error_message = "Expected instance type t3.micro, got ${aws_instance.web.instance_type}"
}
```

#### condition에서 참조 가능한 대상

- **Resource 속성:** `aws_instance.web.id`, `aws_s3_bucket.this.arn`
- **Data source 속성:** `data.aws_ami.latest.id`
- **Output 값:** `output.instance_id`
- **이전 run 블록 출력:** run 블록 간 상태가 공유되므로 이전 run에서 생성된 리소스 참조 가능
- **Named values:** 설정에서 사용 가능한 모든 named value

#### error_message

- 문자열 보간(interpolation) 지원: `${expression}`
- 실패 시 실제 값을 포함하면 디버깅에 유용

#### 복수 assert

하나의 run 블록에 여러 assert 블록 가능:

```hcl
run "verify_instance" {
  command = apply

  assert {
    condition     = aws_instance.web.instance_type == "t3.micro"
    error_message = "Wrong instance type"
  }

  assert {
    condition     = aws_instance.web.tags["Name"] == "tf-core-lab01-instance-web"
    error_message = "Wrong Name tag"
  }

  assert {
    condition     = output.public_ip != ""
    error_message = "Public IP should not be empty"
  }
}
```

### 4. mock_provider (v1.7.0+)

credential 없이, 실제 인프라 생성 없이 테스트할 수 있는 메커니즘.

#### 기본 문법

```hcl
mock_provider "aws" {}
```

이것만으로 AWS provider의 모든 resource와 data source가 mock된다. 실제 API 호출 없음, credential 불필요.

#### 주요 속성

| 속성 | 설명 |
|------|------|
| `alias` | 여러 mock provider 인스턴스 구분 |
| `override_during` | 데이터 생성 시점: `plan` 또는 `apply` (기본값: `apply`) |
| `source` | 외부 mock 정의 파일 경로 (`.tfmock.hcl` / `.tfmock.json`) |

#### mock_resource / mock_data 중첩 블록

mock_provider 내부에서 특정 리소스/데이터 소스의 computed 속성 기본값을 지정한다.

```hcl
mock_provider "aws" {
  mock_resource "aws_instance" {
    defaults = {
      id               = "i-mock12345678"
      private_ip       = "10.0.1.100"
      public_ip        = "1.2.3.4"
      availability_zone = "ap-northeast-2a"
    }
  }

  mock_data "aws_ami" {
    defaults = {
      id           = "ami-mock12345678"
      architecture = "x86_64"
    }
  }
}
```

#### 자동 생성되는 기본값 (defaults 미지정 시)

| 타입 | 기본값 |
|------|--------|
| number | `0` |
| boolean | `false` |
| string | 랜덤 8자 영숫자 |
| list/set/map | 빈 컬렉션 |
| object | 필수 하위 속성을 재귀적으로 생성 |

#### credential 불필요

mock_provider는 실제 API를 호출하지 않으므로 AWS credential이 필요 없다. CI 환경에서 unit test를 credential 없이 실행할 수 있다.

#### mock vs real provider 사용 기준

| 구분 | mock_provider | real provider |
|------|--------------|---------------|
| 용도 | Unit test — 로직 검증 | Integration test — 실제 동작 검증 |
| 인프라 생성 | 없음 | 있음 (apply 시) |
| credential | 불필요 | 필요 |
| 속도 | 빠름 | 느림 (API 호출) |
| 검증 범위 | HCL 로직, 변수 전달, 조건식 | 실제 리소스 속성, API 응답 |
| CI 적합성 | 높음 (환경 의존 없음) | 별도 AWS 계정 필요 |

#### 제한사항

- 반복 블록(repeated blocks) 구분 불가: 같은 리소스 타입의 모든 인스턴스에 동일한 mock 값 적용
- computed 속성만 mock 생성. 필수(required) 속성은 설정에서 직접 지정해야 함

### 5. override_resource, override_data, override_module (v1.7.0+)

mock_provider가 provider 전체를 대체하는 반면, override 블록은 **개별 리소스/데이터 소스/모듈**을 타겟팅한다.

#### override_resource

```hcl
override_resource {
  target = aws_s3_bucket.this
  values = {
    arn    = "arn:aws:s3:::test-bucket"
    bucket = "test-bucket"
  }
}
```

- provider를 호출하지 않고 해당 리소스의 속성값을 직접 주입
- `values` 생략 시 자동 생성 (mock_provider와 같은 규칙)

#### override_data

```hcl
override_data {
  target = data.aws_ami.latest
  values = {
    id           = "ami-test12345678"
    architecture = "x86_64"
  }
}
```

모듈 내부의 data source도 타겟팅 가능:

```hcl
override_data {
  target = module.network.data.aws_availability_zones.available
  values = {
    names = ["ap-northeast-2a", "ap-northeast-2c"]
  }
}
```

#### override_module

모듈 전체를 실행하지 않고 output만 주입:

```hcl
override_module {
  target = module.network
  outputs = {
    vpc_id     = "vpc-mock12345"
    subnet_ids = ["subnet-aaa", "subnet-bbb"]
  }
}
```

#### mock_provider와의 차이

| 비교 | mock_provider | override 블록 |
|------|--------------|---------------|
| 범위 | provider 전체 (해당 provider의 모든 리소스) | 개별 리소스/데이터 소스/모듈 |
| 선택성 | 전체 대체 | 특정 대상만 override, 나머지는 실제 실행 |
| 용도 | 전체 unit test | 부분 mocking (일부만 mock, 나머지 실제) |
| 배치 위치 | 파일 루트 레벨만 | 파일 루트 또는 run 블록 내부 |

#### override_during 속성 (v1.11.0+)

override 블록에도 `override_during`을 지정할 수 있다:

```hcl
override_resource {
  target          = aws_instance.web
  values          = { id = "i-mock" }
  override_during = plan    # plan 단계에서 값 주입
}
```

- `plan`: plan 단계에서 값을 생성하고 apply에서 재사용. plan 출력에서 실제 값 확인 가능.
- `apply` (기본값): plan 시 `(known after apply)` 표시, apply에서 값 생성.

#### 우선순위

run 블록 내부 override > 파일 루트 레벨 override > mock_provider 내부 mock_resource/mock_data > mock_provider source 파일

### 6. 테스트 실행 모델

#### command = apply 시 실제 리소스 생성

`command = apply`는 **실제 인프라를 생성한다** (mock_provider 사용 시 제외). 공식 문서에서 명확히 경고:

> "tests within Terraform create real infrastructure"

> "We recommend creating dedicated testing accounts...that you can routinely and safely purge."

#### 자동 destroy

테스트 완료 후 Terraform이 자동으로 리소스를 destroy한다.

- destroy 순서: run 블록의 **역순**
- Terraform이 의존성 순서를 자동으로 처리
- v1.14.0: `prevent_destroy` lifecycle 속성은 테스트 cleanup 시 무시됨
- destroy 실패 시 수동 정리 필요 (전용 테스트 계정 권장)

#### State 관리

- 테스트 state는 **완전히 in-memory**로 관리
- 라이브 인프라 state와 완전 분리
- 같은 모듈을 참조하는 run 블록은 state를 공유
- 각 대체 모듈(`module` 블록)은 별도 state
- `state_key`로 state 공유/분리를 명시적으로 제어 가능 (v1.11.0+)

```hcl
run "setup_a" {
  state_key = "env_a"
  module { source = "./setup" }
}

run "setup_b" {
  state_key = "env_b"
  module { source = "./setup" }
}
# setup_a와 setup_b는 별도 state
```

#### 병렬 실행 (v1.12.0+)

run 블록의 병렬 실행 조건:
1. 서로의 output을 참조하지 않을 것
2. 다른 state 파일을 사용할 것 (다른 `state_key` 또는 다른 `module` source)
3. `parallel = true`가 설정되어 있을 것

```hcl
test {
  parallel = true   # 파일 레벨 기본값
}

run "independent_a" {
  state_key = "a"
  # parallel = true (test 블록에서 상속)
}

run "independent_b" {
  state_key = "b"
  # parallel = true (test 블록에서 상속)
}

run "depends_on_a" {
  variables {
    input = run.independent_a.some_output   # 의존성 → a 완료 후 실행
  }
  parallel = false
}
```

#### run 블록 실행 순서

- 기본: **순차 실행** (파일 내 선언 순서)
- `parallel = true` 설정 시 조건 충족하는 블록끼리 병렬
- 파일 간 순서: 테스트 파일 단위로 실행 (파일 간 병렬 없음)

### 7. Variables in tests

#### 파일 레벨 variables

모든 run 블록에 적용:

```hcl
variables {
  region        = "ap-northeast-2"
  instance_type = "t3.micro"
}
```

#### run 레벨 variables (override)

특정 run 블록에서 파일 레벨 값을 덮어쓴다:

```hcl
run "with_large_instance" {
  variables {
    instance_type = "t3.large"    # 이 run에서만 override
  }
}
```

#### Variable 우선순위 (높은 순)

1. run 블록 variables
2. 테스트 파일 variables
3. 환경 변수 (`TF_VAR_*`)
4. Variable 정의 파일 (`.tfvars`)
5. CLI 입력 (사용 불가 — test 명령에 `-var` 플래그 없음)

#### .tfvars 파일 자동 로드

테스트 디렉토리에서 자동 로드 (v1.7.0+):
- `terraform.tfvars`
- `terraform.tfvars.json`
- `*.auto.tfvars`
- `*.auto.tfvars.json`

#### run 블록 간 변수 참조 (v1.13.0+)

파일 레벨 variables에서 run 블록 출력과 다른 변수를 참조 가능:

```hcl
run "setup" {
  module { source = "./setup" }
}

run "main_test" {
  variables {
    vpc_id = run.setup.vpc_id   # 이전 run 블록의 output 참조
  }
}
```

#### 외부 변수 정의 (v1.13.0+)

테스트 파일 내에서 외부 참조 변수의 정의를 직접 포함 가능:

```hcl
variables {
  external_input = "test-value"
}
```

#### sensitive 변수 (v1.9.0+)

run 블록에서 sensitive 값을 전달하면 동적 sensitivity가 유지된다.

### 8. 실전 패턴

#### Unit test (plan-only + mock provider)

```hcl
# tests/unit.tftest.hcl

mock_provider "aws" {
  mock_data "aws_ami" {
    defaults = {
      id = "ami-test12345678"
    }
  }
}

variables {
  instance_type = "t3.micro"
}

run "plan_only_check" {
  command = plan

  assert {
    condition     = aws_instance.web.instance_type == "t3.micro"
    error_message = "Instance type should be t3.micro"
  }

  assert {
    condition     = aws_instance.web.ami == "ami-test12345678"
    error_message = "AMI should come from data source"
  }
}
```

#### Integration test (apply + real provider)

```hcl
# tests/integration.tftest.hcl

variables {
  instance_type = "t3.micro"
}

run "create_and_verify" {
  command = apply

  assert {
    condition     = aws_instance.web.instance_state == "running"
    error_message = "Instance should be running after apply"
  }

  assert {
    condition     = output.public_ip != ""
    error_message = "Should have a public IP"
  }
}
```

#### 모듈 output 테스트

```hcl
# modules/network/tests/outputs.tftest.hcl

mock_provider "aws" {}

variables {
  vpc_cidr = "10.0.0.0/16"
}

run "verify_outputs" {
  command = apply

  assert {
    condition     = output.vpc_id != ""
    error_message = "VPC ID output should not be empty"
  }

  assert {
    condition     = length(output.subnet_ids) > 0
    error_message = "Should output at least one subnet ID"
  }
}
```

#### apply 후 리소스 속성 테스트

```hcl
run "verify_tags" {
  command = apply

  assert {
    condition     = aws_instance.web.tags["Name"] == "tf-core-lab01-instance-web"
    error_message = "Name tag mismatch"
  }

  assert {
    condition     = aws_instance.web.tags_all["ManagedBy"] == "Terraform"
    error_message = "ManagedBy tag should be set by default_tags"
  }
}
```

#### validation rule 테스트 (expect_failures)

```hcl
run "reject_invalid_instance_type" {
  command = plan

  variables {
    instance_type = "t1.micro"   # validation에서 거부되는 값
  }

  expect_failures = [
    var.instance_type,
  ]
}
```

#### precondition/postcondition 테스트

```hcl
run "test_precondition_failure" {
  command = plan

  variables {
    environment = "invalid"   # precondition 위반
  }

  expect_failures = [
    aws_instance.web,   # 이 리소스의 precondition 실패 예상
  ]
}
```

#### setup 모듈 패턴

테스트 전용 setup 모듈로 사전 인프라를 구성하고 본 테스트에 전달:

```hcl
run "setup_vpc" {
  module {
    source = "./tests/setup"
  }
}

run "test_main" {
  variables {
    vpc_id    = run.setup_vpc.vpc_id
    subnet_id = run.setup_vpc.subnet_id
  }

  assert {
    condition     = aws_instance.web.subnet_id == run.setup_vpc.subnet_id
    error_message = "Instance should be in the setup subnet"
  }
}
```

### 9. 제한사항 및 주의사항

#### terraform_remote_state

공식 문서에서 `terraform_remote_state` 관련 테스트 제한은 명시되어 있지 않으나, mock_provider로 remote state를 mock할 수 없다 (`terraform_remote_state`는 built-in이므로). `override_data`로 개별 data source를 override하는 방식으로 우회 가능.

#### module source 제한

run 블록의 `module` 블록에서:
- 로컬 경로 지원
- Registry 모듈 지원 (source + version)
- Git URL 등 다른 소스는 제한적일 수 있음

#### provider 설정

- v1.9.0+: 테스트 파일의 provider 블록에서 **version constraint 지정 불가**. 메인 설정의 `required_providers`에 정의해야 함.
- provider 블록에서 변수/run 출력 참조는 v1.7.0+에서 지원

#### mock_provider 제한

- 반복 블록(예: `aws_dynamodb_table`의 `replica`) 구분 불가 — 모든 인스턴스에 동일값 적용
- computed 속성만 자동 생성. required 속성은 설정에서 명시해야 함
- mock provider는 provider schema를 그대로 사용하므로 schema 변경 시 mock도 영향 받음

#### 기타

- 테스트 state는 in-memory이므로 디스크에 저장되지 않음. 디버깅 시 `-verbose` 활용
- apply 테스트 실패 시 destroy가 수행되지만, destroy 자체가 실패하면 리소스가 남음
- 테스트 파일은 `terraform init` 시 인식됨 — init 단계에서 테스트에 필요한 provider/module도 설치

## 주의사항

1. **apply 테스트는 실제 비용 발생:** `command = apply` + real provider = 실제 AWS 리소스 생성. 전용 테스트 계정 사용 권장.
2. **mock_provider는 credential 불요:** CI에서 unit test만 실행 시 AWS credential 설정 불필요.
3. **v1.9.0 breaking change:** 테스트 파일 provider 블록에서 version constraint를 사용하면 에러 발생. 기존 테스트 마이그레이션 필요.
4. **destroy 실패 대비:** integration test에서 destroy 실패 시 리소스가 남는다. 정기적 리소스 정리 체계(AWS Nuke 등) 권장.
5. **override_during 기본값 주의:** 기본값은 `apply`이므로 plan-only 테스트에서 override 값을 확인하려면 `override_during = plan` 설정 필요.
6. **병렬 실행 조건 엄격:** state 공유, output 참조가 있으면 병렬 실행되지 않음. 의도치 않은 순차 실행에 주의.
