# 1.1 인터프리터와 바이트코드

## 개요

파이썬을 "인터프리터 언어"라고 부르는 말은 널리 쓰이지만, 실제 실행 방식과 맞지 않는 경우가 많다. 소스 코드를 한 줄씩 곧바로 실행하는 전통적인 인터프리터가 아니라, CPython은 **소스를 바이트코드로 컴파일한 뒤, 가상 머신(VM)에서 그 바이트코드를 실행**한다. "인터프리터"라는 이름 때문에 컴파일 단계가 있다는 사실을 놓치기 쉽다.

이 주제를 이해하면 `.pyc` 파일이 무엇인지, `import`가 왜 "첫 import가 느린" 현상을 만드는지, 그리고 `dis`·`compile()`로 실행 모델을 들여다볼 수 있는 기반이 생긴다. 이후 `__pycache__`, 모듈 로딩, 실행 모드(`-m` 등)에 대한 논의도 이 파이프라인 위에 올라간다.

---

## 핵심 개념

### 실행 파이프라인: 소스 → AST → 바이트코드 → VM

CPython이 `foo.py`를 실행할 때 내부적으로는 다음 단계를 거친다.

1. **소스 코드 파싱**  
   `.py` 텍스트를 구문 분석하여 **AST(Abstract Syntax Tree)** 를 만든다. 이 단계에서 문법 오류가 있으면 `SyntaxError`로 실행이 시작되기 전에 종료된다.

2. **바이트코드 컴파일**  
   AST를 **바이트코드(bytecode)** 로 바꾼다. 바이트코드는 스택 기반 가상 머신용 명령 집합이다. `LOAD_FAST`, `CALL_FUNCTION`, `BINARY_ADD` 같은 opcode와 그 인자로 구성된다.

3. **바이트코드 실행**  
   CPython의 **가상 머신(CEval)** 이 바이트코드를 한 번에 한 opcode씩 실행한다. 이 단계가 "인터프리팅"에 해당한다. 소스가 아니라 **바이트코드를** 해석한다는 점이 전통적인 "한 줄씩 소스 해석"과 다르다.

즉, **컴파일(소스→바이트코드)과 인터프리팅(바이트코드→실행)** 이 분리되어 있다. C/Java처럼 머신 코드나 JVM 바이트코드로 빌드해 두는 것은 아니지만, "파이썬 소스 = 곧바로 실행"이라는 단순 모델은 CPython의 실제 동작을 반영하지 않는다.

### .pyc와 __pycache__

모듈을 `import`할 때, CPython은 해당 모듈의 바이트코드를 **캐시**해 둔다. `__pycache__` 디렉터리 안에 `{모듈이름}.cpython-{버전}.pyc` 형식으로 저장된다. 예: `foo.cpython-312.pyc`.

- **생성 시점**  
  해당 모듈이 **처음 import될 때**, 또는 `.py`가 `.pyc`보다 더 최근에 수정되었을 때(또는 `.pyc`가 없을 때) 바이트코드가 (재)컴파일되고 `.pyc`가 (다시) 쓰인다.

- **직접 실행 vs import**  
  `python script.py`처럼 **스크립트를 직접 실행**할 때는, 그 스크립트에 대해서는 기본적으로 `.pyc`를 디스크에 쓰지 않는다. 다만 `import`를 통해 로드되는 **다른 모듈**들은 `__pycache__`에 `.pyc`가 생긴다.  
  (`-B` 옵션을 주면 `.pyc` 읽기·쓰기를 비활성화한다.)

- **형식**  
  `.pyc` 앞부분에는 매직 넘버, 소스 크기/mtime 등의 메타데이터가 있고, 그 뒤에 `marshal`로 직렬화한 바이트코드가 붙어 있다. CPython 버전이 바뀌면 이 형식이 달라질 수 있어, 이전 버전의 `.pyc`는 새 버전에서 쓰이지 않는다.

### dis와 compile(): 바이트코드 들여다보기

바이트코드를 직접 보려면 표준 라이브러리 `dis`를 사용한다.

- **`dis.dis(x)`**  
  - `x`가 함수·메서드·제너레이터·코드 객체 등이면, 그 안에 들어 있는 바이트코드를 **역어셈블**해서 사람이 읽을 수 있는 opcode 목록으로 보여 준다.  
  - `x`가 모듈·클래스면, 그 안에 정의된 함수·메서드들의 바이트코드를 재귀적으로 보여 준다.  
  - 문자열을 넘기면, 그 문자열을 `compile(..., 'exec')`한 결과의 바이트코드를 역어셈블한다.

- **`compile(source, filename, mode, ...)`**  
  - `source`: 소스 문자열.  
  - `filename`: 에러 메시지·스택에 나올 이름.  
  - `mode`: `'exec'`(전체 스크립트), `'eval'`(단일 표현식), `'single'`(대화형 한 줄. `None` 출력 억제 등 REPL처럼 동작).  
  - 반환값은 **코드 객체(code object)** 이다. 이 객체를 `exec()` 또는 `eval()`에 넘겨 실행할 수 있다. `dis.dis(코드객체)`로 그 바이트코드를 볼 수 있다.

이 두 도구로 "이 한 줄이 실제로는 어떤 opcode sequence인가"를 확인할 수 있다. 예를 들어 `a + b`가 `BINARY_OP`(3.12) 또는 `BINARY_ADD`로, `f(x)`가 `CALL`/`CALL_FUNCTION` 계열로 번역되는 것을 직접 보면, "표현식/문이 바이트코드로 어떻게 펼쳐지는가"에 대한 직관이 생긴다.

### JIT·다른 구현체

CPython은 위와 같은 **순수 인터프리터 방식**으로 바이트코드를 실행한다. JIT(Just-In-Time) 컴파일은 쓰지 않는다.  

다른 구현체는 이 파이프라인을 바꾼다. 예를 들어 **PyPy**는 JIT를 사용해 바이트코드(및 그 위의 RPython 트레이스)를 머신 코드로 컴파일해, 반복되는 루프 등에서 CPython보다 훨씬 빠를 수 있다. **Cython**은 파이썬류 소스를 C로 변환해 네이티브 모듈을 만든다.  
이 글의 범위에서는 CPython의 "소스 → AST → 바이트코드 → VM"만 다루고, JIT나 다른 구현체는 "그런 대안이 있다" 수준으로만 언급한다.

## 코드 예제

### 1) dis로 한 줄의 바이트코드 보기

```python
import dis

# 단일 표현식
dis.dis("x = 1 + 2")
```

출력(CPython 3.12 기준, 환경에 따라 인자 등이 조금 다를 수 있음):

```
  1           0 RESUME                   0
              2 LOAD_CONST               0 (1)
              4 LOAD_CONST               1 (2)
              6 BINARY_OP                0 (+)
             10 STORE_NAME               0 (x)
             12 LOAD_CONST               2 (None)
             14 RETURN_VALUE
```

`1`과 `2`를 스택에 넣고(`LOAD_CONST`), `BINARY_OP`로 덧셈한 뒤, 결과를 이름 `x`에 넣는(`STORE_NAME`) 흐름이 보인다. "`x = 1 + 2`가 이런 opcode sequence로 컴파일된다"는 **아하**가 생기는 부분이다.

### 2) 함수의 바이트코드

```python
import dis

def add(a, b):
    return a + b

dis.dis(add)
```

`add`의 코드 객체에 들어 있는 바이트코드가 역어셈블되어, 인자 로드·`BINARY_OP`·`RETURN_VALUE` 순서를 확인할 수 있다. `dis.dis("def add(a,b): return a+b")`처럼 문자열로 넘겨도 된다.

### 3) compile + dis로 코드 객체 직접 다루기

```python
import dis

src = """
y = 10
z = y * 2
"""
code = compile(src, "<snippet>", "exec")
dis.dis(code)
```

`<snippet>`은 스택 트레이스 등에 나올 이름이다. `'exec'` 모드이므로 여러 문을 포함할 수 있고, `dis.dis(code)`는 그 전체 바이트코드를 보여 준다. `exec(code)`를 호출하면 이 바이트코드가 실행된다. "소스 → 코드 객체 → 실행"이 `compile`과 `exec`로 분리되어 있음을 보여 주는 예이다.

### 4) __pycache__에 생성된 .pyc 확인

```python
# 아무 모듈이나 import (예: json)
import json
```

이후 `__pycache__`를 보면 `json.cpython-312.pyc` 같은 파일이 생겨 있을 수 있다(이미 있고, `json`이 갱신되지 않았으면 수정 시간만 유지). `import`가 "해당 모듈의 바이트코드를 (필요 시) 컴파일해 `__pycache__`에 둔 뒤, 그걸 로드해 실행한다"는 사실을, 디스크上的으로 확인하는 데 쓸 수 있다.

## 주의사항 (Pitfalls)

### 1) "인터프리터 = 한 줄씩 소스 실행"이 아니다

C/Java에 익숙한 사람은 "인터프리터 = 컴파일 없이 소스만 보고 실행"이라고 생각하기 쉽다. CPython은 **소스를 바이트코드로 컴파일한 뒤, 그 바이트코드를 인터프리팅**한다. 그래서:

- 문법 오류(`SyntaxError`)는 **실행 전**, 컴파일 단계에서 잡힌다.
- `import` 시 그 모듈 전체가 **한 번에** 바이트코드로 컴파일된다. "한 줄씩만 컴파일"하는 구조가 아니다.

### 2) 스크립트 직접 실행 시 해당 파일의 .pyc는 보통 생성되지 않는다

`python my_script.py`처럼 **진입점이 되는 스크립트**는, 기본 설정에서는 `__pycache__`에 `.pyc`를 남기지 않는다. `__pycache__`에 보이는 `.pyc`는 대부분 **다른 모듈**을 `import`하면서 만들어진 것이다.  
"스크립트를 두 번 실행했는데 `.pyc`가 없다"고 해서 이상한 것은 아니다.

### 3) .pyc는 CPython 버전·플랫폼에 종속적이다

`.pyc` 파일 이름에 `cpython-312`처럼 버전이 들어가고, 내부 포맷도 CPython 버전마다 바뀔 수 있다. 3.11에서 만든 `__pycache__`를 3.12에서 그대로 쓰려 하면 무시되고 재컴파일될 수 있다.  
`.pyc`를 Git에 올리거나, 배포 시 버전을 섞어 쓰지 않도록 한다.

### 4) dis 출력은 CPython 버전마다 다를 수 있다

`BINARY_ADD`가 3.12에서 `BINARY_OP`로 통합되는 등, opcode 이름·개수가 버전별로 바뀐다. 문서나 블로그의 `dis` 예시가 자신의 버전과 다르면, "개념은 같고 표현만 다르다"고 보면 된다.

### 5) compile()의 filename은 "진짜 경로"가 아니라 표시용이다

`compile(source, "fake_name.py", "exec")`처럼 아무 문자열이나 넘겨도 동작한다. 이 값은 `SyntaxError` 메시지, `traceback`의 파일 이름 등에 쓰인다. 디버깅을 위해 의미 있는 이름을 주는 것이 좋다.

## Deep Insight

### "컴파일"과 "인터프리팅"의 경계

일반적으로 말하는 **컴파일**은 "고수준 표현(소스)을 저수준 표현(바이트코드·머신 코드)으로 번역"하는 과정이다. **인터프리팅**은 "그 저수준 표현을 한 단계씩 실행"하는 과정이다.  
CPython에서는:

- **컴파일**: `.py` → AST → 바이트코드(코드 객체).
- **인터프리팅**: 바이트코드(코드 객체)를 VM(CEval)이 opcode 단위로 실행.

그래서 "파이썬은 컴파일도 하고, 인터프리팅도 한다"가 정확한 설명에 가깝다. "인터프리터"라는 이름은 **최종 단계(바이트코드를 한 번에 한 opcode씩 실행)** 에서 붙었다고 보는 편이 자연스럽다.

### 코드 객체(Code Object)가 담는 것

`compile()`의 반환값인 **코드 객체**에는 바이트코드 opcode sequence뿐 아니라, 그 코드가 쓰는 상수, 이름(변수·함수 등), 로컬 변수 개수, 플래그 등이 들어 있다. `dis`는 이 코드 객체를 읽어 사람이 보기 좋게 풀어 낸다.  
함수를 정의하면, 그 함수 본문은 하나의 **코드 객체**로 컴파일되어 함수 객체의 `__code__`에 붙는다. `dis.dis(f)`는 곧 `f.__code__`에 담긴 바이트코드를 역어셈블하는 것이다.  
이런 구조를 알면 "함수 = 코드 객체 + 클로저 + 글로벌 참조 등을 묶은 객체"라는, 나중에 배우는 실행 모델과도 잘 이어진다.

## 요약

| 항목 | 내용 |
|------|------|
| **실행 단계** | 소스(.py) → AST → 바이트코드(코드 객체) → VM(CEval)이 opcode 단위로 실행. "인터프리터"라 해도 그 전에 **컴파일** 단계가 있다. |
| **.pyc / __pycache__** | `import` 시 모듈 바이트코드를 `__pycache__/{이름}.cpython-{버전}.pyc`에 캐시. 스크립트를 `python script.py`로 직접 실행할 때, 그 **스크립트 파일**에 대한 `.pyc`는 기본적으로 생성하지 않는다. |
| **dis** | `dis.dis(함수|코드객체|문자열)`로 바이트코드를 역어셈블해 opcode 목록을 본다. "이 문/표현식이 어떤 바이트코드로 번역되는가"를 보는 도구. |
| **compile()** | `compile(source, filename, mode)`로 소스를 **코드 객체**로 컴파일. `mode`는 `'exec'`·`'eval'`·`'single'`. `exec()`·`eval()`에 넘겨 실행할 수 있고, `dis.dis(코드객체)`로 내용을 볼 수 있다. |
| **주의** | "인터프리터 = 한 줄씩 소스 실행"이 아니라 "소스→바이트코드 컴파일 + 바이트코드 인터프리팅". `.pyc`는 버전·플랫폼에 따라 호환되지 않을 수 있고, `dis` 출력도 CPython 버전에 따라 다르다. |
