# 1. S3가 의미하는 것

## 1. S3(Simple Storage Service)는 Object Storage다

S3는 파일을 "Object"로 저장하는 Object Storage다. EBS처럼 특정 인스턴스에 붙는 Block Storage와 달리, S3는 네트워크로 접근하는 저장소이며 저장 용량의 상한을 의식하지 않아도 된다.

### ① Block Storage(EBS)와의 차이

- EBS: 인스턴스에 붙는 디스크, 파일시스템을 올려 쓴다, AZ 단위
- S3: Object(파일+메타데이터)로 저장, HTTP 기반 접근, Region 단위

[이미지: 저장소 비교 - EBS는 EC2에 attach되는 디스크, S3는 독립된 객체 저장소 - 접근 방식(attach vs API) 비교]

이 차이는 "애플리케이션 데이터가 어디에 남는가"의 관점을 바꾼다. 업로드 파일을 EBS에 두면 인스턴스 교체 시 데이터가 함께 이동하기 어렵고, S3에 두면 실행 환경(EC2/ASG/ECS)과 저장소가 분리된다.

### ② 내구성과 가용성의 관점

S3는 높은 내구성(11 9s)으로 알려져 있고, 서비스 자체가 Region 내 여러 시설에 데이터를 분산 저장하는 방식으로 설계되어 있다. 이 시리즈에서는 "왜 업로드/정적 파일 저장소로 S3가 표준인가"를 이해하는 것이 목적이다.

# 2. Bucket과 Object

## 1. Bucket은 컨테이너다

Bucket은 Object를 담는 컨테이너이며, 다음 특성이 중요하다.

- Bucket 이름은 글로벌로 유일해야 한다.
- Bucket은 Region을 가진다(생성 시 선택).
- 접근 제어(Bucket Policy, Block Public Access)는 Bucket 단위로 이뤄진다.

[이미지: AWS Console - S3 - Buckets - Bucket list 화면 - Bucket name/Region 컬럼 확인 포인트]

## 2. Object는 파일 + 메타데이터다

Object는 파일 본문과 메타데이터를 함께 가진다. S3에서 "디렉터리"는 실제 폴더가 아니라 Key prefix로 표현된다.

### ① Key는 경로처럼 보이지만 문자열이다

`images/2026/03/a.png` 같은 형태는 폴더가 아니라 Key 문자열이다. Console은 prefix를 폴더처럼 보여주지만, 저장 모델은 객체 단위다.

### ② S3 URI와 HTTP URL

S3를 설명할 때 다음 식별자가 자주 나온다.

- S3 URI: `s3://{bucket-name}/{key}`
- Console URL: 브라우저에서 보는 관리 화면
- Object URL: 공개 설정이 있을 때만 직접 접근 가능(정적 호스팅/CloudFront로 연결할 때 중요)

# 3. Storage Class(개요)

## 1. 접근 패턴과 비용의 트레이드오프다

Storage Class는 "자주 접근하는가"와 "얼마나 오래 보관하는가"에 따라 비용 구조가 다르다.

예:

- Standard: 자주 접근
- Standard-IA: 비빈번 접근
- Glacier 계열: 아카이브

이 시리즈에서는 Storage Class를 상세하게 비교하지 않는다. 실습에서는 기본(Standard)으로 업로드하고, "클래스는 운영에서 비용 전략"이라는 관점을 잡는 것이 목표다.

---

# 4. Versioning

## 1. Versioning은 Object의 히스토리를 남긴다

Bucket Versioning을 활성화하면, 같은 Key로 업로드해도 기존 Object가 덮어쓰여 사라지지 않고 버전으로 남는다.

[이미지: AWS Console - S3 - Bucket - Properties - Versioning 설정 화면 - Enabled 확인 포인트]

### ① 삭제는 Delete marker로 표현된다

Versioning이 켜진 Bucket에서 Object를 삭제하면, 실제 데이터가 즉시 삭제되지 않고 delete marker가 만들어진다. 복원이 가능한 이유가 여기 있다.

### ② 운영에서의 의미

Versioning은 실수로 덮어쓴 파일, 잘못 삭제한 파일을 복구하는 기반이 된다. 다만 버전이 쌓이므로 Lifecycle Policy와 함께 관리하는 경우가 많다.

---

# 핵심 정리

- S3는 Object Storage이며 실행 환경(EC2/ECS)과 저장소를 분리하는 데 적합하다.
- Bucket은 글로벌 유일 이름과 Region을 가지며, 접근 제어는 Bucket 단위로 설정한다.
- Object는 파일+메타데이터이며, 폴더처럼 보이는 구조는 Key prefix 표현이다.
- Versioning은 덮어쓰기/삭제를 "버전"으로 남기며, 삭제는 delete marker로 표현된다.

# 4. 실행 및 결과 검증

설명: 업로드된 Object가 보이고, 버전이 생성되며, 삭제가 delete marker로 표현되면 성공이다.

## 1. Object 목록/Key 확인

[이미지: AWS Console - S3 - Bucket - Objects 목록 - Key prefix로 폴더처럼 보이는 화면]

[이미지: AWS Console - S3 - Bucket - Objects - Show versions 토글 - Version ID 확인]

[이미지: AWS Console - S3 - Object delete - 삭제 후 Show versions에서 delete marker 확인]

다음을 확인한다.

- 삭제 후에도 버전 목록에 데이터 버전이 남아 있다.
- delete marker가 생성된다.
