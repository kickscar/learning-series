# 1. Elastic IP

## 1. Elastic IP가 의미하는 것

Elastic IP(EIP)는 AWS가 제공하는 고정 Public IPv4 주소다. 일반적인 EC2 Public IP는 Instance Stop/Start 과정에서 바뀔 수 있지만, EIP는 사용자가 "주소를 할당(Allocate)하고 리소스에 연결(Associate)하는" 방식으로 관리한다.

EIP의 본질은 "서버의 주소를 고정한다"가 아니라, **Public IP를 Instance 수명에서 분리한다**는 점이다.

### ① Public IP 자동 할당과의 차이

- Public IP 자동 할당: Instance에 임시로 붙는 주소(Stop/Start로 바뀔 수 있음)
- Elastic IP: 계정 단위로 예약해두고, 필요할 때 원하는 리소스에 붙였다 뗄 수 있음

---

## 2. 비용과 운영 관점

EIP는 "남겨두면 비용이 쌓일 수 있는 리소스"에 속한다. 과금 정책은 조건에 따라 달라질 수 있으므로, 실습에서는 다음 원칙을 강제한다.

### ① 실습 후 Release를 기본값으로 둔다

- 연결되지 않은 EIP는 비용이 발생할 수 있다.
- 연결 상태와 무관하게 Public IPv4/EIP 비용이 발생할 수 있는 정책도 존재한다.

따라서 실습이 끝나면 EIP를 Release하는 것을 기본값으로 둔다. 단, 프로젝트 Lab을 바로 이어서 진행해 고정 주소가 필요한 경우에는 Release를 미룬다.

### ② EIP 수명 관리 동작

- Allocate: EIP를 계정에 할당
- Associate: Instance(또는 ENI)에 연결
- Disassociate: 연결 해제
- Release: 계정에서 반환

---

# 핵심 정리

- Elastic IP는 고정 Public IPv4이며, Public IP를 Instance 수명에서 분리하는 방식으로 관리한다.
- EIP는 Allocate/Associate/Disassociate/Release 흐름으로 수명 관리를 한다.
- 비용 리스크가 있으므로 실습 종료 시 Release를 기본값으로 둔다(프로젝트 진행 중이면 정리 시점 조정).

# [실습] lab10: Elastic IP 할당과 연결

Elastic IP를 Allocate하고 EC2 Instance에 Associate한 뒤, Instance Stop/Start 이후에도 동일한 Public IP가 유지되는 것을 확인한다. 마지막에 Disassociate/Release로 정리한다.

### 실습 목표

- Elastic IP를 Allocate한다.
- Elastic IP를 EC2 Instance에 Associate한다.
- Stop/Start 후에도 동일 IP가 유지됨을 확인한다.
- Disassociate/Release로 비용 리스크를 제거한다.

⚠️ 비용 주의: Public IPv4/EIP는 조건에 따라 비용이 발생할 수 있다. 실습 종료 시 Disassociate/Release를 수행한다.

---

# 1. 전체 아키텍처

```mermaid
flowchart LR
  U[사용자] --> Internet[Internet]
  Internet --> EIP[Elastic IP]
  EIP --> IGW[Internet Gateway]
  IGW --> VPC[VPC]
  VPC --> ENI[ENI]
  ENI --> SG[Security Group]
  SG --> EC2[EC2 Instance]
```

이 아키텍처는 외부에서 볼 때 고정 IP(EIP)가 Instance 앞단에 붙는 구조를 보여준다. EIP는 결국 Instance의 ENI에 연결되는 주소다.

# [실습] Lab: Gallery - EC2 기본 배포

EC2 Instance에 애플리케이션을 배포하고, Security Group/EIP를 통해 외부에서 접근 가능한 엔드포인트를 만든 뒤 `:8080`으로 정상 응답을 확인한다. 기본 구성은 `dev` + local storage + H2로 둔다.

### 실습 목표

- EC2 Instance에 Java 실행 환경이 준비되어 있는지 확인하고, 없으면 설치한다.
- Git으로 소스 코드를 내려받고, Maven Wrapper로 빌드한다.
- 빌드된 JAR를 `--server.port=8080`으로 실행한다.
- Security Group으로 `8080` 접근을 허용한다.
- 브라우저에서 `http://{public-endpoint}:8080`으로 정상 응답을 확인한다.

⚠️ 비용 주의: EC2/EIP/Public IPv4는 조건에 따라 비용이 발생할 수 있다. 학습 세션이 끝나면 자원 정리 기준을 적용한다.

---

# 1. 전체 아키텍처

```mermaid
flowchart LR
  User[사용자 브라우저] --> Internet[Internet]
  Internet --> EP[Public Endpoint\n(EIP 또는 Public IP)]
  EP --> IGW[Internet Gateway]
  IGW --> VPC[VPC]
  VPC --> SG[Security Group]
  SG --> EC2[EC2 Instance]
  EC2 --> App[Gallery App\n(local storage + H2)]
```

이 아키텍처는 EC2에 배포한 애플리케이션에 외부에서 접근하는 최소 구조다. 이후 Ch05에서는 EIP 대신 ALB로 엔드포인트를 대체하는 방향으로 확장한다.

# 4. 실행 및 결과 검증

## 1. SSH 접속

```bash
ssh -i **{path-to-key.pem}** **{user}**@**{public-endpoint}**
```

## 2. Java 설치

⚠️ 주의: `lab09`에서 만든 AMI 기반 Instance라면 이미 설치되어 있을 수 있다. `java -version`으로 먼저 확인한 뒤, 없을 때만 설치한다.

```bash

```

## 3. Git/Maven 준비

⚠️ 주의: `lab09` AMI 기반 Instance라면 이미 준비되어 있을 수 있다. 없을 때만 설치한다.


## 4. 소스 코드 clone

```bash
git clone **{gallery-repo-url}** gallery-spring-boot
cd gallery-spring-boot
```

## 5. 빌드(Maven Wrapper)

```bash
./mvnw clean package -DskipTests -Dbuild.finalName=gallery
```

산출물은 `target/gallery.jar`다.

## 6. 애플리케이션 실행(기본 dev + 포트 8080)

⚠️ 주의: `prod`는 기본 포트가 80이라 권한 이슈가 생길 수 있다. 이 시리즈에서는 기본 `dev`로 실행하거나, `prod`를 쓰더라도 `--server.port=8080`으로 덮어쓴다.

```bash
nohup java -jar target/gallery.jar \
--server.port=8080 \
--app.storage.type=local \
--app.storage.local.path=/mnt/data/gallery-uploads \
> app.log 2>&1 &
tail -n 50 app.log
```
