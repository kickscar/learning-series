# Research: 01.03 — Terraform 핵심 개념

> 조사일: 2026-04-15
> 기준: Terraform 1.14.x, hashicorp/aws provider >= 6.0

## 검증 결과

| mermaid 흐름도 | ✓ 정확 | User → init → plan ↔ State → apply ↔ State → Cloud 구조 정확 |
| 핵심 구성 요소 5가지 (Configuration, Provider, State, Plan, Apply) | ✓ 정확 | 개략적 설명 수준으로 적절. 각 요소별 후속 챕터 안내 포함 |

| `.terraform.lock.hcl` 커밋 권장 | ✓ 정확 | HashiCorp 공식 권장: "You should include this file in your version control repository." Provider 버전 고정 + checksum 검증 목적 |
| .gitignore에 `crash.log`, `override.tf` 등 누락 | ⚠ 주의 | draft의 .gitignore 예시에는 핵심 4개만 포함. GitHub 공식 템플릿에는 `crash.log`, `crash.*.log`, `override.tf`, `override.tf.json`, `*_override.tf`, `*_override.tf.json`, `.terraformrc`, `terraform.rc`도 포함. 학습 목적상 핵심 항목만 다루는 것은 합리적이나, 전체 목록 참고 안내가 있으면 좋음 |
| Configuration 설명 | ✓ 정확 | ".tf 확장자 파일 전체를 하나의 Configuration으로 합친다" — merge 동작 정확 |
| Provider 설명 | ✓ 정확 | 플러그인, init 시 다운로드, 리소스 타입 제공 |
| State 설명 (terraform.tfstate) | ✓ 정확 | 팀 환경에서 원격 저장소(S3) 보관 언급 |

## 주의사항

- **`terraform.tf` vs `providers.tf` 구분**: HashiCorp 공식 Style Guide(2026)는 `terraform` 블록(required_providers, required_version)을 `terraform.tf`에, `provider` 블록을 `providers.tf`에 분리 권장한다. draft는 두 블록을 `providers.tf`에 통합했는데, 학습 단계에서는 합리적인 선택이다. 단, 공식 가이드와 다르다는 점을 독자에게 한 줄 안내하면 혼란을 방지할 수 있다.
- **`.gitignore` 완전성**: draft의 .gitignore 예시는 핵심 패턴(`.terraform/`, `*.tfstate`, `*.tfstate.backup`, `*.tfvars`)만 포함한다. GitHub 공식 Terraform.gitignore에는 `crash.log`, `crash.*.log`, `override.tf`, `override.tf.json`, `*_override.tf`, `*_override.tf.json`, `.terraformrc`, `terraform.rc`도 포함된다. 학습 초기라 핵심만 다루는 것은 문제없으나, 참고 링크 또는 "전체 목록은 GitHub gitignore 참고" 한 줄 추가 권장.
- **`*.tfvars` 제외 논쟁**: 일부 커뮤니티에서 `terraform.tfvars`(민감정보 없는 기본값)는 커밋해야 한다고 주장하나, GitHub 공식 템플릿과 대다수 가이드는 `*.tfvars` 전체 제외를 권장한다. draft의 입장은 안전한 기본값으로 정확하다.
- **.terraform.lock.hcl은 Provider만 추적**: lock 파일은 Provider 의존성만 기록하며 Module 버전은 추적하지 않는다. draft에서 이 구분을 명시하면 더 정확하다.
