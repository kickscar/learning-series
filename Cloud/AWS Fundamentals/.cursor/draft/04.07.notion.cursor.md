# 1. Traffic Flow 7단계 모델

## 1. 단계 1: 사용자 브라우저 및 공용 인터넷(User Browser & Public Internet)

사용자는 브라우저에서 엔드포인트(도메인, ALB DNS, Public IP 등)로 요청을 보낸다. 이 시점의 관점은 "내 요청이 AWS에 도달했는가"다.

## 2. 단계 2: Internet Gateway(IGW)

IGW는 VPC의 인터넷 관문이다. 외부에서 들어오는 트래픽도, VPC에서 나가는 트래픽도 IGW를 통해 인터넷으로 이어진다.

## 3. 단계 3: Route Table

Route Table은 목적지(Destination)와 Target으로 라우팅을 결정한다. Public/Private의 본질은 `0.0.0.0/0`의 Target이 IGW인지, NAT인지, 없는지로 결정된다.

## 4. 단계 4: Network ACL(NACL)

NACL은 Subnet 경계에서 트래픽을 제어한다. stateless이며 Allow/Deny를 번호 순서로 평가한다. 운영에서 기본 트래픽 제어 도구는 SG이지만, NACL이 앞단에서 막으면 SG까지 도달하지 못한다.

## 5. 단계 5: Subnet(Public 또는 Private)

Subnet은 배치 구역이며, 라우팅과 보안 경계가 적용되는 단위다. Public/Private는 이름이 아니라, 라우팅과 Public IP 조건으로 결정된다.

## 6. 단계 6: Security Group(SG)

SG는 ENI 수준의 방화벽이며 stateful이다. 운영에서 가장 흔한 트래픽 제어 지점이며, "누가 누구에게 어떤 포트로 들어오는가"를 명확히 하는 도구다.

## 7. 단계 7: Compute(EC2, Docker, ECS 등)

마지막으로 Compute가 요청을 처리한다. 이 시리즈는 EC2에서 시작해 ECS(Fargate)로 전환한다. Compute 형태가 바뀌어도 2~6단계(네트워크/보안 경계)의 구조는 그대로 유지된다.

[이미지: 네트워크 구조 - 1~7단계 트래픽 흐름(Internet -> IGW -> Route Table -> NACL -> Subnet -> SG -> Compute) - 각 단계에 "차단 포인트" 주석]

이 그림은 "요청이 어디에서 막힐 수 있는가"를 한 장으로 요약한다. 트러블슈팅은 이 순서를 따라가며 원인을 계층별로 분리하는 과정이다.

# 2. Public Subnet Inbound 흐름

## 1. 외부 접근의 최소 조건

외부에서 Public Subnet의 리소스(예: Public EC2)에 접근하려면 다음이 필요하다.

- Subnet route: `0.0.0.0/0 -> IGW`
- 리소스가 Public IPv4 또는 EIP를 가진다
- SG inbound이 열려 있다(예: SSH 22 from 내 IP)
- (NACL을 쓴다면) NACL에서도 허용된다

이 조건이 하나라도 빠지면 "접속이 안 된다". 중요한 것은 이것이 "한 가지 설정 문제"가 아니라, 흐름의 문제라는 점이다.

---

# 3. Private Subnet 격리와 Outbound

## 1. Private는 기본이 격리다

Private Subnet은 외부에서 접근 경로가 없다. 운영에서는 Private 전용 Route Table을 만들고 local route만 두어 격리 상태로 시작하는 것이 자연스럽다.

## 2. Outbound는 NAT로 연다

패키지 설치, 의존성 다운로드 같은 운영 요구가 생기면 NAT Gateway로 Outbound를 연다. 이때도 Inbound는 열리지 않는다.

따라서 Private Subnet의 애플리케이션을 외부에 노출하려면 별도의 엔드포인트가 필요하다. 표준 엔드포인트가 ALB이며, 이는 다음 Chapter에서 구성한다.

---

# 4. 트러블슈팅 순서

## 1. "어디에서 막히는가"를 순서대로 확인한다

연결이 안 될 때는 감으로 만지지 않고, 다음 순서로 확인한다.

- Public IPv4/EIP가 있는가(외부 접근 경로의 전제)
- SG inbound/outbound이 의도대로인가
- NACL inbound/outbound이 의도대로인가(사용 중인 경우)
- Route Table의 `0.0.0.0/0` Target이 의도대로인가
  - Public: IGW
  - Private Outbound: NAT
  - Private 격리: 없음(local만)
- IGW/NAT 상태가 Available인가

이 순서가 "Traffic Flow 7단계 모델"을 운영의 체크리스트로 바꾼 형태다.

---

# 핵심 정리

- Traffic Flow는 1~7단계(Internet -> IGW -> Route Table -> NACL -> Subnet -> SG -> Compute)로 분해해 이해한다.
- Public/Private의 본질은 라우팅(Target)이며, Public은 IGW route + Public IP 조건이 필요하다.
- Private는 기본 격리로 시작하고, 필요할 때 NAT로 Outbound를 연다.
- Private 애플리케이션의 외부 노출은 NAT가 아니라 ALB 같은 엔드포인트로 해결한다(다음 Chapter).

# [실습] Lab: Gallery - Custom VPC 이전

Ch03에서 Default VPC에 배포했던 Gallery 실행 환경을 Custom VPC로 이전한다. 이 실습은 앞 Section Lab에서 만들었던 리소스를 "재사용"하는 흐름이 아니라, 복습 의미로 Gallery 프로젝트에 맞는 이름/구성으로 네트워크를 다시 쌓아 올리는 형태로 진행한다.

전제: 기존 네트워크 Section 실습 리소스는 정리되어 있는 상태를 기준으로 한다. Gallery 프로젝트 리소스는 `aws-fund-gallery-...` 네이밍으로 분리해 충돌을 피한다.

---

### 실습 목표

- Gallery 프로젝트 전용 VPC를 새로 만들고(`10.10.0.0/16`), 그 안에 네트워크 리소스(Public/Private Subnet, IGW, Route Table, NAT)를 구성한다.
- Private Subnet에 Gallery EC2를 배치하고, NAT 기반 Session Manager로 운영 접속 경로를 만든다.
- NAT Gateway로 Private EC2의 Outbound를 열고 Gallery를 설치/실행한다.
- Outbound는 되지만 Inbound(외부 접근)는 아직 안 되는 것이 정상임을 확인하고, 해결은 다음 Chapter(ALB)로 연결한다.

⚠️ 비용 주의: 이 실습에서는 EC2, NAT Gateway(필요 시) 등을 사용한다. 학습 종료 시점에 프로젝트 전체 정리 기준을 적용한다.

## 5. Private EC2 생성(Gallery 실행용, Public IPv4 없음)

설명: Gallery를 실행할 EC2를 Private Subnet(AZ-a)에 배치한다.

[이미지: AWS Console - EC2 - Launch instance - VPC/Subnet(private-a) 선택]

- 따라서 EC2 Security Group inbound는 "empty(또는 최소)"가 기본값이 된다.

## 8. Session Manager 접속(Private EC2)

설명: NAT로 Outbound가 열린 뒤, Session Manager로 Private EC2에 접속한다. 이 실습에서는 VPC Endpoint를 구성하지 않으며, NAT가 있기 때문에 Endpoint 없이도 SSM Agent가 통신할 수 있다.

[이미지: AWS Console - EC2 - Connect - Session Manager - Connected 확인]

## 9. Gallery 설치/실행(Private EC2)

설명: Session Manager로 Private EC2에 접속해 Gallery를 설치하고 `:8080`으로 실행한다.

[이미지: 터미널 - git clone/mvnw build - 빌드 성공]
[이미지: 터미널 - Gallery 실행 - 8080 바인딩 로그]

예시(개념 설명용):

```bash
git clone **{gallery-repo-url}** gallery-spring-boot
cd gallery-spring-boot
./mvnw clean package -DskipTests -Dbuild.finalName=gallery
nohup java -jar target/gallery.jar --server.port=8080 > app.log 2>&1 &
tail -n 50 app.log
```

## 10. SG 확인(8080 외부 노출 금지)

설명: 이 단계에서는 외부에서 `:8080`으로 들어오는 엔드포인트를 만들지 않는다. Security Group Inbound에 8080이 광범위하게 열려 있지 않은지 확인한다.

[이미지: AWS Console - EC2 - Security Groups - Gallery SG Inbound - 8080 0.0.0.0/0 없음 확인]
