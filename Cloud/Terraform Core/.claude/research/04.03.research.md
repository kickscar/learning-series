## 질문별 상세 답변

### Q1. `use_lockfile = true` — 정확한 문법

**타입**: boolean
**기본값**: `false` (TF 1.14.x 기준, 변경 계획 없음)
**도입**: TF 1.10 (실험적), TF 1.11 (GA — "experimental" 표기 제거, `dynamodb_table` deprecated)

```hcl
terraform {
  backend "s3" {
    bucket       = "my-terraform-state"
    key          = "project/terraform.tfstate"
    region       = "ap-northeast-2"
    encrypt      = true
    use_lockfile = true
  }
}
```

`use_lockfile`은 S3 조건부 쓰기(conditional writes)를 이용한 네이티브 잠금 메커니즘을 활성화한다.

### Q2. S3 native locking — 동시 apply 시 동작

**메커니즘**: S3 `PutObject` API에 `If-None-Match` 헤더를 포함하여 `.tflock` 파일을 업로드한다. 파일이 이미 존재하면(다른 사용자가 lock 보유 중) S3가 HTTP 409 Conflict 또는 412 Precondition Failed를 반환한다.

**에러 메시지 (실제 출력)**:
```text
Error: Error acquiring the state lock

Error message: operation error S3: PutObject, https response error
StatusCode: 409, RequestID: <id>, api error ConditionalRequestConflict:
The conditional request cannot succeed due to a conflicting operation
against this resource.

Terraform acquires a state lock to protect the state from being written
by multiple users at the same time. Please resolve the issue above and
try again.
```

**lock 파일 내용**: lock ID와 메타데이터(사용자 정보, 작업 유형 등)를 포함하는 JSON 객체.

**lock 해제**: 작업 완료 후 `.tflock` 파일을 `DeleteObject`로 삭제하여 lock을 해제한다.

**`terraform force-unlock`**: lock이 비정상적으로 남아 있을 때 수동 해제. lock ID가 필요하며, 다른 사용자의 활성 lock 해제는 위험하다.

### Q3. DynamoDB `dynamodb_table` — deprecated 여부

**상태**: TF 1.11부터 **deprecated**. 향후 minor 버전에서 제거 예정.

**마이그레이션 지원**: `use_lockfile`과 `dynamodb_table`을 동시에 설정할 수 있다. 이 경우 S3 lock 획득 → DynamoDB lock 획득 순서로 동작하며, 해제는 역순이다. 이전 버전과의 호환을 위한 전환 기간 지원이다.

**AWS 공식 권장** (AWS Prescriptive Guidance):
> "Amazon S3 native state locking (recommended): Available since Terraform 1.10.0"
> "DynamoDB state locking (deprecated): Legacy approach that will be removed in future Terraform versions"

**결론**: draft에서 DynamoDB locking은 "레거시 방식"으로 간략히 언급하고, S3 native locking을 주력으로 다룬다.

### Q4. `-migrate-state` vs `-reconfigure`

| 플래그 | 동작 | 사용 시점 |
|--------|------|----------|
| `-migrate-state` | 기존 state를 새 backend로 **복사** | local → S3 전환, S3 bucket 변경 등 |
| `-reconfigure` | backend 설정만 교체, state 이전 **없음** | 새로 시작, 기존 state 불필요 시 |
| `-force-copy` | `-migrate-state` 자동 활성화 + 대화형 프롬프트 생략 | CI/CD 자동화 |

**`-migrate-state` 상세 동작**:
1. 현재 backend에서 state를 읽음
2. 새 backend 초기화
3. state를 새 backend에 쓰기
4. workspace가 여러 개면 대화형 프롬프트로 확인

**lineage 유지**: 마이그레이션 시 lineage는 변경되지 않는다 (04.02 research에서 확인).

**중요**: backend 변경 시 `-migrate-state`도 `-reconfigure`도 지정하지 않으면 에러가 발생한다. 반드시 둘 중 하나를 명시해야 한다.

### Q5. backend "s3" — 필수 인수 vs 선택 인수

**필수 (Required)**:

| 인수 | 타입 | 설명 |
|------|------|------|
| `bucket` | string | S3 버킷 이름 |
| `key` | string | 버킷 내 state 파일 경로 |
| `region` | string | S3 버킷의 AWS 리전 (`AWS_DEFAULT_REGION` 환경변수로 대체 가능) |

**주요 선택 (Optional)**:

| 인수 | 타입 | 기본값 | 설명 |
|------|------|--------|------|
| `encrypt` | bool | `false` | 서버 측 암호화 (SSE-S3) |
| `use_lockfile` | bool | `false` | S3 native locking 활성화 |
| `kms_key_id` | string | — | SSE-KMS 암호화용 KMS Key ARN |
| `sse_customer_key` | string | — | SSE-C 암호화용 키 (base64) |
| `acl` | string | — | Canned ACL |
| `workspace_key_prefix` | string | `"env:"` | workspace별 state 경로 접두사 |
| `profile` | string | — | AWS CLI 프로파일 |
| `access_key` | string | — | AWS Access Key |
| `secret_key` | string | — | AWS Secret Key |
| `token` | string | — | MFA 토큰 |
| `max_retries` | number | `5` | API 재시도 횟수 |
| `assume_role` | block | — | Cross-account 접근 (role_arn 등) |
| `dynamodb_table` | string | — | **Deprecated** — DynamoDB locking |

### Q6. Partial config — 정확한 문법

**방법 1: 파일 기반** (`-backend-config=PATH`)

`backend.hcl` 파일 (권장 확장자: `*.s3.tfbackend`):
```hcl
bucket       = "my-terraform-state"
region       = "ap-northeast-2"
encrypt      = true
use_lockfile = true
```

```bash
$ terraform init -backend-config=backend.hcl
```

파일 형식: HCL top-level attribute만 사용. `terraform` 블록이나 `backend` 블록으로 감싸지 않는다.

**방법 2: CLI key=value** (`-backend-config="KEY=VALUE"`)

```bash
$ terraform init \
  -backend-config="bucket=my-terraform-state" \
  -backend-config="region=ap-northeast-2"
```

**main.tf의 backend 블록** (partial config 사용 시):
```hcl
terraform {
  backend "s3" {
    key = "project/terraform.tfstate"
    # 나머지는 -backend-config로 주입
  }
}
```

**우선순위**: CLI key=value > 파일 > backend 블록 (동일 키가 여러 곳에 있으면 CLI가 우선)

### Q7. S3 버킷 IAM 권한

**State 파일 관련 (필수)**:

```json
{
  "Effect": "Allow",
  "Action": [
    "s3:ListBucket"
  ],
  "Resource": "arn:aws:s3:::my-terraform-state"
},
{
  "Effect": "Allow",
  "Action": [
    "s3:GetObject",
    "s3:PutObject"
  ],
  "Resource": "arn:aws:s3:::my-terraform-state/project/terraform.tfstate"
}
```

**Lock 파일 관련 (`use_lockfile = true` 시 추가)**:

```json
{
  "Effect": "Allow",
  "Action": [
    "s3:GetObject",
    "s3:PutObject",
    "s3:DeleteObject"
  ],
  "Resource": "arn:aws:s3:::my-terraform-state/project/terraform.tfstate.tflock"
}
```

**Workspace 사용 시**: `s3:DeleteObject` 권한이 workspace state 경로에도 필요하다.

**KMS 사용 시**: `kms:Encrypt`, `kms:Decrypt`, `kms:GenerateDataKey` 권한이 추가로 필요하다.

### Q8. S3 versioning — 필수 vs 권장

**결론: 필수가 아닌 강력 권장 (highly recommended)**

공식 문서: "It is highly recommended that you enable Bucket Versioning on the S3 Bucket to allow for state recovery in the case of accidental deletions and human error."

AWS Prescriptive Guidance에서도 versioning + AWS Backup을 권장한다.

**이유**:
- State 파일 실수로 삭제/손상 시 이전 버전으로 복구 가능
- serial/lineage 기반 충돌 보호에 추가적인 안전망 제공
- `terraform state push -force` 같은 위험한 작업에서도 복구 경로 확보

**draft 작성 시**: lab01에서 S3 버킷 생성 시 versioning 활성화를 기본으로 포함. "필수는 아니지만 사실상 필수"라는 톤으로 서술한다.

### Q9. backend 블록에서 변수/locals 참조 가능 여부

**불가능 — 리터럴 값만 허용된다.**

공식 문서: "A backend block cannot refer to named values (like input variables, locals, or data source attributes)."

```hcl
# 불가능 — 에러 발생
terraform {
  backend "s3" {
    bucket = var.state_bucket    # Error!
    key    = local.state_key     # Error!
    region = var.region           # Error!
  }
}
```

**이유**: backend 초기화는 Terraform 실행의 가장 첫 단계이다. 변수 평가, locals 계산, data source 조회보다 먼저 수행되므로 이 시점에서 named value에 접근할 수 없다.

**대안**: partial configuration으로 런타임에 값을 주입한다 (Q6 참고).

### Q10. `encrypt = true` — 암호화 방식

**기본 동작**: `encrypt = true`이면 **SSE-S3 (AES-256)** 암호화를 사용한다. S3가 관리하는 키로 서버 측 암호화를 수행한다.

**SSE-KMS 사용**: `kms_key_id`를 지정하면 SSE-KMS로 전환된다.
```hcl
terraform {
  backend "s3" {
    bucket     = "my-terraform-state"
    key        = "project/terraform.tfstate"
    region     = "ap-northeast-2"
    encrypt    = true
    kms_key_id = "arn:aws:kms:ap-northeast-2:123456789012:key/12345678-1234-..."
  }
}
```

**SSE-C 사용**: `sse_customer_key`로 고객 제공 키 암호화가 가능하다. base64 인코딩된 256-bit 키를 제공해야 하며, 환경변수 `AWS_SSE_CUSTOMER_KEY`로 주입하는 것을 권장한다.

**`encrypt` 기본값**: `false`이다. 명시적으로 `true`를 설정해야 한다. 단, AWS S3는 2023년부터 모든 새 객체에 SSE-S3를 기본 적용하므로, `encrypt = true`를 설정하지 않아도 실질적으로 암호화는 적용된다. 그러나 명시적 설정이 Terraform 공식 모범 사례이다.

## 주의사항

### S3 native locking과 versioned bucket의 알려진 이슈

GitHub Issue #36445: versioning이 활성화된 S3 버킷에서 `.tflock` 파일의 이전 버전이 쌓이는 문제가 보고되었다. Lock 파일은 작업마다 생성/삭제되므로 versioned bucket에서는 delete marker + 이전 버전이 누적된다. S3 Lifecycle 규칙으로 비현행 버전을 자동 삭제하는 것이 권장된다.

**draft 작성 시**: lab01에서 S3 버킷 생성 시 이 점을 간략히 언급할 수 있으나, lifecycle 규칙까지 다루면 범위가 넓어지므로 "참고" 수준으로 처리한다.

### `encrypt` 기본값 `false`와 AWS S3 기본 암호화

2023년 1월부터 AWS S3는 모든 새 객체에 SSE-S3 기본 암호화를 적용한다. 따라서 Terraform의 `encrypt` 인수를 설정하지 않아도 실질적으로 state 파일은 암호화된다. 그러나 `encrypt = true`는 Terraform 측에서 명시적으로 암호화를 요청하는 것이므로, 공식 문서와 모범 사례에 따라 설정을 권장한다.

### backend 변경 시 `-migrate-state` 필수

backend 설정을 변경한 후 `terraform init`을 실행하면 `-migrate-state` 또는 `-reconfigure` 중 하나를 반드시 지정해야 한다. 아무것도 지정하지 않으면 에러가 발생한다. lab02에서 이 흐름을 실습한다.

### Lab 설계 시 S3 버킷 이름 유니크 제약

S3 버킷 이름은 전역적으로 고유해야 한다. lab01에서 버킷 생성 시 독자가 자신만의 고유 이름을 사용하도록 안내해야 한다. 예시 이름에 `{your-unique-suffix}` 같은 플레이스홀더를 포함하는 것이 좋다.
