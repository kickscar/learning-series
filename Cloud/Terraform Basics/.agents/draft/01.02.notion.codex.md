# **개요**

본 Section에서는 CloudFormation과 Terraform을 비교하여 두 도구가 어떤 공통점과 차이점을 가지는지 정리한다. 독자는 이미 CloudFormation을 입문 수준에서 경험한 상태이므로, 이 문서는 CloudFormation 자체를 깊게 설명하기보다 Terraform을 이해하기 위한 비교 기준을 제공하는 데 목적이 있다.

핵심은 “둘 다 infrastructure as Code(IaC) 도구이지만, 상태를 관리하는 방식과 실행 모델, 적용 범위, 확장성, 협업 방식이 다르다”는 점을 명확히 이해하는 것이다.

# **핵심 포인트**

- CloudFormation과 Terraform은 모두 선언형(Declarative) IaC 도구이지만, 내부 운영 모델은 동일하지 않다.
- CloudFormation은 AWS stack 중심으로 동작하고, Terraform은 provider와 `Terraform state` 중심으로 동작한다.
- Terraform은 `plan`과 `apply`를 기준으로 변경을 계산하고, CloudFormation은 stack update 방식으로 변경을 수행한다.
- Terraform은 multi-cloud와 provider 확장성이 강점이고, CloudFormation은 AWS 네이티브 통합이 강점이다.
- Terraform Basics에서 CloudFormation 비교의 목적은 Terraform의 특징을 더 선명하게 이해하는 데 있다.

# **비교 목적**

CloudFormation 경험이 있는 학습자는 Terraform을 처음 접할 때 “둘 다 인프라를 코드로 만드는 도구 아닌가?”라는 수준에서 출발하기 쉽다. 이 관점은 부분적으로 맞지만, 실제 운영에서는 도구가 인프라를 해석하고 변경을 적용하는 방식의 차이가 중요하게 작동한다.

이 비교가 필요한 이유는 다음과 같다.

- Terraform이 왜 `state`를 핵심 개념으로 가지는지 이해하기 위해
- Terraform에서 `plan` 검토가 왜 중요한지 이해하기 위해
- provider 기반 구조가 어떤 확장성을 가지는지 이해하기 위해
- CloudFormation 경험을 Terraform 학습 자산으로 전환하기 위해

즉, 이 Section의 목적은 두 도구 중 하나를 우열 비교하는 데 있지 않고, Terraform의 실행 모델과 설계 철학을 더 분명히 읽어내는 데 있다.

# **CloudFormation과 Terraform : 공통점**

두 도구는 모두 선언형(Declarative) IaC 도구라는 공통점을 가진다. 사용자는 원하는 최종 상태를 코드로 정의하고, 도구는 현재 상태와 비교하여 필요한 변경을 수행한다.

두 도구의 공통점은 다음과 같다.

| 항목 | 공통점 |
| --- | --- |
| IaC 방식 | 인프라를 코드로 정의하고 변경을 관리한다. |
| 선언형 접근 | 최종 상태를 정의하고 도구가 필요한 변경을 계산한다. |
| 반복 가능성 | 같은 정의를 기반으로 환경을 반복 생성할 수 있다. |
| 버전 관리 | 코드를 Git 같은 VCS에 저장하고 변경 이력을 관리할 수 있다. |
| 자동화 연계 | CI/CD와 연결하여 인프라 변경 자동화에 활용할 수 있다. |

이 공통점 때문에 CloudFormation 경험자는 Terraform을 비교적 빠르게 받아들일 수 있다. 하지만 실제 차이는 도구의 내부 기준점과 운영 모델에서 드러난다.

# **CloudFormation과 Terraform : 핵심 차이**

## **비교 표**

| 비교 항목 | CloudFormation | Terraform |
| --- | --- | --- |
| 동작 범위 | AWS 전용 | provider 기반 multi-platform |
| 관리 단위 | Stack | Configuration + State |
| 변경 검토 | Change Set 또는 Stack update 기준 | `terraform plan` 기준 |
| 상태 관리 | AWS가 stack 상태를 서비스 내부에서 관리 | `Terraform state`를 별도 관리 |
| 확장 방식 | AWS 리소스와 서비스에 최적화 | provider ecosystem을 통한 확장 |
| 운영 포인트 | AWS 네이티브 통합 | state, backend, module, provider 관리 |

이 표에서 가장 중요한 차이는 `상태 관리의 위치`와 `확장 구조`다.

## **관리 기준 : Stack vs Terraform state**

CloudFormation은 stack을 중심으로 리소스를 관리한다. 사용자는 template를 기준으로 stack을 생성하거나 업데이트하고, CloudFormation 서비스가 그 stack의 상태를 관리한다.

Terraform은 조금 다르다. Terraform은 Configuration만으로 동작하지 않고, 별도의 `Terraform state`를 기준으로 현재 관리 중인 리소스를 추적한다. 이후 `plan`과 `apply`는 이 state를 기반으로 계산된다.

이 차이 때문에 Terraform 학습에서는 state가 매우 중요하다. CloudFormation 경험자에게는 “도구가 관리 상태를 어디에 두는가”가 가장 먼저 체감되는 차이점 중 하나다.

> CloudFormation이 stack 중심으로 상태를 관리한다면, Terraform은 configuration과 state를 함께 기준으로 변경을 계산한다.

## **변경 적용 방식 : Stack Update vs Plan / Apply**

CloudFormation에서 일반적인 변경 흐름은 template 수정 후 stack update를 수행하는 방식이다. Change Set을 통해 변경을 미리 볼 수 있지만, 실제 학습과 운영 경험에서는 stack update 중심으로 인식되는 경우가 많다.

Terraform은 `terraform plan`과 `terraform apply`를 명확히 분리한다. `plan`은 현재 상태와 목표 상태를 비교해 어떤 변경이 발생할지를 계산하고, `apply`는 그 결과를 실제 환경에 반영한다.

이 구조는 Terraform 운영에서 중요한 습관을 만든다.

- 먼저 `plan`을 확인한다.
- 변경 범위를 검토한다.
- 그 다음 `apply`를 수행한다.

즉, Terraform에서는 변경 검토 단계가 더 독립적이고 명시적으로 드러난다.

## **확장성 : AWS 전용 vs Provider Ecosystem**

CloudFormation은 AWS에 최적화된 IaC 도구다. AWS 서비스와의 통합성이 강하고, AWS 네이티브 기능을 빠르게 반영하는 장점이 있다.

Terraform은 provider 기반 구조를 사용한다. AWS provider를 사용하면 AWS를 다루고, 다른 provider를 사용하면 Azure, GCP, GitHub, Datadog, Kubernetes 같은 외부 시스템도 같은 문법과 workflow로 다룰 수 있다.

이 구조는 Terraform의 장점을 만든다.

- multi-cloud 대응
- SaaS와 클라우드 자원의 통합 관리
- 동일한 workflow로 여러 플랫폼 운영

반대로 운영 난이도 측면에서는 provider 버전 관리, state 관리, backend 설계 같은 별도 고려사항도 생긴다.

# **Terraform 관점에서 중요한 의미**

CloudFormation 경험자가 Terraform으로 넘어올 때 특히 중요하게 봐야 할 포인트는 다음과 같다.

### **① Terraform은 state 중심 운영 감각이 필요하다**

CloudFormation에서는 stack이 상태 관리의 중심이라면, Terraform에서는 `Terraform state`가 핵심 운영 자산이다. 따라서 backend, state locking, drift, import 같은 주제가 자연스럽게 중요해진다.

### **② Terraform은 provider와 module을 통해 더 넓게 확장된다**

Terraform은 단순히 AWS 대체 도구가 아니라, provider ecosystem과 module 구조를 기반으로 재사용성과 확장성을 높이는 도구다. 이 점이 이후 module, backend, production practice 학습으로 이어진다.

### **③ Terraform은 plan 검토 문화가 중요하다**

Terraform에서는 `plan`을 읽고 변경 범위를 해석하는 능력이 중요하다. 이 능력은 단순 CLI 사용법이 아니라 Terraform 운영 모델 자체를 이해하는 문제다.

### **④ CloudFormation 지식은 버려지는 것이 아니라 비교 기준이 된다**

CloudFormation 경험은 Terraform 학습에 도움이 된다. 다만 CloudFormation template 작성 습관을 그대로 옮기기보다, Terraform의 state와 provider 중심 구조에 맞게 사고 방식을 바꾸는 것이 중요하다.

# **언제 Terraform을 더 선호할 수 있는가**

다음과 같은 상황에서는 Terraform이 더 자연스러운 선택이 될 수 있다.

| 상황 | Terraform이 유리한 이유 |
| --- | --- |
| multi-cloud 또는 hybrid 운영 | provider 기반으로 여러 플랫폼을 같은 방식으로 다룰 수 있다. |
| SaaS와 클라우드 리소스를 함께 관리 | GitHub, Datadog, Kubernetes 등도 같은 workflow로 연결할 수 있다. |
| module 기반 재사용 설계 | 반복 인프라를 module 구조로 추상화하기 쉽다. |
| plan 검토 중심 운영 | apply 전 변경 범위를 명확하게 검토할 수 있다. |

반대로 AWS 네이티브 통합과 AWS 서비스 중심 운영이 절대적으로 중요한 환경에서는 CloudFormation이 더 자연스러운 선택일 수도 있다. 중요한 것은 도구를 추상적으로 선호하는 것이 아니라, 운영 구조와 요구사항에 맞게 판단하는 것이다.

# **정리**

이 Section에서는 CloudFormation과 Terraform의 공통점과 차이를 비교했다. 두 도구는 모두 선언형 IaC 도구이지만, CloudFormation은 AWS stack 중심, Terraform은 provider와 `Terraform state` 중심이라는 차이를 가진다.

Terraform을 학습할 때 이 비교가 중요한 이유는, Terraform을 단순히 “CloudFormation과 비슷한 또 다른 IaC 도구”로 이해하는 수준을 넘어서기 위해서다. Terraform은 state, plan, provider, module을 중심으로 고유한 운영 모델을 가진다.

다음 Section에서는 실제로 Terraform CLI를 설치하고 실행 환경을 준비하면서, 이러한 차이가 실제 학습 workflow에 어떻게 연결되는지 살펴본다.
